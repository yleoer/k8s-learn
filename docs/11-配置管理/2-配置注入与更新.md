# 配置注入与更新

Pod 可以通过 `env`、`envFrom`、容器命令参数或投射卷消费 ConfigMap。不同路径的更新传播方式不同，选择时需要同时考虑应用接口、重载能力和文件覆盖行为。

## 注入环境变量

下面的 ConfigMap 同时演示按键引用和批量导入：

```yaml [env-config.yaml]
apiVersion: v1
kind: ConfigMap
metadata:
  name: env-config
data:
  APP_MODE: "production"
  LOG_LEVEL: "info"
  HTTP_PORT: "8080"
---
apiVersion: v1
kind: Pod
metadata:
  name: config-env-demo
spec:
  restartPolicy: Never
  containers:
    - name: app
      image: busybox:1.38
      command:
        - sh
        - -c
        - env | sort && sleep 3600
      env:
        - name: PORT
          valueFrom:
            configMapKeyRef:
              name: env-config
              key: HTTP_PORT
        - name: LOG_LEVEL
          value: debug
      envFrom:
        - prefix: APP_
          configMapRef:
            name: env-config
```

```bash
kubectl create -f env-config.yaml
kubectl exec config-env-demo -- env
```

`envFrom` 会把每个键转成环境变量，本例还会添加 `APP_` 前缀。显式 `env` 与 `envFrom` 产生同名变量时，`env` 的值优先；多个 `envFrom` 来源产生同名变量时，后面的来源优先。

Kubernetes v1.34 起，环境变量名校验已放宽为除 `=` 之外的可打印 ASCII 字符；ConfigMap 键允许的字母、数字、`-`、`_` 和 `.` 都可以通过 `envFrom` 注入。应用框架、Shell 和跨平台工具仍可能只接受传统的字母、数字与下划线命名，因此长期配置更适合使用 `APP_MODE` 这类兼容名称，或通过显式 `env.name` 映射。

> [!IMPORTANT]
> 环境变量只在容器启动时生成。ConfigMap 更新后，已运行容器中的环境变量不会变化，需要重建 Pod，例如对 Deployment 执行 `kubectl rollout restart deploy/<name>`。

## 命令与参数

`command` 和 `args` 不能直接使用 `configMapKeyRef`，可以先把键注入环境变量，再通过 `$(VAR_NAME)` 引用。下面只是容器字段片段，不是完整 Pod 清单：

```yaml
containers:
  - name: app
    image: busybox:1.38
    command: ["sh", "-c"]
    args: ['printf "mode=%s\n" "$(APP_MODE)" && sleep 3600']
    env:
      - name: APP_MODE
        valueFrom:
          configMapKeyRef:
            name: env-config
            key: APP_MODE
```

Kubernetes 在启动容器前根据容器环境展开 `command` 和 `args` 中的 `$(VAR_NAME)`；`env.value` 内部的变量引用只能使用此前已经定义的变量。无法解析的引用会保持原样，`$$(VAR_NAME)` 用于保留字面量。通过 Shell 启动进程时还可能发生第二次 Shell 变量展开，需要区分两层语义。

## 挂载为文件

普通 ConfigMap 卷会把键投射为目录下的文件：

```yaml [config-volume.yaml]
apiVersion: v1
kind: ConfigMap
metadata:
  name: web-config
data:
  app.conf: |-
    listen=8080
    log_level=info
  message.txt: |-
    configuration loaded
---
apiVersion: v1
kind: Pod
metadata:
  name: config-volume-demo
spec:
  restartPolicy: Never
  containers:
    - name: app
      image: busybox:1.38
      command: ["sh", "-c", "ls -l /etc/app && cat /etc/app/runtime.conf && sleep 3600"]
      volumeMounts:
        - name: config
          mountPath: /etc/app
          readOnly: true
  volumes:
    - name: config
      configMap:
        name: web-config
        defaultMode: 0440
        items:
          - key: app.conf
            path: runtime.conf
          - key: message.txt
            path: message.txt
            mode: 0400
```

```bash
kubectl create -f config-volume.yaml
kubectl exec config-volume-demo -- cat /etc/app/runtime.conf
```

`items` 只投射列出的键，并可通过 `path` 改名。`defaultMode` 对全部文件生效，单个条目的 `mode` 优先；YAML 可使用八进制权限，JSON 中必须写十进制数值。Windows 节点不按 POSIX 文件权限解释这些字段。

如果 ConfigMap 或 `items` 中要求的键不存在，且未设置 `optional: true`，kubelet 无法准备卷，Pod 通常停留在 `ContainerCreating`。可选配置可以这样声明：

```yaml{5}
volumes:
  - name: optional-config
    configMap:
      name: optional-config
      optional: true
```

这只是用于说明 `optional` 字段关系的片段，不是完整 Pod 清单。可选对象或键不存在时不会阻止 Pod 启动，应用必须能够处理文件缺失。

## 组合投射卷

`projected` 卷可以把 ConfigMap、Secret、Downward API 和 ServiceAccount Token 等来源合并到同一目录。下面只展示 ConfigMap 与 Secret 的字段关系，不是完整 Pod 清单：

```yaml
volumes:
  - name: runtime-data
    projected:
      defaultMode: 0440
      sources:
        - configMap:
            name: env-config
            items:
              - key: APP_MODE
                path: config/app-mode
        - secret:
            name: app-secret
            items:
              - key: password
                path: credentials/password
                mode: 0400
```

所有来源都必须与 Pod 位于同一命名空间。`defaultMode` 在 `projected` 层统一设置，单个 `items` 条目仍可覆盖权限；ConfigMap 和 Secret 的文件会按各自更新机制最终一致地变化，但通过 `subPath` 挂载投射卷时同样不会接收更新。

## 目录覆盖与 subPath

把卷挂载到 `/etc/app` 后，容器镜像原本位于该目录的文件会被挂载遮蔽。只替换单个文件时可以使用 `subPath`：

```yaml [single-file-config.yaml]
apiVersion: v1
kind: ConfigMap
metadata:
  name: nginx-server-config
data:
  default.conf: |-
    server {
        listen 8080;
        server_name _;
        location /healthz { return 200 "ok\n"; }
    }
---
apiVersion: v1
kind: Pod
metadata:
  name: nginx-config-demo
spec:
  containers:
    - name: nginx
      image: nginx:1.31-alpine
      ports:
        - name: http
          containerPort: 8080
      volumeMounts:
        - name: config
          mountPath: /etc/nginx/conf.d/default.conf
          subPath: default.conf
          readOnly: true
  volumes:
    - name: config
      configMap:
        name: nginx-server-config
```

```bash
kubectl create -f single-file-config.yaml
```

> [!WARNING]
> `subPath` 挂载不会收到 ConfigMap 后续更新。需要热更新时，应挂载整个目录，并让应用监听或定期重新读取目标文件；如果应用不支持重载，则通过工作负载滚动更新 Pod。

## 更新传播

普通 ConfigMap 卷中的文件会最终一致地更新。kubelet 在周期同步时检查源对象，并可能使用本地缓存，因此从 API 对象更新到 Pod 文件变化之间存在延迟，最长通常由 kubelet 同步周期与缓存传播延迟共同决定。

更新 ConfigMap 后可以观察文件：

```bash
kubectl edit cm web-config
kubectl exec config-volume-demo -- cat /etc/app/runtime.conf
```

文件变化不等于应用已经加载新配置。应用需要自行实现文件监听、信号重载或定期读取；否则仍需重建 Pod。

Deployment、StatefulSet 等工作负载控制器不会因为其引用的 ConfigMap 内容变化而自动滚动更新。声明式发布流程常把配置内容摘要写入 Pod 模板注解，或使用带版本的 ConfigMap 名称；摘要或引用名变化会更新 Pod 模板，从而触发控制器滚动替换 Pod。

## 排查路径

```bash
kubectl get cm
kubectl describe cm web-config
kubectl describe po config-volume-demo
kubectl get ev --sort-by=.metadata.creationTimestamp
```

常见问题包括对象或键名拼写错误、Pod 与 ConfigMap 不在同一命名空间、`items.path` 冲突、应用不接受环境变量命名、文件权限不满足进程用户要求，以及使用 `subPath` 后误以为文件会自动更新。
