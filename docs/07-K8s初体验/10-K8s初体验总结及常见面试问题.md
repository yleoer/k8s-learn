# K8s 初体验总结及常见面试问题

本章完成了 Kubernetes 初体验的基础闭环：使用 kubectl 连接和操作集群，使用 Namespace 做逻辑隔离，使用 Pod 运行容器，并通过状态、事件和日志定位常见问题。

## 核心知识总结

kubectl 是用户与 Kubernetes 集群交互的主要入口。它把命令请求提交给 APIServer，再由控制面和工作节点组件完成资源持久化、调度、创建和状态上报。

Namespace 是逻辑隔离方式，适合按环境、团队或项目划分资源边界。它需要配合 RBAC、ResourceQuota、LimitRange 和 NetworkPolicy 才能形成更完整的治理能力。

Pod 是 Kubernetes 的最小调度单元。一个 Pod 可以包含一个或多个容器，同一 Pod 内的容器共享网络和存储卷。生产中通常使用 Deployment、StatefulSet、DaemonSet 等控制器管理 Pod。

## 常用命令回顾

查看集群与上下文：

```bash
kubectl cluster-info
kubectl version
kubectl config get-contexts
kubectl config current-context
kubectl config use-context <context-name>
```

查看资源：

```bash
kubectl get pod
kubectl get pod -A
kubectl get pod -o wide
kubectl describe pod <pod-name>
```

创建与更新资源：

```bash
kubectl create namespace dev
kubectl apply -f resource.yaml
kubectl create deployment nginx --image=nginx:1.25
kubectl create deployment nginx --image=nginx:1.25 --dry-run=client -o yaml
```

排查问题：

```bash
kubectl logs <pod-name>
kubectl logs <pod-name> --previous
kubectl exec -it <pod-name> -- sh
kubectl get events --sort-by=.metadata.creationTimestamp
```

字段查询：

```bash
kubectl explain pod
kubectl explain pod.spec.containers
kubectl api-resources
```

## 面试问题：kubectl 的作用是什么

kubectl 是 Kubernetes 的官方命令行客户端，用于向 APIServer 发送请求，完成资源查看、创建、修改、删除、日志查看、容器调试、上下文切换等操作。

它不直接管理容器，也不直接操作工作节点。所有请求都会经过 APIServer，由 Kubernetes 控制面和节点组件完成后续处理。

## 面试问题：create 和 apply 有什么区别

`create` 偏命令式创建，资源已经存在时通常会报错。它适合快速创建资源或生成 YAML 模板。

`apply` 偏声明式管理，资源不存在时创建，资源存在时更新。生产中更推荐使用 `apply -f` 配合 YAML 文件进行资源管理，因为它更适合版本控制和重复执行。

## 面试问题：Namespace 能实现什么隔离

Namespace 能实现资源名称和资源管理范围上的逻辑隔离。它可以把 Pod、Deployment、Service、ConfigMap、Secret 等命名空间级资源划分到不同空间。

Namespace 本身不是强安全边界。权限控制需要配合 RBAC，资源限制需要配合 ResourceQuota 和 LimitRange，网络隔离需要配合 NetworkPolicy。

## 面试问题：default、kube-system、kube-public、kube-node-lease 分别是什么

`default` 是默认 Namespace，未指定 Namespace 的资源通常会创建到这里。

`kube-system` 用于放置 Kubernetes 系统组件和集群插件。

`kube-public` 通常用于存放可公开读取的非敏感信息。

`kube-node-lease` 用于保存节点 Lease 对象，帮助控制面判断节点心跳状态。

## 面试问题：Pod 和容器是什么关系

Pod 是 Kubernetes 的最小调度单元，容器运行在 Pod 内部。一个 Pod 可以包含一个或多个容器。

同一个 Pod 内的容器共享网络命名空间和存储卷，可以通过 `localhost` 通信，也可以共享文件。但无关的服务不应该放进同一个 Pod。

## 面试问题：为什么 Kubernetes 不直接调度容器

真实应用中经常存在多个容器需要紧密协作，例如业务容器和日志采集 Sidecar。Pod 提供了统一调度、共享网络、共享存储和生命周期协同的边界。

因此 Kubernetes 调度 Pod，而不是单独调度容器。

## 面试问题：Pod 一直 Pending 怎么排查

先查看 Pod 详情和事件：

```bash
kubectl describe pod <pod-name>
kubectl get events --sort-by=.metadata.creationTimestamp
```

常见原因包括节点资源不足、节点选择器不匹配、污点没有容忍、PVC 未绑定、调度约束无法满足等。

## 面试问题：CrashLoopBackOff 怎么排查

先看当前日志和上一次容器日志：

```bash
kubectl logs <pod-name>
kubectl logs <pod-name> --previous
kubectl describe pod <pod-name>
```

常见原因包括启动命令错误、配置缺失、程序启动后退出、依赖服务不可用、健康检查配置不合理等。

## 后续学习方向

直接创建裸 Pod 适合理解原理，但生产中通常通过控制器管理 Pod。下一阶段需要重点学习 Pod 字段、生命周期、健康检查、资源限制和优雅退出，为后续 Deployment 与服务发布打好基础。
