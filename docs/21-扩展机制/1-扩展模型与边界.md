# 扩展模型与边界

Kubernetes 可通过 CRD、聚合 API、Admission webhook、设备插件和调度扩展等方式扩展。CRD 最适合声明式、自定义对象的 API 表达；若需要完整自定义 API Server 行为、不同存储或复杂协议，再评估聚合 API。

Operator 不是新的 Kubernetes 资源类型，而是一个控制器模式：监视自定义或内置资源，比较期望状态与实际状态，并持续采取幂等动作使系统收敛。它可以封装安装、配置、备份、故障恢复等领域操作，但不能消除底层系统的容量、数据一致性或灾难恢复责任。

## 选择边界

| 需求 | 适合的机制 |
| --- | --- |
| 参数化安装一组已有资源 | Helm 或 Kustomize |
| 定义新对象并保存声明状态 | CRD |
| 持续维护新对象的实际状态 | CRD 加控制器 / Operator |
| 拦截和校验现有 API 请求 | ValidatingAdmissionPolicy 或 webhook |
| 提供完整自定义 API 行为 | 聚合 API |

不要把一次性部署脚本包装成 Operator。只有需要持续协调、状态观测与生命周期管理时，控制器的复杂度才有回报。

## API 能力观察

先观察集群已注册的扩展 API 和自定义资源，再决定新增对象是否真的需要新的 API 面：

```bash
kubectl api-resources --api-group=apiextensions.k8s.io
kubectl get crd
kubectl api-resources --verbs=list --namespaced=true
```

需要尝试 CRD 时，复用[CRD 与自定义资源](./2-CRD与自定义资源.md#创建一个命名空间资源)中的完整 `crontab-crd.yaml`。该示例只验证 API 注册与 schema，不会自动创建业务工作负载；持续维护实际状态还需要后续的控制器。
