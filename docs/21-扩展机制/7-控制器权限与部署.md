# 控制器权限与部署

控制器的 ServiceAccount 只应拥有协调所需资源的最小权限。通常需要读取和写入自定义资源及其 `status` 子资源，并对受管资源授予限定的 get、list、watch、create、patch、update 或 delete；是否需要每种 verb 应从协调逻辑推导。

## 权限审查点

| 能力 | 审查重点 |
| --- | --- |
| 自定义资源读写 | spec 与 status 是否分离；是否需要 `finalizers` 更新 |
| 受管工作负载 | 是否只处理预期 apiGroups 和资源类型 |
| Secret 访问 | 是否可以改为引用名或外部密钥系统 |
| 跨命名空间 | 是否确实需要 cluster scope；是否限制 watch 范围 |
| 外部 API | 凭据轮换、超时、重试、审计与网络出口 |

控制器 Deployment 的镜像应使用固定版本和可验证来源，并配置 resources、探针、只读根文件系统等适用的运行时安全项。示例镜像必须根据选定实现的官方支持矩阵确认后再写入，不以未验证的第三方镜像替代。

## 最小权限清单

下面的完整清单为 `team-a` 中的控制器 ServiceAccount 授予读取和更新 `crontabs` 及其 `status` 的最小命名空间权限。它不包含跨命名空间 watch、Secret 读取或受管工作负载权限：

```yaml [crontab-controller-rbac.yaml]
apiVersion: v1
kind: ServiceAccount
metadata:
  name: crontab-controller
  namespace: team-a
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: crontab-controller
  namespace: team-a
rules:
  - apiGroups: ["stable.example.com"]
    resources: ["crontabs"]
    verbs: ["get", "list", "watch", "patch", "update"]
  - apiGroups: ["stable.example.com"]
    resources: ["crontabs/status"]
    verbs: ["get", "patch", "update"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: crontab-controller
  namespace: team-a
subjects:
  - kind: ServiceAccount
    name: crontab-controller
    namespace: team-a
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: Role
  name: crontab-controller
```

```bash
kubectl create -f crontab-controller-rbac.yaml
kubectl auth can-i update crontabs/status -n team-a \
  --as=system:serviceaccount:team-a:crontab-controller
kubectl auth can-i get secrets -n team-a \
  --as=system:serviceaccount:team-a:crontab-controller
```

预期 status 更新为 `yes`，读取 Secret 为 `no`。实际控制器若创建 Deployment、写入 finalizer 或访问集群范围资源，应在此基础上逐项增加所需权限，不能以 `cluster-admin` 代替审查。
