# 污点操作与排障

污点变更会立刻影响后续调度，`NoExecute` 还可能影响现有 Pod。生产变更应先列出现有污点、受影响工作负载和回滚命令，再执行。

## 常用操作

```bash
kubectl describe no <node-name>
kubectl taint no <node-name> maintenance.example.com/window=true:NoSchedule
kubectl taint no <node-name> maintenance.example.com/window=true:NoSchedule-
kubectl get po -A -o custom-columns='NAMESPACE:.metadata.namespace,NAME:.metadata.name,TOLERATIONS:.spec.tolerations'
```

同一键与效果重新设置可更新值。删除时末尾的连字符是命令语法的一部分。为避免误删相近污点，应在执行删除前用 `kubectl describe no` 确认键、值和效果。

## 排查路径

若事件出现 `had untolerated taint`，依次检查节点污点、Pod 的 `tolerations`、`operator`、`value` 和 `effect`。若 Pod 已在节点上被驱逐，还应确认是否新增 `NoExecute` 污点、容忍秒数是否到期，以及控制器能否在其他节点重建副本。

> [!NOTE]
> 使用 `--all` 或空键的批量污点操作会扩大影响范围。节点维护应优先使用 `cordon` 与 `drain`，不要把临时维护污点当成通用驱逐机制。
