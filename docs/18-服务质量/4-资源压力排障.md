# 资源压力排障

资源压力排障需要从 Pod、节点和集群容量三个层面同时定位。只重启被驱逐 Pod 会丢失事件线索，也无法修复持续的节点内存、磁盘或 PID 压力。

## 基本检查

```bash
kubectl top no
kubectl top po -A --containers
kubectl get po -A --field-selector=status.phase=Failed
kubectl get ev -A --sort-by=.metadata.creationTimestamp
kubectl describe no <node-name>
```

`kubectl top` 依赖 Metrics Server，反映的是近期使用量；调度器主要依据 requests，不会以 `top` 输出替代资源预留计算。节点 `Allocatable` 与 `Capacity` 的差值还包含系统、kubelet 和驱逐保留。

## 修复次序

先确认是否存在内存泄漏、异常日志写入、临时文件堆积或 PID 耗尽，再校准容器资源、扩容节点池或调整工作负载副本。对业务关键度的排序使用 PriorityClass；对自愿中断的可用性使用 PDB；不要用 QoS 类别替代这两项治理。
