# 拓扑分布约束

`topologySpreadConstraints` 控制一组匹配 Pod 在拓扑域之间的计数差。它面向副本分布，不要求每个副本都与某个既有 Pod 建立配对关系，适合跨节点、可用区和机架的服务副本。

约束中的 `labelSelector` 必须匹配当前工作负载的 Pod 标签。`maxSkew` 定义允许的最大不均衡度；`whenUnsatisfiable: DoNotSchedule` 是硬约束，`ScheduleAnyway` 是评分偏好。

## 跨可用区和主机分布

```yaml [zone-spread-api.yaml]
apiVersion: apps/v1
kind: Deployment
metadata:
  name: zone-spread-api
spec:
  replicas: 6
  selector:
    matchLabels:
      app.kubernetes.io/name: zone-spread-api
  template:
    metadata:
      labels:
        app.kubernetes.io/name: zone-spread-api
    spec:
      topologySpreadConstraints:
        - maxSkew: 1
          minDomains: 3
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app.kubernetes.io/name: zone-spread-api
        - maxSkew: 1
          topologyKey: kubernetes.io/hostname
          whenUnsatisfiable: ScheduleAnyway
          labelSelector:
            matchLabels:
              app.kubernetes.io/name: zone-spread-api
      containers:
        - name: api
          image: nginx:1.31-alpine
          ports:
            - containerPort: 80
```

`minDomains: 3` 只能与 `DoNotSchedule` 一同使用，且要求值大于零。可用拓扑域不足时，调度器以零作为全局最小值计算 `maxSkew`，因此该清单会阻止继续向少于三个可用区的集群放置新副本。

> [!NOTE]
> 一个 Pod 不能为同一 `topologyKey` 与同一 `whenUnsatisfiable` 组合重复定义约束。集群默认拓扑分布规则由调度器配置决定，应用应显式写出自身的可用性要求。

## 节点亲和性和污点的影响

默认的 `nodeAffinityPolicy` 与 `nodeTaintsPolicy` 为 `Honor`。这表示节点亲和性筛掉的节点和 Pod 不可容忍污点所在节点，不参与 eligible domains 的计算。把它们改为 `Ignore` 前，应明确希望用哪些节点作为分布基线。

## 观察结果

```bash
kubectl create -f zone-spread-api.yaml
kubectl get po -l app.kubernetes.io/name=zone-spread-api -o wide
kubectl get no -L topology.kubernetes.io/zone
```

输出中的 Pod 所在节点与节点标签共同决定分布，不应只凭 Pod 名称序号判断均衡性。
