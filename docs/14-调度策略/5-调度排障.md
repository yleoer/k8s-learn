# 调度排障

Pod 处于 `Pending` 时，应先区分尚未被调度、已绑定但容器未启动，以及已经被其他控制器替换三种状态。亲和性、污点、配额和资源不足都会以事件形式说明过滤原因。

## 检查顺序

```bash
kubectl get po <pod-name> -o wide
kubectl describe po <pod-name>
kubectl get ev --field-selector involvedObject.name=<pod-name> \
  --sort-by=.metadata.creationTimestamp
kubectl get no --show-labels
kubectl describe no <node-name>
```

`describe po` 中的 `FailedScheduling` 会列出每类不合格节点的数量，例如 CPU 不足、节点亲和性不匹配、不可容忍污点或拓扑分布不满足。先修正声明或容量，再等待控制器重试；删除 Pod 并不能解决相同的调度条件。

## 常见检查点

| 现象 | 核对内容 |
| --- | --- |
| `node(s) didn't match Pod's node affinity` | 标签键、值、运算符、硬约束是否正确 |
| `node(s) had taint` | 节点污点与 Pod 容忍是否完全匹配 |
| `Insufficient cpu` 或 `memory` | requests、节点可分配资源、已占用请求和配额 |
| `topology spread constraint` | 节点拓扑标签、选择器、可用域和 `maxSkew` |
| 副本集中在少数节点 | 约束是否为软规则，或其他硬规则是否缩小了候选集 |

需要验证一个临时调度假设时，可创建与原 Pod 使用相同 `spec` 的调试 Pod；不要修改生产控制器的选择器标签进行试验。完成后删除临时资源。
