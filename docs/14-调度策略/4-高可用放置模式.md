# 高可用放置模式

高可用放置不是单一字段的效果，而是副本数、故障域、资源余量、服务发现和存储语义的组合。调度器只在当前资源和规则下选择节点，无法保证跨机房链路、存储系统或应用自身具备高可用能力。

## 规则选择

| 目标 | 首选机制 | 结果 |
| --- | --- | --- |
| 仅进入特定节点池 | `nodeSelector` 或节点亲和性 | 限定硬件、合规或运行时范围 |
| 关键副本不得同机 | 硬 Pod 反亲和性 | 节点不足时保留 Pending 副本 |
| 多副本跨可用区均衡 | 拓扑分布约束 | 控制域间副本差 |
| 降低跨区访问概率 | 软 Pod 亲和性 | 尽量靠近依赖，不能保证 |
| 避免非关键节点 | 软节点亲和性 | 可在资源紧张时回退 |

## 一致的标签模型

为工作负载使用 `app.kubernetes.io/name` 作为选择器标签，为节点使用位置与能力标签。应用标签和节点标签分别由不同的责任主体维护，避免将节点名、租户名和临时发布版本混入长期选择器。

发布中需要阻止同一修订版彼此影响时，可在拓扑分布约束中使用 `matchLabelKeys`。该字段将指定标签键的值从待调度 Pod 合并到选择器，适合使用 `pod-template-hash` 区分滚动更新版本；是否使用取决于发布策略，不应把它当成普遍默认值。

## 容量边界

硬反亲和性和硬拓扑分布会降低单个故障域可承载的最大副本数。扩容前应计算每个故障域的剩余可调度资源，并至少验证一次失去一个域后的副本放置。`PodDisruptionBudget` 约束的是自愿中断，与调度放置互补，不能代替多域分布。

> [!CAUTION]
> 在只有一个可用区或少量节点的开发集群中复制生产硬约束，常会导致工作负载无法启动。验证可用区标签、节点数和资源请求后，再决定是否使用硬约束。

## 组合验证

复用[拓扑分布约束](./3-拓扑分布约束.md#跨可用区和主机分布)中的完整 `zone-spread-api.yaml` 创建 Deployment。该清单使用跨可用区的硬拓扑分布约束，随后可在测试集群观察各副本的节点与可用区分布：

```bash
kubectl create -f zone-spread-api.yaml
kubectl get po -l app.kubernetes.io/name=zone-spread-api -o wide
kubectl describe deploy zone-spread-api
```

节点缺少 `topology.kubernetes.io/zone` 标签或某个可用区没有足够容量时，硬约束会让部分副本保持 `Pending`。此时按[调度排障](./5-调度排障.md#检查顺序)查看事件，而不是临时放宽 selector 或删除已有副本。
