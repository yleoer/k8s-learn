# 镜像容器管理

containerd 可以通过 `ctr` 管理镜像和容器，也可以通过 CRI 被 kubelet 调用。学习这部分时，要始终注意 namespace、仓库协议、认证和 Kubernetes 管理边界。

## 镜像查看与拉取

查看 Kubernetes 使用的镜像：

```bash
sudo ctr -n k8s.io images ls
sudo crictl images
```

使用 `crictl` 拉取，适合验证 Kubernetes 节点拉取能力：

```bash
sudo crictl pull nginx:1.27-alpine
```

使用 `ctr` 拉取，建议写完整镜像地址：

```bash
sudo ctr -n k8s.io images pull docker.io/library/nginx:1.27-alpine
```

HTTP 仓库：

```bash
sudo ctr -n k8s.io images pull --plain-http 192.168.2.108/library/nginx:1.27-alpine
```

需要认证：

```bash
sudo ctr -n k8s.io images pull \
  --user admin:<password> \
  harbor.example.com/library/nginx:1.27-alpine
```

## 镜像删除、打标签和推送

删除镜像：

```bash
sudo ctr -n k8s.io images rm docker.io/library/nginx:1.27-alpine
sudo crictl rmi nginx:1.27-alpine
```

打标签：

```bash
sudo ctr -n k8s.io images tag \
  docker.io/library/nginx:1.27-alpine \
  harbor.example.com/library/nginx:1.27-alpine
```

推送镜像：

```bash
sudo ctr -n k8s.io images push \
  --user admin:<password> \
  harbor.example.com/library/nginx:1.27-alpine
```

生产环境不建议在节点上手工推送业务镜像，镜像应由 CI/CD 构建并推送到仓库。节点侧更多用于拉取、验证和排查。

## 导出与导入

导出：

```bash
sudo ctr -n k8s.io images export nginx-1.27-alpine.tar docker.io/library/nginx:1.27-alpine
```

导入：

```bash
sudo ctr -n k8s.io images import nginx-1.27-alpine.tar
```

导入后检查：

```bash
sudo ctr -n k8s.io images ls | grep nginx
sudo crictl images | grep nginx
```

离线环境中，需要把镜像导入到所有目标节点。Pod 使用本地镜像时通常配置：

```yaml
imagePullPolicy: IfNotPresent
```

## 挂载镜像查看文件

临时挂载镜像根文件系统：

```bash
sudo mkdir -p /mnt/nginx-rootfs
sudo ctr -n k8s.io images mount docker.io/library/nginx:1.27-alpine /mnt/nginx-rootfs
ls -lah /mnt/nginx-rootfs
```

卸载：

```bash
sudo ctr -n k8s.io images unmount /mnt/nginx-rootfs
```

这个操作适合排查镜像内文件是否存在，不应用来修改镜像内容。镜像修改应回到 Dockerfile 或构建流程。

## 容器与 task

containerd 中要区分 container 和 task：

| 对象 | 含义 |
| --- | --- |
| container | 容器元数据和配置 |
| task | 运行中的容器进程 |

查看 Kubernetes 容器：

```bash
sudo crictl ps
sudo crictl ps -a
sudo crictl pods
```

查看日志：

```bash
sudo crictl logs <container-id>
sudo crictl logs --tail=100 <container-id>
```

进入容器：

```bash
sudo crictl exec -it <container-id> sh
```

如果镜像中没有 `sh`，进入会失败。精简生产镜像没有调试工具是常见现象，应结合日志、事件和临时 debug 容器排查。

## ctr 查看容器和 task

查看 container：

```bash
sudo ctr -n k8s.io containers ls
```

查看 task：

```bash
sudo ctr -n k8s.io tasks ls
```

手工创建 containerd 容器仅用于实验理解，不建议在 Kubernetes 节点生产环境随意创建：

```bash
sudo ctr images pull docker.io/library/nginx:1.27-alpine
sudo ctr containers create docker.io/library/nginx:1.27-alpine nginx-test
sudo ctr tasks start -d nginx-test
sudo ctr tasks ls
```

停止并清理：

```bash
sudo ctr tasks kill nginx-test
sudo ctr tasks rm nginx-test
sudo ctr containers rm nginx-test
```

Kubernetes 管理的容器不要随意手工删除。应优先通过 Kubernetes 资源处理，例如删除 Pod、缩容 Deployment、drain 节点。

## 本节回顾

- Kubernetes 节点镜像管理优先关注 `k8s.io` namespace。
- `crictl pull` 最贴近 kubelet 拉取镜像的验证方式。
- container 是配置对象，task 是运行进程。
- Kubernetes 容器排查优先使用 `kubectl` 和 `crictl`。
