# 镜像容器管理

containerd 提供镜像和容器生命周期管理能力。操作时需要明确两个边界：containerd namespace 决定镜像和容器的可见范围，kubelet 管理的容器不应被底层工具随意干预。

## 镜像查看与拉取

查看 Kubernetes 使用的镜像：

```bash
# CRI 接口，接近 kubelet 视角
sudo crictl images

# containerd 原生接口，需要指定 k8s.io namespace
sudo ctr -n k8s.io images ls
```

使用 `crictl` 拉取镜像，适合验证 Kubernetes 节点的拉取能力：

```bash
sudo crictl pull nginx:1.27-alpine
```

使用 `ctr` 拉取镜像时，建议写完整镜像地址：

```bash
sudo ctr -n k8s.io images pull docker.io/library/nginx:1.27-alpine
```

HTTP 仓库需要使用 `--plain-http`：

```bash
sudo ctr -n k8s.io images pull --plain-http 192.168.2.108/library/nginx:1.27-alpine
```

需要认证的仓库可以临时传入账号密码：

```bash
sudo ctr -n k8s.io images pull \
  --user 'admin:<password>' \
  harbor.example.com/library/nginx:1.27-alpine
```

## 打标签、推送与删除

打标签：

```bash
sudo ctr -n k8s.io images tag \
  docker.io/library/nginx:1.27-alpine \
  harbor.example.com/library/nginx:1.27-alpine
```

推送：

```bash
sudo ctr -n k8s.io images push \
  --user 'admin:<password>' \
  harbor.example.com/library/nginx:1.27-alpine
```

删除镜像：

```bash
sudo ctr -n k8s.io images rm docker.io/library/nginx:1.27-alpine
sudo crictl rmi nginx:1.27-alpine
```

生产环境的镜像构建和推送应由 CI/CD 完成。节点侧命令主要用于拉取验证、离线导入和故障排查。

## 导入与导出

离线环境或单机实验场景可以使用 `ctr` 导出镜像：

```bash
sudo ctr -n k8s.io images export nginx-1.27-alpine.tar docker.io/library/nginx:1.27-alpine
```

导入到 Kubernetes 使用的 `k8s.io` namespace：

```bash
sudo ctr -n k8s.io images import nginx-1.27-alpine.tar
```

多架构镜像导入时可保留所有平台：

```bash
sudo ctr -n k8s.io images import --all-platforms app.tar
```

导入后，Pod 应配置使用本地已有镜像。以下片段是用于说明字段关系的最小 Pod 示例：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: imported-image-demo
spec:
  containers:
    - name: app
      image: app:v1
      imagePullPolicy: IfNotPresent
```

多节点集群中，每台可能运行该 Pod 的节点都必须导入同一份镜像。只在一台节点上导入，不能保证 Pod 调度到其他节点后正常启动。

## 挂载镜像查看内容

临时挂载镜像根文件系统，可用于确认文件是否存在或排查目录结构：

```bash
sudo mkdir -p /mnt/nginx-rootfs
sudo ctr -n k8s.io images mount docker.io/library/nginx:1.27-alpine /mnt/nginx-rootfs
ls -lah /mnt/nginx-rootfs
```

卸载挂载点：

```bash
sudo ctr -n k8s.io images unmount /mnt/nginx-rootfs
```

镜像挂载结果是只读视图。若需要修改镜像内容，应回到 Dockerfile 或构建流水线重新构建镜像。

## 容器与 task

containerd 区分 container 和 task 两类对象：

| 对象 | 含义 | 查看命令 |
| --- | --- | --- |
| container | 容器配置和元数据，包括镜像、入口命令、挂载、环境变量和 OCI 运行时 spec 等 | `ctr -n k8s.io containers ls` |
| task | 实际运行的容器进程，由 container 启动而来 | `ctr -n k8s.io tasks ls` |

container 是元数据对象，task 是运行中的进程对象。通常先创建 container，再从 container 启动 task。

## crictl 操作

查看 Kubernetes 容器和 Pod sandbox：

```bash
sudo crictl ps
sudo crictl ps -a
sudo crictl pods
```

查看容器日志：

```bash
sudo crictl logs <container-id>
sudo crictl logs --tail=100 <container-id>
```

进入容器：

```bash
sudo crictl exec -it <container-id> sh
```

如果镜像中没有 `sh`，例如 distroless 镜像，`exec` 会失败。精简生产镜像缺少调试工具是正常设计，排障时应结合日志、事件、探针状态和临时 debug 容器。

## ctr 实验操作

以下命令用于理解 containerd 的 container 到 task 生命周期，应在实验环境中执行，不应直接操作 kubelet 管理的业务容器。

创建容器对象：

```bash
sudo ctr images pull docker.io/library/nginx:1.27-alpine
sudo ctr containers create docker.io/library/nginx:1.27-alpine nginx-test
```

启动 task：

```bash
sudo ctr tasks start -d nginx-test
sudo ctr tasks ls
```

停止并清理：

```bash
sudo ctr tasks kill nginx-test
sudo ctr tasks rm nginx-test
sudo ctr containers rm nginx-test
```

Kubernetes 管理的容器应通过 `kubectl` 和 `crictl` 观察，通过控制器、Pod 删除、滚动重启或节点维护流程处理，不应使用 `ctr` 直接干预。
