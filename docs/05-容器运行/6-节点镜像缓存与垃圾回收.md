# 节点镜像缓存与垃圾回收

镜像从 Harbor 拉取到节点后，会进入容器运行时的本地镜像存储。节点镜像缓存可以减少重复拉取，但也会持续占用磁盘空间；kubelet 需要在镜像可复用和磁盘压力之间做回收。

## 缓存边界

Kubernetes 不把镜像缓存作为集群级资源管理。镜像缓存存在于每个节点本地，由 kubelet 通过 CRI 调用运行时拉取和删除镜像。

常见对象边界如下：

| 对象 | 所在层级 | 说明 |
| --- | --- | --- |
| Pod `image` 字段 | Kubernetes API | 记录期望运行的镜像引用 |
| `imagePullPolicy` | Pod spec | 决定 kubelet 是否尝试拉取镜像 |
| image store | 容器运行时 | 保存镜像清单、层和内容对象 |
| snapshot | 容器运行时 | 容器根文件系统快照 |
| kubelet image GC | 节点 kubelet | 根据磁盘阈值和镜像使用状态触发清理 |

Docker、containerd 和 CRI-O 的本地镜像存储互不等价。Kubernetes 使用 containerd 时，应以 `crictl images` 和 `ctr -n k8s.io images ls` 观察节点镜像，而不是以 `docker images` 判断 Pod 是否能使用镜像。

## 拉取策略与缓存

`imagePullPolicy` 会影响缓存是否被复用：

| 策略 | 行为 |
| --- | --- |
| `IfNotPresent` | 节点已有该镜像时复用本地镜像，没有时再拉取 |
| `Always` | 每次启动容器前都向 Registry 解析镜像，运行时可复用已有层 |
| `Never` | 只使用节点本地镜像，本地没有则启动失败 |

如果镜像使用 digest，kubelet 可以稳定定位内容；如果镜像只使用可移动 tag，节点上已有缓存可能导致不同节点运行的实际镜像内容不一致。生产发布应使用明确 tag，并在需要严格可追溯时记录或使用 digest。

## kubelet 镜像垃圾回收

kubelet 会根据本地镜像存储的磁盘使用情况执行镜像垃圾回收。当前 kubelet 配置中与镜像 GC 直接相关的字段包括：

| 字段 | 默认值 | 说明 |
| --- | --- | --- |
| `imageGCHighThresholdPercent` | `85` | 磁盘使用率达到该百分比后，总是运行镜像 GC |
| `imageGCLowThresholdPercent` | `80` | GC 目标是把使用率降到该百分比以下 |
| `imageMinimumGCAge` | `2m` | 未使用镜像至少达到该年龄后才可被 GC |
| `imageMaximumGCAge` | `0s` | 未使用镜像超过该年龄后可被 GC；`0s` 表示禁用按最大年龄清理 |

这几个字段属于 kubelet 配置，不应继续依赖已弃用的同名命令行参数。使用 kubeadm 集群时，通常通过 kubelet 配置文件或 kubeadm 维护的 kubelet 配置进行调整。

> [!NOTE]
> kubelet 命令行参数 `--image-gc-high-threshold` 和 `--image-gc-low-threshold` 仍然存在，但已被标记为弃用，官方要求通过 kubelet 配置文件中的 `imageGCHighThresholdPercent` 和 `imageGCLowThresholdPercent` 字段配置。

示例片段：

```yaml
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
imageGCHighThresholdPercent: 85
imageGCLowThresholdPercent: 80
imageMinimumGCAge: 2m
imageMaximumGCAge: 0s
```

该片段只说明字段关系，不应直接覆盖现有节点配置。修改 kubelet 配置前，应先确认集群部署方式、配置来源和重启影响。

## 节点磁盘压力

节点磁盘压力不只来自镜像层。kubelet 的节点压力驱逐会观察多个信号，包括 `nodefs.available`、`imagefs.available`、`containerfs.available` 以及对应 inode 指标。

这些信号的边界大致如下：

| 信号 | 说明 |
| --- | --- |
| `nodefs.available` | 节点主文件系统可用空间 |
| `imagefs.available` | 运行时镜像文件系统可用空间 |
| `containerfs.available` | 容器可写层和本地容器数据文件系统可用空间 |
| `*.inodesFree` | 对应文件系统剩余 inode |

不同运行时和节点磁盘布局会影响这些信号是否分离。独立挂载镜像数据目录可以让镜像增长与系统盘隔离，但仍需要监控 imagefs 和 containerfs 的容量与 inode。

## containerd 清理边界

containerd 管理镜像、内容对象、快照和容器 task。Kubernetes 节点上，kubelet 通过 CRI 负责业务 Pod 的生命周期；排障时可以观察底层对象，但不应随意删除 kubelet 管理中的镜像、容器和快照。

常用观察命令：

```bash
sudo crictl images
sudo crictl ps -a
sudo crictl pods
sudo ctr -n k8s.io images ls
sudo ctr -n k8s.io snapshots ls
```

手工删除镜像前，应先确认没有运行中 Pod 依赖该镜像：

```bash
sudo crictl ps -a | grep '<image>'
sudo crictl rmi <image>
```

`ctr` 能看到更底层对象，但绕过 kubelet 直接删除 snapshots、containers 或 tasks 可能破坏 kubelet 对当前 Pod 的状态认知。业务 Pod 应优先通过 `kubectl delete pod`、滚动更新、驱逐或节点维护流程处理。

## 排查记录

节点镜像缓存和垃圾回收问题通常按以下顺序排查：

1. 查看节点磁盘和 inode：

```bash
df -h
df -ih
```

2. 查看 kubelet 和 containerd 视角的镜像：

```bash
sudo crictl images
sudo ctr -n k8s.io images ls
```

3. 查看节点是否出现磁盘压力：

```bash
kubectl describe node <node-name>
kubectl get events -A --sort-by=.metadata.creationTimestamp
```

4. 查看 kubelet 与 containerd 日志：

```bash
sudo journalctl -u kubelet -n 200 --no-pager
sudo journalctl -u containerd -n 200 --no-pager
```

5. 清理前确认 Pod 调度和维护窗口：

```bash
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data
```

`drain` 会驱逐节点上的可驱逐 Pod，使用前需要确认 PDB、DaemonSet、本地数据和业务容量。只为释放镜像缓存时，优先让 kubelet GC 工作；手工清理适合磁盘压力已经影响节点稳定、且完成影响评估后的场景。

## 配置建议

- 节点镜像数据目录使用独立磁盘或独立逻辑卷，避免镜像层挤占系统盘。
- 基础镜像和业务镜像使用明确 tag，关键发布记录保存 digest。
- 对频繁发布的临时镜像设置 Harbor 标签保留策略，减少节点拉取和仓库存储压力。
- 保持 `imageGCHighThresholdPercent` 大于 `imageGCLowThresholdPercent`，不要把阈值设置到接近满盘。
- 监控 `nodefs`、`imagefs`、`containerfs` 的容量和 inode，而不是只看 `df -h /`。
- 避免使用 `ctr` 直接修改 kubelet 管理的容器和快照。
