# 客户端工具与命名空间

containerd 生态中常见三个客户端：`ctr`、`crictl`、`nerdctl`。它们都能操作运行时相关资源，但定位不同。选错工具会导致“镜像明明存在，Kubernetes 却看不到”这类困惑。

## 三个工具怎么选

| 工具 | 面向对象 | 适合场景 | 不适合场景 |
| --- | --- | --- | --- |
| `ctr` | containerd 原生接口 | 低层排查、验证 containerd 功能 | 日常运维体验一般，不完全贴近 Kubernetes |
| `crictl` | CRI 接口 | Kubernetes 节点排查 Pod、容器、镜像 | 构建镜像、Compose、本地开发 |
| `nerdctl` | Docker 兼容体验 | 用 containerd 获得类似 Docker 的操作体验 | 直接替代 Kubernetes 排障链路 |

推荐记法：

- 排查 Kubernetes 节点：优先 `kubectl` + `crictl`。
- 排查 containerd 底层：使用 `ctr`。
- 想用 Docker 风格管理 containerd：使用 `nerdctl`。

## 常用命令对照

| 操作 | Docker | nerdctl | crictl | ctr |
| --- | --- | --- | --- | --- |
| 查看版本 | `docker version` | `nerdctl version` | `crictl version` | `ctr version` |
| 查看镜像 | `docker images` | `nerdctl images` | `crictl images` | `ctr -n k8s.io images ls` |
| 拉取镜像 | `docker pull nginx` | `nerdctl pull nginx` | `crictl pull nginx` | `ctr -n k8s.io images pull docker.io/library/nginx:latest` |
| 查看容器 | `docker ps -a` | `nerdctl ps -a` | `crictl ps -a` | `ctr -n k8s.io containers ls` |
| 查看 Pod sandbox | 无直接等价 | 无直接等价 | `crictl pods` | 不推荐 |
| 查看日志 | `docker logs` | `nerdctl logs` | `crictl logs` | 不常用 |
| 进入容器 | `docker exec` | `nerdctl exec` | `crictl exec` | 不常用 |
| 删除镜像 | `docker rmi` | `nerdctl rmi` | `crictl rmi` | `ctr -n k8s.io images rm` |

## crictl 基础配置

在 Kubernetes 节点上建议配置 `/etc/crictl.yaml`：

```bash
sudo tee /etc/crictl.yaml >/dev/null <<'EOF'
runtime-endpoint: unix:///run/containerd/containerd.sock
image-endpoint: unix:///run/containerd/containerd.sock
timeout: 10
debug: false
EOF
```

验证：

```bash
sudo crictl info
sudo crictl ps -a
sudo crictl pods
sudo crictl images
```

## containerd namespace

containerd namespace 用来隔离不同上层系统管理的镜像、容器、快照等资源。它和 Kubernetes Namespace 不是一个概念。

| 使用方 | containerd namespace |
| --- | --- |
| Kubernetes CRI | `k8s.io` |
| ctr 默认操作 | `default` |
| Docker Engine | 可能使用 `moby` |
| 其他上层系统 | 自定义 namespace |

查看 namespace：

```bash
sudo ctr namespaces ls
sudo ctr ns ls -q
```

查看 Kubernetes 使用的镜像：

```bash
sudo ctr -n k8s.io images ls
sudo crictl images
```

如果镜像导入到了 `default`，Kubernetes 不一定能直接使用。离线导入给 Kubernetes 使用时，应指定 `k8s.io`：

```bash
sudo ctr -n k8s.io images import app.tar
```

## Kubernetes Namespace 与 containerd namespace

| 对比项 | Kubernetes Namespace | containerd namespace |
| --- | --- | --- |
| 所在层级 | Kubernetes API 层 | 节点运行时层 |
| 作用对象 | Pod、Service、ConfigMap 等 API 资源 | 镜像、容器、快照、任务 |
| 常用命令 | `kubectl get ns` | `ctr ns ls` |
| 典型名称 | `default`、`kube-system` | `default`、`k8s.io` |
| 是否跨节点 | 是，属于集群资源 | 否，属于单节点本地状态 |

不要看到两个地方都有 namespace 就认为它们能对应起来。`kubectl get pods -n default` 与 `ctr -n default` 完全不是一个语义。

## nerdctl 基础使用

注意：项目名是 `nerdctl`，不是 `nerdctr`。

下载安装：

```text
https://github.com/containerd/nerdctl/releases
```

基础命令：

```bash
nerdctl version
sudo nerdctl pull nginx:1.27-alpine
sudo nerdctl run -d --name web -p 8080:80 nginx:1.27-alpine
sudo nerdctl ps
sudo nerdctl logs web
sudo nerdctl rm -f web
```

查看 Kubernetes namespace：

```bash
sudo nerdctl -n k8s.io images
sudo nerdctl -n k8s.io ps -a
```

不建议用 nerdctl 直接删除 Kubernetes 管理的容器。处理业务 Pod 时，应通过 `kubectl delete pod`、`kubectl rollout restart`、`kubectl drain` 等方式。

## 常见问题

### ctr 能看到镜像，Pod 仍然拉取失败

先确认镜像是否在 `k8s.io` namespace：

```bash
sudo ctr images ls | grep nginx
sudo ctr -n k8s.io images ls | grep nginx
sudo crictl images | grep nginx
```

### crictl 连不上运行时

检查 socket 和配置：

```bash
ls -l /run/containerd/containerd.sock
sudo systemctl status containerd
cat /etc/crictl.yaml
```

### Docker 镜像和 containerd 镜像是不是一套

不是。Docker 拉取的镜像和 containerd 的 `k8s.io` namespace 不是同一套存储。Docker 中 `docker images` 能看到，不代表 `crictl images` 能看到。

## 本节回顾

- `crictl` 是 Kubernetes 节点排查首选。
- `ctr` 更底层，适合验证 containerd 本身。
- `nerdctl` 提供更接近 Docker 的体验。
- containerd namespace 是本章关键概念，Kubernetes 常用 `k8s.io`。
