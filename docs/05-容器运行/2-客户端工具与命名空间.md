# 客户端工具与命名空间

操作 containerd 常用 `ctr`、`crictl` 和 `nerdctl` 三类工具。它们面向的接口和默认视图不同，如果没有指定同一 containerd namespace，镜像和容器结果可能不一致。

## 三个工具的定位

| 工具 | 面向接口 | 典型用途 | 适用边界 |
| --- | --- | --- | --- |
| `ctr` | containerd 原生 API | 验证 containerd 底层能力、离线导入镜像 | 偏底层调试，日常运维体验较弱 |
| `crictl` | CRI | 排查 Kubernetes 节点上的 Pod、容器和镜像 | 只覆盖 CRI 能力，不用于构建镜像 |
| `nerdctl` | containerd，提供 Docker 风格 CLI | 本地开发、实验、Docker 风格操作场景 | 不用于替代 kubelet 管理业务 Pod |

选择原则如下：

- 排查 Kubernetes 节点优先使用 `crictl`，因为它通过 CRI 看到的结果更接近 kubelet 视角
- 排查 containerd 底层行为使用 `ctr`，例如验证镜像是否进入 `k8s.io` namespace
- 需要 Docker 风格操作体验时使用 `nerdctl`，例如本地运行测试容器

## 命令对照表

| 操作 | Docker | nerdctl | crictl | ctr（`k8s.io` namespace） |
| --- | --- | --- | --- | --- |
| 版本 | `docker version` | `nerdctl version` | `crictl version` | `ctr version` |
| 镜像列表 | `docker images` | `nerdctl images` | `crictl images` | `ctr -n k8s.io images ls` |
| 拉取镜像 | `docker pull` | `nerdctl pull` | `crictl pull` | `ctr -n k8s.io images pull` |
| 删除镜像 | `docker rmi` | `nerdctl rmi` | `crictl rmi` | `ctr -n k8s.io images rm` |
| 容器列表 | `docker ps -a` | `nerdctl ps -a` | `crictl ps -a` | `ctr -n k8s.io containers ls` |
| Pod sandbox | 无直接等价 | 无直接等价 | `crictl pods` | 不常用 |
| 日志 | `docker logs` | `nerdctl logs` | `crictl logs` | 不常用 |
| 进入容器 | `docker exec` | `nerdctl exec` | `crictl exec` | 不常用 |

## crictl 配置

Kubernetes 节点上建议显式配置 `/etc/crictl.yaml`，避免 `crictl` 自动探测端点造成延迟或误连。完整文件复用集群部署中的[安装并配置 crictl](../01-集群部署/2-运行时与组件安装.md#安装并配置-crictl)。

常用验证命令：

```bash
sudo crictl info       # CRI 版本和运行时能力
sudo crictl images     # Kubernetes 可见的镜像
sudo crictl ps -a      # 所有容器，包含已退出容器
sudo crictl pods       # Pod sandbox
```

## containerd namespace

containerd 使用 namespace 隔离不同上层系统的运行时资源。它与 Kubernetes namespace 是两个不同层面的概念：前者属于单节点运行时资源分组，后者属于集群 API 资源隔离。

| 上层系统 | 常见 containerd namespace |
| --- | --- |
| Kubernetes CRI | `k8s.io` |
| `ctr` 默认操作 | `default` |
| Docker Engine 内部使用 | `moby` |

查看 containerd namespace：

```bash
sudo ctr namespaces ls
sudo ctr ns ls -q
```

查看 Kubernetes 使用的镜像：

```bash
sudo ctr -n k8s.io images ls
sudo crictl images
```

`crictl images` 通过 CRI 查询，通常对应 Kubernetes 使用的 `k8s.io` namespace。若镜像只存在于 `default` namespace，kubelet 无法直接使用该镜像。

离线导入镜像到 Kubernetes 节点时，必须指定 `k8s.io` namespace：

```bash
sudo ctr -n k8s.io images import app.tar
```

随后 Pod 中应使用本地已有镜像策略。以下片段是用于说明字段关系的最小 Pod 示例：

```yaml{9}
apiVersion: v1
kind: Pod
metadata:
  name: local-image-demo
spec:
  containers:
    - name: app
      image: app:v1
      imagePullPolicy: IfNotPresent
```

## Kubernetes Namespace 与 containerd namespace

| 对比项 | Kubernetes namespace | containerd namespace |
| --- | --- | --- |
| 所在层 | 集群 API 层 | 节点运行时层 |
| 隔离对象 | Pod、Service、ConfigMap 等资源 | 镜像、容器、快照等运行时对象 |
| 查看命令 | `kubectl get ns` | `ctr ns ls` |
| 跨节点 | 集群内共享 | 每个节点独立 |
| 典型值 | `default`、`kube-system` | `default`、`k8s.io`、`moby` |

`kubectl get pods -n default` 中的 `default` 是 Kubernetes namespace；`ctr -n default images ls` 中的 `default` 是 containerd namespace，二者没有对应关系。

## nerdctl

项目名是 `nerdctl`，它为 containerd 提供接近 Docker CLI 的操作体验。

发布地址：

```text
https://github.com/containerd/nerdctl/releases
```

基础命令：

```bash
nerdctl version
sudo nerdctl pull nginx:1.31-alpine
sudo nerdctl run -d --name web -p 8080:80 nginx:1.31-alpine
sudo nerdctl ps
sudo nerdctl logs web
sudo nerdctl rm -f web
```

查看 Kubernetes 使用的 containerd namespace：

```bash
sudo nerdctl -n k8s.io images
sudo nerdctl -n k8s.io ps -a
```

> [!WARNING]
> 不建议使用 `nerdctl` 直接删除 Kubernetes 管理的容器。业务 Pod 应通过 `kubectl delete pod`、`kubectl rollout restart` 或节点维护流程处理。

## 高频排障场景

### ctr 能看到镜像但 Pod 仍然 ImagePullBackOff

先确认镜像所在的 containerd namespace：

```bash
sudo ctr images ls | grep nginx
sudo ctr -n k8s.io images ls | grep nginx
sudo crictl images | grep nginx
```

如果镜像在 `default` namespace 而不在 `k8s.io` namespace，Kubernetes 无法使用该镜像。

### crictl 无法连接运行时

检查 socket、服务和配置：

```bash
ls -l /run/containerd/containerd.sock
sudo systemctl status containerd
cat /etc/crictl.yaml
```

`runtime-endpoint` 和 `image-endpoint` 应指向实际的 containerd socket。

### Docker 镜像与 containerd 镜像不一致

Docker 使用 Docker Engine 的镜像存储，Kubernetes 节点上的 CRI 通常使用 containerd 的 `k8s.io` namespace。`docker images` 能看到镜像，并不表示 `crictl images` 一定能看到。排查 Kubernetes 镜像问题时，应以 `crictl images` 和 `crictl pull` 为准。
