# 镜像仓库配置

私有镜像仓库如果使用 HTTP，或者使用自签 HTTPS 证书但节点没有信任 CA，Kubernetes Pod 可能会出现 `ImagePullBackOff`、`x509: certificate signed by unknown authority`、`server gave HTTP response to HTTPS client` 等错误。

生产环境推荐使用可信 HTTPS 证书，或把自签 CA 分发到所有节点。`insecure` 只适合实验环境、封闭内网临时测试，不能作为生产默认方案。

## 先判断问题类型

| 错误信息 | 常见原因 | 处理方向 |
| --- | --- | --- |
| `server gave HTTP response to HTTPS client` | 仓库是 HTTP，但客户端按 HTTPS 访问 | 配置 HTTP endpoint 或 hosts.toml |
| `x509: certificate signed by unknown authority` | 自签 HTTPS 证书未被节点信任 | 分发 CA 证书到节点 |
| `authentication required` | 仓库需要认证 | 配置 imagePullSecrets 或登录凭据 |
| `no such host` | DNS 或 hosts 解析失败 | 检查域名解析和节点网络 |
| `connection refused` | 仓库地址或端口不通 | 检查 Harbor/Registry 服务和防火墙 |

## 推荐方式：hosts.toml

较新的 containerd 推荐使用 `certs.d/<registry>/hosts.toml` 管理镜像仓库访问策略。先确认主配置是否启用了 `config_path`：

```bash
grep -n "config_path" /etc/containerd/config.toml
```

containerd 1.x 常见配置：

```toml
[plugins."io.containerd.grpc.v1.cri".registry]
  config_path = "/etc/containerd/certs.d"
```

containerd 2.x 环境可能类似：

```toml
[plugins.'io.containerd.cri.v1.images'.registry]
  config_path = '/etc/containerd/certs.d'
```

实际以本机生成的 `/etc/containerd/config.toml` 为准。不要机械复制旧教程，先确认当前配置文件中使用的插件路径。

## HTTP 仓库配置

假设 Harbor 地址为 `192.168.2.108`，协议为 HTTP：

```bash
sudo mkdir -p /etc/containerd/certs.d/192.168.2.108
sudo tee /etc/containerd/certs.d/192.168.2.108/hosts.toml >/dev/null <<'EOF'
server = "http://192.168.2.108"

[host."http://192.168.2.108"]
  capabilities = ["pull", "resolve", "push"]
EOF
```

重启：

```bash
sudo systemctl restart containerd
sudo systemctl restart kubelet
```

验证：

```bash
sudo crictl pull 192.168.2.108/library/nginx:1.27-alpine
```

如果要用 `ctr` 验证 HTTP 仓库：

```bash
sudo ctr -n k8s.io images pull --plain-http 192.168.2.108/library/nginx:1.27-alpine
```

## 自签 HTTPS 仓库配置

把 Harbor CA 证书复制到所有节点：

```bash
sudo mkdir -p /etc/containerd/certs.d/harbor.example.com
sudo cp ca.crt /etc/containerd/certs.d/harbor.example.com/ca.crt
```

写入 hosts：

```bash
sudo tee /etc/containerd/certs.d/harbor.example.com/hosts.toml >/dev/null <<'EOF'
server = "https://harbor.example.com"

[host."https://harbor.example.com"]
  capabilities = ["pull", "resolve", "push"]
  ca = "/etc/containerd/certs.d/harbor.example.com/ca.crt"
EOF
```

验证：

```bash
sudo systemctl restart containerd
sudo crictl pull harbor.example.com/library/nginx:1.27-alpine
```

测试环境可临时跳过校验：

```toml
server = "https://harbor.example.com"

[host."https://harbor.example.com"]
  capabilities = ["pull", "resolve", "push"]
  skip_verify = true
```

`skip_verify = true` 会跳过 TLS 校验，存在中间人攻击风险。生产环境应下发 CA 或使用可信证书。

## 旧式 mirrors 配置

部分 containerd 1.x 文档会使用 `registry.mirrors`：

```toml
[plugins."io.containerd.grpc.v1.cri".registry.mirrors]
  [plugins."io.containerd.grpc.v1.cri".registry.mirrors."192.168.2.108"]
    endpoint = ["http://192.168.2.108"]
```

如果你的版本已经使用 `config_path`，优先把仓库配置拆到 `/etc/containerd/certs.d/<registry>/hosts.toml`，便于多仓库维护。

## 所有节点都要配置

Kubernetes Pod 可能被调度到任意 worker 节点。只在 control-plane 节点配置 Harbor 访问是不够的。需要在所有可能拉取镜像的节点上配置：

- DNS 或 `/etc/hosts`。
- CA 信任或 HTTP/insecure 策略。
- containerd registry 配置。
- 如果仓库需要认证，还需要 Kubernetes Secret 或节点侧凭据。

逐台验证：

```bash
sudo crictl pull 192.168.2.108/library/nginx:1.27-alpine
```

## imagePullSecrets

创建 Secret：

```bash
kubectl create secret docker-registry harbor-auth \
  --docker-server=harbor.example.com \
  --docker-username=<username> \
  --docker-password=<password> \
  --docker-email=devops@example.com \
  -n <namespace>
```

Deployment 中引用：

```yaml
spec:
  template:
    spec:
      imagePullSecrets:
        - name: harbor-auth
```

注意：`imagePullSecrets` 只解决认证，不解决证书信任。

## 本节回顾

- 生产环境优先使用可信 HTTPS 或分发自签 CA。
- HTTP/insecure 配置需要所有节点都设置。
- Docker 配置成功不代表 containerd 配置成功。
- `crictl pull` 是验证 Kubernetes 节点拉取镜像能力的关键命令。
