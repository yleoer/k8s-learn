# 镜像仓库配置

第 04 章说明了 Docker 侧的仓库访问配置。本章聚焦 Kubernetes 节点实际使用的 containerd：Docker 能访问仓库，并不代表 kubelet 通过 containerd 也能拉取镜像。

## 典型错误

Kubernetes Pod 拉取私有仓库镜像时，常见错误和根因如下：

| 错误信息 | 根因 | 处理方式 |
| --- | --- | --- |
| `server gave HTTP response to HTTPS client` | 仓库使用 HTTP，客户端按 HTTPS 访问 | 配置 HTTP `hosts.toml` |
| `x509: certificate signed by unknown authority` | 节点不信任仓库证书 | 分发 CA 证书或在 `hosts.toml` 中配置 `ca` |
| `authentication required` / `unauthorized` | 私有仓库需要认证或凭据错误 | 配置 `imagePullSecrets`，检查账号权限 |
| `no such host` | 节点无法解析仓库域名 | 检查 DNS 或 `/etc/hosts` |
| `connection refused` / `i/o timeout` | 仓库端口不通或网络被阻断 | 检查 Harbor 服务、端口和防火墙 |

## 推荐方式：hosts.toml

较新版本的 containerd 推荐通过 `/etc/containerd/certs.d/<registry>/hosts.toml` 管理仓库访问策略。每个仓库使用独立目录，便于分别配置 HTTP、HTTPS 证书和能力范围。

目录名必须与镜像地址中的 Registry 部分一致。带端口时需要包含端口，例如：

```text
/etc/containerd/certs.d/harbor.example.com/hosts.toml
/etc/containerd/certs.d/harbor.example.com:8443/hosts.toml
/etc/containerd/certs.d/192.168.2.108/hosts.toml
```

先确认主配置中已经启用 `config_path`：

```bash
grep -n "config_path" /etc/containerd/config.toml
```

containerd 1.x 常见配置如下：

```toml
[plugins."io.containerd.grpc.v1.cri".registry]
  config_path = "/etc/containerd/certs.d"
```

containerd 2.x 环境常见配置如下：

```toml
[plugins.'io.containerd.cri.v1.images'.registry]
  config_path = '/etc/containerd/certs.d'
```

如果未配置 `config_path`，`hosts.toml` 不会被 CRI 插件读取。修改 `/etc/containerd/config.toml` 后，需要重启 containerd；`hosts.toml` 文件本身在每次拉取时读取，目录下的变更动态生效，不需重启 daemon。

## HTTP 仓库配置

假设 Harbor 地址为 `192.168.2.108`，并使用 HTTP 协议：

```bash
sudo mkdir -p /etc/containerd/certs.d/192.168.2.108
sudo tee /etc/containerd/certs.d/192.168.2.108/hosts.toml >/dev/null <<'EOF'
server = "http://192.168.2.108"

[host."http://192.168.2.108"]
  capabilities = ["pull", "resolve", "push"]
EOF
```

重启服务：

```bash
sudo systemctl restart containerd
```

验证拉取：

```bash
sudo crictl pull 192.168.2.108/library/nginx:1.31-alpine
```

如果使用 `ctr` 直接拉取 HTTP 仓库，需要显式添加 `--plain-http`：

```bash
sudo ctr -n k8s.io images pull --plain-http 192.168.2.108/library/nginx:1.31-alpine
```

## 自签 HTTPS 仓库

自签 HTTPS 或内部 CA 签发证书的仓库，应将 CA 证书分发到所有 Kubernetes 节点，并在 `hosts.toml` 中引用。

```bash
sudo mkdir -p /etc/containerd/certs.d/harbor.example.com
sudo cp ca.crt /etc/containerd/certs.d/harbor.example.com/ca.crt
```

写入 `hosts.toml`：

```bash
sudo tee /etc/containerd/certs.d/harbor.example.com/hosts.toml >/dev/null <<'EOF'
server = "https://harbor.example.com"

[host."https://harbor.example.com"]
  capabilities = ["pull", "resolve", "push"]
  ca = "/etc/containerd/certs.d/harbor.example.com/ca.crt"
EOF
```

重启并验证：

```bash
sudo systemctl restart containerd
sudo crictl pull harbor.example.com/library/nginx:1.31-alpine
```

测试环境可以临时跳过 TLS 校验：

```toml
server = "https://harbor.example.com"

[host."https://harbor.example.com"]
  capabilities = ["pull", "resolve", "push"]
  skip_verify = true
```

`skip_verify = true` 会跳过服务端证书校验，存在中间人攻击风险。生产环境应分发 CA 证书或使用受信任证书，不应长期依赖该配置。

## 需要认证的仓库

containerd 的 `hosts.toml` 主要解决连接方式、协议和证书信任问题。Kubernetes 拉取私有仓库镜像时，认证通常由 `imagePullSecrets` 提供：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  replicas: 2
  selector:
    matchLabels:
      app: api-server
  template:
    metadata:
      labels:
        app: api-server
    spec:
      imagePullSecrets:
        - name: harbor-secret
      containers:
        - name: api-server
          image: harbor.example.com/business/api-server:v1.0.0
```

`imagePullSecrets` 解决 Pod 拉取私有镜像时的身份认证问题，相关 Secret 必须与 Pod 位于同一命名空间，并且类型通常为 `kubernetes.io/dockerconfigjson` 或 `kubernetes.io/dockercfg`。`hosts.toml` 解决节点上的 containerd 如何连接仓库的问题。证书配置错误时，即使 Secret 正确也会出现 `x509` 错误；Secret 错误时，即使证书配置正确也会出现 `unauthorized` 或 `authentication required`。

手工使用 `ctr` 验证认证仓库时，可以临时传入账号：

```bash
sudo ctr -n k8s.io images pull \
  --user '<username>:<password>' \
  harbor.example.com/business/api-server:v1.0.0
```

生产环境不建议将普通账号密码长期写入节点全局配置，应优先使用 Kubernetes Secret 或权限受限的 Robot 账号。

## 所有节点都要配置

Pod 可能被调度到任意 worker 节点。只在 control-plane 节点配置 Harbor 访问并不足够，所有可能拉取镜像的节点都需要满足以下条件：

1. DNS 或 `/etc/hosts` 能解析 Harbor 地址。
2. containerd 已读取对应的 `hosts.toml` 或系统证书信任配置。
3. Pod 所在 Kubernetes 命名空间中存在可用的 `imagePullSecrets`。
4. 节点上执行 `sudo crictl pull <image>` 能成功。

逐台验证示例：

```bash
sudo crictl pull 192.168.2.108/library/nginx:1.31-alpine
```

任一节点验证失败，Pod 调度到该节点后都可能进入 `ImagePullBackOff`。

## 旧式 mirrors 配置

部分 containerd 1.x 文档使用 `registry.mirrors` 配置仓库地址：

```toml
[plugins."io.containerd.grpc.v1.cri".registry.mirrors."192.168.2.108"]
  endpoint = ["http://192.168.2.108"]
```

旧式 `mirrors` 配置不便于按仓库独立管理 TLS、CA 和跳过校验策略，后续维护成本更高。`registry.mirrors` 和 `registry.configs` 自 containerd v1.5 起已被标记为弃用，官方计划在 v2.4 移除。当前版本已经启用 `config_path` 时，应优先使用 `hosts.toml`。
