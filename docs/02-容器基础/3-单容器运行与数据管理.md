# 单容器运行与数据管理

容器是由镜像创建的运行实例。镜像解决“交付什么”的问题，容器解决“如何运行”的问题。本文围绕单容器的创建、资源和重启策略、观察与排障、停止清理，以及数据挂载与备份展开。

进行单容器操作时，应始终关注三个问题：容器是否启动成功，应用是否真正可用，数据是否能跨容器生命周期保留。多个关联容器的声明见[Compose 服务编排](./5-Compose服务编排.md)。

## 启动容器

前台启动临时容器：

```bash
docker run -it --rm nginx:1.31-alpine sh
```

其中，`-it` 用于创建交互式终端，`--rm` 表示容器退出后自动删除。该方式适合临时测试镜像内容，不适合运行长期服务。

后台启动并映射端口：

```bash
docker run -d --name web1 -p 8080:80 nginx:1.31-alpine
```

`-d` 是 `--detach` 的短参数，表示 Docker 在后台启动容器并立即将控制权返回给当前终端。命令成功时通常输出容器 ID；这只说明容器已被创建并启动，不能替代后续的状态、日志和服务访问验证。

`-p 8080:80` 表示将宿主机 `8080` 端口映射到容器内 `80` 端口，格式为 `宿主机端口:容器端口`。

验证容器状态和服务访问：

```bash
docker ps
curl http://127.0.0.1:8080
```

如果只希望服务监听在本机回环地址，可以显式绑定地址：

```bash
docker run -d --name web-local -p 127.0.0.1:8081:80 nginx:1.31-alpine
```

默认的 `-p 8080:80` 通常会监听所有网卡地址（IPv4 0.0.0.0 及 IPv6 [::]）。生产环境中应结合防火墙、安全组和反向代理策略，避免将调试端口直接暴露到公网。

## 资源限制与重启策略

长期运行的容器应设置资源上限和重启策略：

```bash
docker run -d --name web2 \
  --restart unless-stopped \
  --memory 256m \
  --cpus 0.5 \
  --pids-limit 256 \
  -p 8082:80 \
  nginx:1.31-alpine
```

常用参数如下：

| 参数                         | 作用                           |
|----------------------------|------------------------------|
| `--restart unless-stopped` | Docker 启动后自动拉起容器，手动停止除外      |
| `--memory` / `-m`          | 设置内存上限，如 `256m`、`1g`         |
| `--cpus`                   | 设置可使用的 CPU 核心数上限，如 `0.5`、`2` |
| `--pids-limit`             | 限制容器内进程数量上限，防止进程泄漏耗尽宿主机资源    |

> [!NOTE]
> 宿主机关闭 swap 时（Kubernetes 节点的常见配置），`--memory` 就是容器内存的硬上限，超限进程会被内核 OOM Kill，可通过 `docker inspect` 输出中的 `State.OOMKilled` 确认。

查看实时资源使用情况：

```bash
docker stats web2
```

这些参数与 Kubernetes Pod 中的 `resources.requests` 和 `resources.limits` 概念紧密相关。Docker 中先理解资源上限，有助于后续理解 Pod 为什么会出现 OOMKilled、CPU throttling 等现象。

## 查看容器状态

查看运行中容器：

```bash
docker ps
```

查看所有容器，包括已退出容器：

```bash
docker ps -a
```

常用过滤和简化输出：

```bash
docker ps -q
docker ps --filter name=web
docker ps --filter status=exited
```

输出中可以看到 `CONTAINER ID`、`IMAGE`、`STATUS`、`CREATED` 和 `NAMES` 等字段，部分版本还会在 `STATUS` 列显示容器运行状态和健康检查结果。

::: details 输出类似如下

```text
CONTAINER ID   IMAGE               COMMAND                  CREATED          STATUS          PORTS                                     NAMES
b0f3edd89760   nginx:1.31-alpine   "/docker-entrypoint.…"   48 seconds ago   Up 47 seconds   0.0.0.0:8082->80/tcp, [::]:8082->80/tcp   web2
```

:::

查看容器详细信息：

```bash
docker inspect web2
```

排查时常关注以下字段：

| 字段                         | 含义               |
|----------------------------|------------------|
| `State.Status`             | 容器当前状态           |
| `State.ExitCode`           | 退出码，非 0 通常表示异常退出 |
| `State.OOMKilled`          | 是否因为内存不足被终止      |
| `Config.Image`             | 启动容器使用的镜像        |
| `Config.Cmd`               | 容器默认命令           |
| `NetworkSettings.Networks` | 容器网络、IP 和网关      |
| `Mounts`                   | 挂载信息             |
| `HostConfig.PortBindings`  | 端口映射信息           |

## 查看日志

查看容器日志：

```bash
docker logs web2
docker logs -f web2
docker logs --tail 10 web2
docker logs --since 10m --tail 100 --timestamps web2
```

`docker logs` 默认读取容器的标准输出和标准错误。生产环境中的应用应将日志写入 stdout/stderr，而不是只写入容器内部文件。这样 Docker、Kubernetes 和日志采集组件才能统一收集。

常用参数如下：

| 参数             | 作用            |
|----------------|---------------|
| `-f`           | 持续跟随日志输出      |
| `--tail 100`   | 只显示最后 100 行   |
| `--since 10m`  | 显示最近 10 分钟的日志 |
| `--timestamps` | 显示日志时间戳       |

如果容器反复退出，先查看 `docker ps -a` 中的状态，再查看日志。很多启动失败问题，例如配置文件错误、端口冲突、缺少环境变量、权限不足，都会在日志中留下线索。

## 进入容器

进入容器执行 Shell：

```bash
docker exec -it web2 sh
```

如果镜像内包含 bash，也可以使用：

```bash
docker exec -it web2 bash
```

常用排查命令：

```bash
docker exec web2 ps aux
docker exec web2 env
docker exec web2 ls -lah /usr/share/nginx/html
```

进入容器适合临时排查，不应把容器内手工修改配置作为长期运行方式。长期配置应通过镜像、挂载、环境变量或配置中心管理。

部分精简镜像可能不包含 `bash`、`ps`、`curl` 等工具。这并不一定是镜像问题，生产镜像通常会主动减少工具数量以降低体积和攻击面。排查时可以优先使用日志、`docker inspect`、临时调试容器或重新构建调试镜像。

## 停止和删除容器

停止和删除容器：

```bash
docker stop web2
docker kill web2
docker rm web2
docker rm -f web2
```

命令差异如下：

| 命令                                  | 行为                                                                   |
|-------------------------------------|----------------------------------------------------------------------|
| <span class="command-nowrap"><code>docker stop</code></span> | 先发送 SIGTERM（或镜像 `STOPSIGNAL` 指定的停止信号），默认 10 秒后仍未退出则发送 SIGKILL，适合优雅停止 |
| <span class="command-nowrap"><code>docker kill</code></span> | 默认直接发送 SIGKILL，适合无法正常停止的场景                                           |
| <span class="command-nowrap"><code>docker rm</code></span> | 删除已停止容器                                                              |
| <span class="command-nowrap"><code>docker rm -f</code></span> | 强制停止并删除容器                                                            |

删除所有已退出容器：

```bash
docker container prune
docker container prune -f
```

强制删除所有容器：

```bash
docker rm -f $(docker ps -aq)
```

最后一个命令影响范围极大，仅适合明确知道当前主机没有需要保留的容器时使用。生产主机上应逐个确认容器用途，避免误删仍需保留日志或数据的容器。

## 清理策略

查看 Docker 磁盘占用：

```bash
docker system df
docker system df -v
```

常见清理命令如下：

| 命令                       | 会清理什么                     | 风险                 |
|--------------------------|---------------------------|--------------------|
| <span class="command-nowrap"><code>docker container prune</code></span> | 已停止容器                     | 可能删除仍需查看日志或保留状态的容器 |
| <span class="command-nowrap"><code>docker image prune</code></span> | 悬空镜像                      | 风险较低，但仍需确认         |
| <span class="command-nowrap"><code>docker image prune -a</code></span> | 未被容器使用的镜像                 | 可能导致后续重新拉取镜像       |
| <span class="command-nowrap"><code>docker volume prune</code></span> | 未被容器引用的匿名卷；加 `-a` 后才包含命名卷 | 可能删除业务数据，执行前必须仔细确认 |
| <span class="command-nowrap"><code>docker system prune -a</code></span> | 停止容器、无用网络、构建缓存和未使用镜像      | 范围较大，不建议在生产主机随意执行  |

清理前应先确认容器、镜像和 volume 的引用关系。对于生产服务，数据清理应有备份和变更审批，不应依赖临时判断。

## 日志大小控制

使用默认 `json-file` 日志驱动时，长期运行的容器可能导致日志文件持续增长。可以在 `/etc/docker/daemon.json` 中配置日志轮转：

```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "100m",
    "max-file": "3"
  }
}
```

| 字段 | 含义 |
| --- | --- |
| <span class="command-nowrap"><code>log-driver</code></span> | Docker Daemon 为新建容器使用的默认日志驱动。`json-file` 将容器的 stdout 和 stderr 写入 Docker 管理的 JSON 日志文件。 |
| <span class="command-nowrap"><code>log-opts</code></span> | 传递给日志驱动的选项。Daemon 配置中的选项值必须写成字符串，因此 `"100m"` 和 `"3"` 都带引号。 |
| <span class="command-nowrap"><code>max-size</code></span> | 单个日志文件达到此大小后触发轮转。这里的 `100m` 表示约 100 MiB。 |
| <span class="command-nowrap"><code>max-file</code></span> | 最多保留的日志文件数量。它只有在设置 `max-size` 后才生效；这里保留当前文件及轮转文件在内的最多 3 个文件。 |

## 数据与挂载

容器删除后，其可写层会随之删除。应用二进制、默认配置和静态资源适合随镜像发布；运行时配置可使用环境变量或只读 bind mount；业务数据应使用 named volume、受管存储或外部存储。

| 数据类型 | 示例 | 推荐处理方式 |
| --- | --- | --- |
| 镜像内置文件 | 应用二进制、默认配置、静态资源 | 写入 Dockerfile，随镜像发布。 |
| 运行时配置 | nginx 配置、启动参数 | 使用环境变量、bind mount 或配置服务。 |
| 业务数据 | 数据库文件、上传文件、队列数据 | 使用 named volume、专用数据目录或外部存储。 |
| 临时文件 | 缓存、临时下载文件 | 写入临时目录，容器删除后可丢弃。 |

### bind mount

bind mount 将宿主机文件或目录挂载到容器，适合本地开发、挂载配置和由宿主机管理的静态目录。挂载点会遮蔽镜像内的同路径内容；挂载配置文件时通常应加 `:ro`：

```bash
docker run --rm \
  -v /etc/os-release:/etc/os-release:ro \
  nginx:1.31-alpine cat /etc/os-release
```

`--mount` 的字段更明确，适合复杂挂载：

```bash
docker run --rm \
  --mount type=bind,src="$PWD/html",dst=/usr/share/nginx/html,readonly \
  nginx:1.31-alpine ls -lah /usr/share/nginx/html
```

使用 bind mount 前，应确认宿主机路径存在、容器目标路径正确、文件权限允许容器进程读取，且 Docker Desktop 已允许共享该目录。

挂载宿主机目录运行 nginx，并验证容器实际看到的挂载信息：

```bash
mkdir -p ./html
echo 'hello nginx' > ./html/index.html
docker run -d --name web-bind \
  -p 8080:80 \
  -v "$PWD/html:/usr/share/nginx/html:ro" \
  nginx:1.31-alpine
curl http://127.0.0.1:8080
docker inspect web-bind --format '{{json .Mounts}}'
```

如果页面没有按预期更新，优先检查宿主机路径是否存在、容器内挂载路径是否为应用实际读取路径、文件权限是否允许容器进程读取，以及 Windows 或 macOS Docker Desktop 是否允许共享该目录。生产环境使用 bind mount 时，还应明确目录归属、权限、备份策略和变更流程，避免环境不可复现或误删宿主机文件。

### named volume

named volume 由 Docker 管理，适合需要跨容器生命周期保留的数据：

```bash
docker volume create nginx-data
docker run -d --name web-volume \
  -v nginx-data:/usr/share/nginx/html \
  nginx:1.31-alpine
docker volume inspect nginx-data
```

| 方式 | 示例 | 适合场景 |
| --- | --- | --- |
| bind mount | `-v "$PWD/html:/usr/share/nginx/html"` | 本地开发、挂载配置、明确指定宿主机目录。 |
| named volume | `-v nginx-data:/usr/share/nginx/html` | 保存运行数据，由 Docker 管理数据目录。 |
| anonymous volume | `-v /var/lib/mysql` | 临时实验或镜像默认声明，不适合作为长期依赖。 |

删除容器不会自动删除 named volume。`docker volume prune` 默认只清理未使用的匿名卷；`docker volume prune -a`、`docker volume rm` 和 `docker compose down -v` 可能删除命名卷及其数据。

与 bind mount 相比，named volume 不要求使用者直接处理宿主机上的具体路径，更适合由 Docker 管理的运行数据。涉及业务数据时，应先确认数据卷名称、引用容器、备份文件和恢复路径，再执行任何删除操作。

### 文件复制与备份

`docker cp` 适合临时取证或传入调试文件，不应代替镜像构建和配置分发：

```bash
docker cp web2:/var/log/nginx/access.log ./access.log
docker cp ./index.html web2:/usr/share/nginx/html/index.html
```

前一条命令将容器内日志复制到宿主机，后一条命令把宿主机文件复制进容器。频繁依赖 `docker cp` 修改容器内容，通常说明镜像构建、配置管理或部署流程没有标准化；长期配置应通过镜像、挂载、环境变量或编排系统配置对象管理。

可使用临时容器备份 named volume：

```bash
mkdir -p ./backup
docker run --rm \
  -v nginx-data:/data \
  -v "$PWD/backup:/backup" \
  alpine:3.23 tar czf /backup/nginx-data.tgz -C /data .

docker volume create nginx-data-restore
docker run --rm \
  -v nginx-data-restore:/data \
  -v "$PWD/backup:/backup" \
  alpine:3.23 tar xzf /backup/nginx-data.tgz -C /data
```

恢复后可启动临时容器验证新卷内容：

```bash
docker run --rm \
  -v nginx-data-restore:/data \
  alpine:3.23 ls -lah /data
```

数据库不应只依赖打包数据目录备份。应使用 `mysqldump`、`pg_dump`、物理备份、WAL 归档或存储快照等数据库自身机制，并定期验证恢复。

## 单服务部署记录

部署单机服务前，需要明确以下信息：

- 使用哪个镜像和明确版本标签。
- 容器监听端口和宿主机暴露范围。
- 是否需要挂载配置文件或持久化业务数据。
- 是否需要环境变量、启动参数、重启策略和资源限制。
- 服务启动后如何验证，并在失败时从哪里获取证据。

下面的命令复用 `html` 目录，启动后仅通过本机回环地址访问：

```bash
mkdir -p ./html
echo 'hello nginx' > ./html/index.html
docker run -d \
  --name nginx-demo \
  --restart unless-stopped \
  --memory 256m \
  --cpus 0.5 \
  -p 127.0.0.1:8080:80 \
  -v "$PWD/html:/usr/share/nginx/html:ro" \
  nginx:1.31-alpine

docker ps --filter name=nginx-demo
docker logs --tail 50 nginx-demo
curl http://127.0.0.1:8080
```

如果验证失败，应按状态、日志、端口、挂载路径、文件权限和宿主机防火墙的顺序排查。常用命令及参数见本章附录 [Docker 命令速查](./appendix-Docker命令速查.md)。

Docker 单机部署适合实验、开发、测试和少量轻量服务。服务数量较多，或需要高可用、滚动升级、自动恢复、服务发现、统一配置和弹性扩缩容时，应使用 Kubernetes 或其他编排平台。即使在单机场景中，也应使用明确版本镜像、为长期服务设置重启与资源限制、配置日志轮转、为重要数据进行备份恢复演练，并在发布端口前确认防火墙、安全组和访问来源。
