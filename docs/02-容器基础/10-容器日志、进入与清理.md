# 容器日志、进入与清理

容器运行后，常见操作是查看日志、进入容器调试、停止和删除容器。

本节重点是掌握三个动作：

- 用日志判断应用是否正常启动。
- 用 `exec` 临时进入容器排查。
- 清理容器和镜像前知道会删除什么。

## 查看日志

```bash
docker logs web1
docker logs -f web1
docker logs --tail 10 web1
docker logs -f --tail 10 web1
```

```text
$ docker logs --tail 5 web1

172.17.0.1 - - [16/Jun/2026:10:31:00 +0000] "GET / HTTP/1.1" 200 612 "-" "Mozilla/5.0..."
```

</details>

`docker logs` 默认读取容器标准输出和标准错误。生产镜像应把应用日志输出到 stdout/stderr，而不是只写在容器内部文件里。这样 Docker、Kubernetes、日志采集组件才能统一收集。

常用参数：

| 参数 | 作用 |
| --- | --- |
| `-f` | 持续跟随日志 |
| `--tail 100` | 只看最后 100 行 |
| `--since 10m` | 查看最近 10 分钟日志 |
| `--timestamps` | 显示时间戳 |

示例：

```bash
docker logs --since 10m --tail 100 --timestamps web1
```

## 进入容器

```bash
docker exec -it web1 sh
```

如果镜像内有 bash，也可以：

```bash
docker exec -it web1 bash
```

进入容器适合临时排查，不建议在容器里手工改配置作为长期运行方式。镜像和配置应该通过 Dockerfile、挂载或环境变量管理。

常用排查命令：

```bash
docker exec web1 ps aux
docker exec web1 env
docker exec web1 ls -lah /usr/share/nginx/html
```

如果镜像非常精简，可能没有 `sh`、`ps`、`curl` 等工具。生产镜像精简是好事，排查时可以通过日志、健康检查、临时调试容器或重新构建 debug 镜像处理。

## 删除容器

删除已停止容器：

```bash
docker rm web1
```

强制删除运行中的容器：

```bash
docker rm -f web1
```

删除所有已退出容器：

```bash
docker container prune
docker container prune -f
```

强制删除所有容器要特别小心：

```bash
docker rm -f $(docker ps -aq)
```

PowerShell 中可以使用：

```powershell
docker ps -aq | ForEach-Object { docker rm -f $_ }
```

## 常见清理命令

```bash
docker system prune
docker system prune -a
```

清理前建议先确认没有需要保留的容器和镜像。

| 命令 | 会清理什么 | 风险 |
| --- | --- | --- |
| `docker container prune` | 已停止容器 | 可能删除还想保留日志的容器 |
| `docker image prune` | 悬空镜像 | 风险较低，但仍需确认 |
| `docker image prune -a` | 当前没有容器使用的镜像 | 可能导致下次启动需要重新拉取镜像 |
| `docker system prune -a` | 容器、网络、悬空构建缓存、未使用镜像 | 范围较大，不建议在生产机随手执行 |
| `docker volume prune` | 未被容器引用的卷 | 可能删除业务数据，执行前必须确认 |

## 日志大小控制

如果使用默认 `json-file` 日志驱动，长时间运行的容器可能让日志文件不断增长。可以在 `/etc/docker/daemon.json` 中限制日志大小：

```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "100m",
    "max-file": "3"
  }
}
```

修改后重启 Docker：

```bash
sudo systemctl restart docker
```

已存在容器不一定自动应用新日志配置，生产环境应结合发布窗口滚动重建容器。

## 本章回顾

- 容器应用应优先把日志输出到 stdout/stderr。
- `docker exec` 适合临时排查，不应把手工修改容器作为发布方式。
- `prune` 类命令需要先理解删除范围，尤其要谨慎处理 volume。
- 日志增长需要配置轮转，否则可能占满磁盘。
