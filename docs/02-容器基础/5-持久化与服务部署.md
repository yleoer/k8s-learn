# 持久化与服务部署

容器默认文件系统是临时的。容器删除后，其可写层也会被删除。因此，运行配置、业务数据和日志归档等内容，需要通过挂载、volume 或外部存储进行管理。

本节重点说明容器数据应该放在哪里、不同挂载方式适合什么场景，以及部署 Docker 服务时如何形成可验证、可回滚、可排查的操作流程。

## 数据类型与处理方式

不同类型的数据应采用不同处理方式。不能把所有文件都放进镜像，也不能把所有数据都留在容器可写层。

| 类型 | 示例 | 推荐处理方式 |
| --- | --- | --- |
| 镜像内置文件 | 应用二进制、默认配置、静态资源 | 写入 Dockerfile，随镜像发布 |
| 运行时配置 | nginx 配置、应用配置、启动参数 | 使用 bind mount、环境变量、配置中心或编排平台配置对象 |
| 业务数据 | 数据库文件、上传文件、队列数据 | 使用 Docker volume、专用数据目录或外部存储 |
| 临时文件 | 缓存、临时下载文件、构建中间文件 | 写入临时目录，容器删除后可丢弃 |

设计容器化部署时，应优先判断数据是否需要跨容器生命周期保留。需要保留的数据不能只依赖容器可写层。

## bind mount

bind mount 将宿主机上的文件或目录挂载到容器内。它适合本地开发、挂载配置文件、挂载静态页面，以及明确希望由宿主机管理目录结构的场景。

挂载宿主机文件：

```bash
docker run -it --rm -v /etc/hosts:/etc/hosts nginx:1.27-alpine sh
cat /etc/hosts
```

如果容器内原本存在同名文件，挂载后会被宿主机文件覆盖。目录挂载也有同样特性：挂载点下原有内容会被宿主机目录视图覆盖，但并不代表镜像内文件被真正删除。

挂载宿主机目录运行 nginx：

```bash
mkdir -p ./html
echo 'hello nginx' > ./html/index.html
docker run -d --name web -p 8080:80 -v "$PWD/html:/usr/share/nginx/html" nginx:1.27-alpine
curl http://127.0.0.1:8080
```

验证挂载结果：

```bash
docker inspect web --format '{{json .Mounts}}'
```

如果页面没有按预期更新，优先检查以下内容：

- 宿主机路径是否存在，路径是否写错。
- 容器内挂载路径是否为应用实际读取路径。
- 文件权限是否允许容器进程读取。
- Windows 或 macOS Docker Desktop 是否允许共享该目录。

生产环境使用 bind mount 时，应明确目录归属、权限、备份策略和变更流程。随意挂载宿主机目录会增加误删文件、权限混乱和环境不可复现的风险。

## Docker volume

Docker volume 由 Docker 管理，适合保存容器运行数据。与 bind mount 相比，named volume 不需要直接关心宿主机上的具体路径，生命周期也更适合由 Docker 统一管理。

创建并使用 volume：

```bash
docker volume create nginx-data
docker run -d --name web-volume -v nginx-data:/usr/share/nginx/html nginx:1.27-alpine
docker volume ls
docker volume inspect nginx-data
```

bind mount 与 volume 的选择：

| 方式 | 示例 | 适合场景 |
| --- | --- | --- |
| bind mount | `-v "$PWD/html:/usr/share/nginx/html"` | 本地开发、挂载配置、明确指定宿主机目录 |
| named volume | `-v nginx-data:/usr/share/nginx/html` | 保存运行数据，让 Docker 管理数据目录 |
| anonymous volume | `-v /var/lib/mysql` | 临时实验或镜像默认声明，不建议作为长期依赖 |

删除容器不会自动删除 named volume，但执行 `docker volume rm`、`docker volume prune` 或 `docker compose down -v` 时会删除卷及其数据。涉及业务数据时，应先确认备份和恢复路径。

## 文件拷贝

从宿主机复制文件到容器：

```bash
docker cp install.sh <container-id>:/tmp/install.sh
```

从容器复制文件到宿主机：

```bash
docker cp <container-id>:/var/log/nginx/access.log ./access.log
```

`docker cp` 适合临时拷贝文件和故障排查，不适合作为长期配置分发方式。长期配置应通过镜像、挂载、环境变量、配置中心或编排系统配置对象进行管理。

如果频繁依赖 `docker cp` 修改容器内容，通常说明镜像构建、配置管理或部署流程还没有标准化。

## 备份与恢复 volume

备份 named volume 可以通过临时容器挂载同一个卷来实现：

```bash
mkdir -p ./backup
docker run --rm \
  -v nginx-data:/data \
  -v "$PWD/backup:/backup" \
  alpine \
  tar czf /backup/nginx-data.tgz -C /data .
```

恢复到新的 volume：

```bash
docker volume create nginx-data-restore
docker run --rm \
  -v nginx-data-restore:/data \
  -v "$PWD/backup:/backup" \
  alpine \
  tar xzf /backup/nginx-data.tgz -C /data
```

验证恢复结果：

```bash
docker run --rm -v nginx-data-restore:/data alpine ls -lah /data
```

对于数据库类服务，生产环境不应只依赖打包数据目录作为备份方式。应优先使用数据库自身备份工具，例如 `mysqldump`、`pg_dump`、物理备份、WAL 归档或存储快照，并定期验证恢复可用性。

## 服务部署流程

部署 Docker 服务前，应先明确以下信息：

- 使用哪个镜像和版本标签。
- 容器监听端口和宿主机暴露端口。
- 是否需要挂载配置文件。
- 业务数据是否需要持久化。
- 是否需要环境变量或启动参数。
- 是否需要设置重启策略。
- CPU、内存和日志大小是否需要限制。
- 服务启动后如何验证。

示例部署 nginx：

```bash
mkdir -p ./html
echo 'hello nginx' > ./html/index.html
docker run -d \
  --name nginx-demo \
  --restart unless-stopped \
  --memory 256m \
  --cpus 0.5 \
  -p 8080:80 \
  -v "$PWD/html:/usr/share/nginx/html" \
  nginx:1.27-alpine
```

验证服务：

```bash
docker ps --filter name=nginx-demo
docker logs --tail 50 nginx-demo
curl http://127.0.0.1:8080
```

如果验证失败，应先查看容器状态和日志，再检查端口、挂载、权限和防火墙。常用命令已整理到本章附录 [Docker 命令速查](./appendix-Docker命令速查.md)。

## 生产实践建议

Docker 单机部署适合学习、开发、测试和少量轻量服务。生产环境如果服务数量较多、需要高可用、滚动升级、自动恢复、服务发现、统一配置和弹性扩缩容，应考虑使用 Kubernetes 或其他编排平台。

即使在单机 Docker 场景中，也建议遵循以下原则：

- 使用明确版本镜像，不依赖 `latest`。
- 为长期服务设置重启策略。
- 为容器设置资源限制和日志轮转。
- 将业务数据放在 volume、专用目录或外部存储中。
- 重要数据必须有备份和恢复演练。
- 配置变更应可追踪，避免直接进入容器手工修改。
- 暴露端口前确认防火墙、安全组和访问来源。
