# 镜像管理

镜像是容器交付的核心对象。本篇记录镜像地址、Tag、digest，以及搜索、拉取、检查、打标、删除和离线传输等本地生命周期操作。

## 基础检查

开始操作镜像前，先确认 Docker Client 能够连接 Daemon，并确认镜像和容器对象可见：

```bash
docker version
docker info
docker container ls
docker image ls
```

这四条命令分别验证 Client 与 Server 是否连通、运行环境是否符合预期、容器对象和镜像对象是否可见。

## 镜像地址、Tag 与 digest

镜像地址由 Registry、命名空间、镜像名和 Tag 组成。

例如 `registry.example.com/platform/web:1.2.0` 中，`registry.example.com` 是 Registry，`platform` 是命名空间，`web` 是镜像名，`1.2.0` 是 Tag。

省略 Registry 时，Docker 默认从 Docker Hub 拉取；省略 Tag 时，默认使用 `latest`：

```bash
docker image pull redis
```

该命令通常等价于 `docker.io/library/redis:latest`。`latest` 只是普通标签，不表示最新稳定版本，也不适合作为生产发布依据。应使用明确版本标签：

```bash
docker image pull nginx:1.31-alpine
docker image pull redis:8.8-alpine
```

Tag 是可读的发布标识，可能被重新指向新的镜像内容；digest 是内容哈希，精确指向镜像。对可复现性和供应链要求较高的发布，应记录部署使用的 digest。

```bash
docker image inspect nginx:1.31-alpine --format '{{index .RepoDigests 0}}'
```

::: details 输出类似如下

```bash
$ docker image inspect nginx:1.31-alpine --format '{{index .RepoDigests 0}}'
nginx@sha256:54f2a904c251d5a34adf545a72d32515a15e08418dae0266e23be2e18c66fefa
```

:::

## 搜索与拉取

`docker search` 适合初步发现 Docker Hub 中的公开镜像：

```bash
docker search redis
```

::: details 输出类似如下

```bash
$ docker search redis
NAME                              DESCRIPTION                                       STARS     OFFICIAL
mcp/redis                         Access to Redis database operations.              14
redis                             Redis is the world’s fastest data platform f…   13601     [OK]
redis/redis-stack-server          redis-stack-server installs a Redis server w…    104
redis/redis-stack                 redis-stack installs a Redis server with add…    162
```

:::

`OFFICIAL [OK]` 表示 Docker Hub 将该镜像标记为官方维护镜像。生产环境选择镜像时，还应结合镜像说明、维护状态、支持架构、版本策略、数据目录和漏洞扫描结果判断。

该命令不能代替镜像选型。正式使用前应到 [Docker Hub 搜索页](https://hub.docker.com/search) 按镜像名和维护方检索，并在镜像详情页核对 Tag、维护状态、支持架构、版本策略、启动参数、数据目录和漏洞情况。

拉取时使用对象化命令 `docker image pull`；`docker pull` 是等价别名：

```bash
docker image pull nginx:1.31-alpine
docker image pull --platform linux/arm64 nginx:1.31-alpine
```

`--platform` 用于在多平台环境中选择目标平台。只有宿主机具备相应的运行或模拟能力时，拉取的镜像才能正常运行。

## 查看与检查

```bash
docker image ls
docker image ls nginx
docker image ls --format 'table {{.Repository}}\t{{.Tag}}\t{{.ID}}\t{{.Size}}'
docker image inspect nginx:1.31-alpine
docker image history nginx:1.31-alpine
```

- `docker image ls` 列出本地镜像引用；
- `--format` 适合稳定输出给脚本处理；
- `inspect` 用于查看镜像架构、入口命令、环境变量、工作目录和 RepoDigests；
- `history` 用于查看镜像层，辅助分析体积。

Docker Engine 29 的默认列表格式会区分本地磁盘占用与保留的压缩内容，显示列会随实际存储实现变化。脚本不应依赖默认表格列，应使用 `--format` 或 `docker image inspect`。

::: details 输出类似如下

```bash
$ docker image ls
IMAGE                ID             DISK USAGE   CONTENT SIZE   EXTRA
nginx:1.31-alpine    ea51152ef8c4   62.2MB       0B              U
redis:8-alpine       3a02d38405dc   114MB        0B
```

:::

| 字段 | 说明 |
| --- | --- |
| `IMAGE` | 镜像名称和 Tag。 |
| `ID` | 镜像短 ID，用于本地快速识别。 |
| `DISK USAGE` | 镜像在本机占用的磁盘空间，包含解压后的层数据。 |
| `CONTENT SIZE` | 本地保留的压缩镜像内容大小；未保留时可能显示 `0B`。 |
| `EXTRA` | 附加状态，例如 `U` 表示镜像仍被容器使用或引用。 |

可通过 `docker image inspect` 的 `--format` 参数只输出镜像架构；使用 `RepoDigests` 可记录精确的内容引用：

```bash
docker image inspect nginx:1.31-alpine --format '{{.Architecture}}'
```

## 打标签与删除

`docker image tag` 为同一镜像增加新引用，不会复制镜像内容：

```bash
docker image tag nginx:1.31-alpine nginx:demo
docker image tag nginx:1.31-alpine harbor.example.com/base/nginx:1.31-alpine
```

删除一个 Tag 通常只移除引用。没有任何标签或容器引用的镜像层才可能被回收：

```bash
docker image rm nginx:demo
docker image prune
docker image prune -a
```

- `docker image rm` 只删除指定引用；镜像仍被容器使用时默认会拒绝删除。
- `docker image prune` 只清理悬空镜像，`-a` 则扩大为所有未被容器使用的镜像。
- 若需要按时间限制清理范围，可使用 `docker image prune -a --filter 'until=168h'`。

> [!CAUTION]
> `docker image prune -a` 会删除所有未被容器使用的镜像，不限于悬空镜像。执行前先用 `docker image ls` 和 `docker system df` 确认可删除范围。

## 离线传输

`docker image save` 导出一个或多个本地镜像及其标签；`docker image load` 从归档中恢复镜像及标签：

```bash
docker image save -o nginx-1.31-alpine.tar nginx:1.31-alpine
docker image load -i nginx-1.31-alpine.tar
```

`save` 与 `load` 适合镜像离线分发。它们不替代镜像仓库：多人协作、权限控制、扫描、签名和版本留存仍应通过仓库完成。

归档文件较大时，`docker image save` 可将标准输出接入 `gzip`，生成更小的压缩归档：

```bash
docker image save nginx:1.31-alpine | gzip > nginx-1.31-alpine.tar.gz
docker image load -i nginx-1.31-alpine.tar.gz
```

查看压缩前后的归档大小：

```bash
ls -lh nginx-1.31-alpine.tar nginx-1.31-alpine.tar.gz
```

::: details 输出类似如下

```text
-rw-------  1 yleoer yleoer  61M Jul 15 11:24 nginx-1.31-alpine.tar
-rw-rw-r--  1 yleoer yleoer  25M Jul 15 11:25 nginx-1.31-alpine.tar.gz
```

:::

该示例中，压缩归档约为原始 tar 文件的 41%，传输量减少约 36 MiB。实际压缩率取决于镜像层中可压缩内容的比例。

`docker image load` 可以直接读取 gzip 压缩归档，不需要先手动解压。宿主机需要安装 `gzip`；导入完成后仍应使用 `docker image ls` 和 `docker image inspect` 验证 Tag、架构和 digest 是否符合预期。
