# 容器操作与排障

容器是由镜像创建的运行实例。镜像解决“交付什么”的问题，容器解决“如何运行”的问题。本文围绕容器启动、端口映射、资源限制、日志查看、进入容器、停止删除和清理策略展开。

进行容器操作记录时，应始终关注三个问题：容器是否启动成功，应用是否真正可用，异常发生时从哪里获取证据。

## 启动容器

前台启动临时容器：

```bash
docker run -it --rm nginx:1.27-alpine sh
```

其中，`-it` 用于创建交互式终端，`--rm` 表示容器退出后自动删除。该方式适合临时测试镜像内容，不适合运行长期服务。

后台启动并映射端口：

```bash
docker run -d --name web1 -p 8080:80 nginx:1.27-alpine
```

`-p 8080:80` 表示将宿主机 `8080` 端口映射到容器内 `80` 端口，格式为 `宿主机端口:容器端口`。

验证容器状态和服务访问：

```bash
docker ps
curl http://127.0.0.1:8080
```

如果只希望服务监听在本机回环地址，可以显式绑定地址：

```bash
docker run -d --name web-local -p 127.0.0.1:8081:80 nginx:1.27-alpine
```

默认的 `-p 8080:80` 通常会监听所有网卡地址（IPv4 0.0.0.0 及 IPv6 [::]）。生产环境中应结合防火墙、安全组和反向代理策略，避免将调试端口直接暴露到公网。

## 资源限制与重启策略

长期运行的容器应设置资源上限和重启策略：

```bash
docker run -d --name web2 \
  --restart unless-stopped \
  --memory 256m \
  --cpus 0.5 \
  -p 8082:80 \
  nginx:1.27-alpine
```

常用参数如下：

| 参数 | 作用 |
| --- | --- |
| `--restart unless-stopped` | Docker 启动后自动拉起容器，手动停止除外 |
| `--memory` / `-m` | 设置内存上限，如 `256m`、`1g` |

注意：Docker 默认使用 cgroups v2（取决于宿主机内核和 Docker 版本）。使用 cgroups v2 时，`--memory-swap` 的行为与 v1 不同；未显式设置 `--memory-swap` 时，swap 默认等于 memory 值（即不允许额外 swap）。
| `--cpus` | 设置可使用的 CPU 核心数上限，如 `0.5`、`2` |
| `--memory-swap` | 设置内存加 swap 的总上限，需结合宿主机策略使用 |

查看实时资源使用情况：

```bash
docker stats web2
```

这些参数与 Kubernetes Pod 中的 `resources.requests` 和 `resources.limits` 概念紧密相关。Docker 中先理解资源上限，有助于后续理解 Pod 为什么会出现 OOMKilled、CPU throttling 等现象。

## 查看容器状态

查看运行中容器：

```bash
docker ps
```

查看所有容器，包括已退出容器：

```bash
docker ps -a
```

常用过滤和简化输出：

```bash
docker ps -q
docker ps --filter name=web
docker ps --filter status=exited
```

输出中可以看到 `CONTAINER ID`、`IMAGE`、`STATUS`、`CREATED` 和 `NAMES` 等字段，部分版本还会在 `STATUS` 列显示容器运行状态和健康检查结果。

```text
CONTAINER ID   IMAGE               COMMAND                  CREATED          STATUS          PORTS                                     NAMES
88661c6589fe   nginx:1.27-alpine   "/docker-entrypoint.…"   2 seconds ago    Up 2 seconds    0.0.0.0:8080->80/tcp, [::]:8080->80/tcp   web1
```

查看容器详细信息：

```bash
docker inspect web1
```

排查时常关注以下字段：

| 字段 | 含义 |
| --- | --- |
| `State.Status` | 容器当前状态 |
| `State.ExitCode` | 退出码，非 0 通常表示异常退出 |
| `State.OOMKilled` | 是否因为内存不足被终止 |
| `Config.Image` | 启动容器使用的镜像 |
| `Config.Cmd` | 容器默认命令 |
| `NetworkSettings.Networks` | 容器网络、IP 和网关 |
| `Mounts` | 挂载信息 |
| `HostConfig.PortBindings` | 端口映射信息 |

## 查看日志

查看容器日志：

```bash
docker logs web1
docker logs -f web1
docker logs --tail 10 web1
docker logs --since 10m --tail 100 --timestamps web1
```

`docker logs` 默认读取容器的标准输出和标准错误。生产环境中的应用应将日志写入 stdout/stderr，而不是只写入容器内部文件。这样 Docker、Kubernetes 和日志采集组件才能统一收集。

常用参数如下：

| 参数 | 作用 |
| --- | --- |
| `-f` | 持续跟随日志输出 |
| `--tail 100` | 只显示最后 100 行 |
| `--since 10m` | 显示最近 10 分钟的日志 |
| `--timestamps` | 显示日志时间戳 |

如果容器反复退出，先查看 `docker ps -a` 中的状态，再查看日志。很多启动失败问题，例如配置文件错误、端口冲突、缺少环境变量、权限不足，都会在日志中留下线索。

## 进入容器

进入容器执行 Shell：

```bash
docker exec -it web1 sh
```

如果镜像内包含 bash，也可以使用：

```bash
docker exec -it web1 bash
```

常用排查命令：

```bash
docker exec web1 ps aux
docker exec web1 env
docker exec web1 ls -lah /usr/share/nginx/html
```

进入容器适合临时排查，不应把容器内手工修改配置作为长期运行方式。长期配置应通过镜像、挂载、环境变量或配置中心管理。

部分精简镜像可能不包含 `bash`、`ps`、`curl` 等工具。这并不一定是镜像问题，生产镜像通常会主动减少工具数量以降低体积和攻击面。排查时可以优先使用日志、`docker inspect`、临时调试容器或重新构建调试镜像。

## 停止和删除容器

停止和删除容器：

```bash
docker stop web1
docker kill web1
docker rm web1
docker rm -f web2
```

命令差异如下：

| 命令 | 行为 |
| --- | --- |
| `docker stop` | 先发送 SIGTERM，超时后发送 SIGKILL，适合优雅停止 |
| `docker kill` | 直接发送 SIGKILL，适合无法正常停止的场景 |
| `docker rm` | 删除已停止容器 |
| `docker rm -f` | 强制停止并删除容器 |

删除所有已退出容器：

```bash
docker container prune
docker container prune -f
```

强制删除所有容器：

```bash
docker rm -f $(docker ps -aq)
```

最后一个命令影响范围极大，仅适合明确知道当前主机没有需要保留的容器时使用。生产主机上应逐个确认容器用途，避免误删仍需保留日志或数据的容器。

## 清理策略

查看 Docker 磁盘占用：

```bash
docker system df
docker system df -v
```

常见清理命令如下：

| 命令 | 会清理什么 | 风险 |
| --- | --- | --- |
| `docker container prune` | 已停止容器 | 可能删除仍需查看日志或保留状态的容器 |
| `docker image prune` | 悬空镜像 | 风险较低，但仍需确认 |
| `docker image prune -a` | 未被容器使用的镜像 | 可能导致后续重新拉取镜像 |
| `docker volume prune` | 未被容器引用的数据卷 | 可能删除业务数据，执行前必须仔细确认 |
| `docker system prune -a` | 停止容器、无用网络、构建缓存和未使用镜像 | 范围较大，不建议在生产主机随意执行 |

清理前应先确认容器、镜像和 volume 的引用关系。对于生产服务，数据清理应有备份和变更审批，不应依赖临时判断。

## 日志大小控制

使用默认 `json-file` 日志驱动时，长期运行的容器可能导致日志文件持续增长。可以在 `/etc/docker/daemon.json` 中配置日志轮转：

```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "100m",
    "max-file": "3"
  }
}
```

修改后重启 Docker：

```bash
sudo systemctl restart docker
```

已有容器不会自动应用新的日志配置，通常需要重建容器。生产环境应结合发布窗口滚动处理，并确认重启不会影响业务。
