# 环境规划与版本选择

本节用于确立 Kubernetes 实验环境的基础规划，涵盖操作系统与组件版本选型、集群网络地址规划、节点角色划分及主机名解析配置等准备事项，为后续 kubeadm 初始化、Calico 网络插件安装、worker 节点接入及问题排查提供统一的环境基准。

## 默认环境

| 项目 | 选择 |
| --- | --- |
| 操作系统 | Ubuntu 24.04 LTS，代号 `noble` |
| Kubernetes | v1.36.x |
| 容器运行时 | containerd |
| CNI 网络插件 | Calico |
| 安装方式 | kubeadm |
| Pod 网段 | `10.244.0.0/16` |
| Service 网段 | `10.96.0.0/12` |

Kubernetes APT 仓库按小版本独立分仓，例如 `v1.36`、`v1.35`。安装 `kubelet`、`kubeadm`、`kubectl` 时，三个组件应使用同一小版本仓库，以确保版本一致性。

## 节点规划

| 角色 | 主机名 | 示例 IP | 说明 |
| --- | --- | --- | --- |
| control-plane | `master` | `192.168.2.108` | 控制平面节点 |
| worker | `worker01` | `192.168.2.109` | 工作节点 |
| worker | `worker02` | `192.168.2.110` | 工作节点 |

每台节点建议至少配备 2 核 CPU、2 GiB 内存与 30 GiB 磁盘空间，且需满足以下前提条件：节点间网络互通、主机名与 IP 地址固定不变、系统时间已同步。实验环境可先以单 control-plane 加单 worker 节点的最小拓扑完成初步验证，待流程跑通后再按需扩展为多节点或高可用架构。

## 主机名和解析配置

在各节点上分别设置对应的主机名：

```bash
sudo hostnamectl set-hostname master   # 在 control-plane 节点执行
sudo hostnamectl set-hostname worker01 # 在 worker01 节点执行
sudo hostnamectl set-hostname worker02 # 在 worker02 节点执行
```

若环境中没有内部 DNS，需在所有节点的 `/etc/hosts` 中添加静态解析条目：

```bash
sudo tee -a /etc/hosts >/dev/null <<'EOF'
192.168.2.108 master
192.168.2.109 worker01
192.168.2.110 worker02
EOF
```

完成配置后，验证网络连通性与主机名解析：

```bash
hostname
hostname -I
ping -c 3 worker01
ping -c 3 worker02
```

## 规划约定

- **Pod CIDR 与 Calico IPPool 保持一致**：`kubeadm init` 指定的 `--pod-network-cidr` 必须与 Calico IPPool 的地址范围完全匹配，否则将导致 Pod 网络不通。
- **节点通信使用稳定内网 IP**：控制平面与工作节点之间的通信应绑定固定的内网 IP，避免因 IP 变动引发集群异常。
