# 验证集群与常见排障

本文通过部署一个 nginx 测试应用，对集群调度、Pod 网络、Service 转发和资源指标 API 进行端到端验证，并汇总初始化阶段常见的排障命令与处理思路。

## 部署测试应用

```bash
kubectl create deployment sample-nginx --image=nginx:1.27
kubectl expose deployment sample-nginx --port=80 --type=NodePort
```

查看部署状态：

```bash
kubectl get deploy,pod,svc -l app=sample-nginx -o wide
```

## 访问验证

获取 NodePort，并通过节点 IP 访问：

```bash
NODE_PORT=$(kubectl get svc sample-nginx -o jsonpath='{.spec.ports[0].nodePort}')
NODE_IP=<node-ip>
curl "http://${NODE_IP}:${NODE_PORT}"
```

其中 `<node-ip>` 为任意节点的可达 IP 地址。若从集群外部访问，还需确认节点防火墙、云安全组或上层网络 ACL 已放行对应 NodePort。

也可从集群内部创建临时 curl Pod 验证 Service 转发：

```bash
kubectl run sample-curl --rm -i --restart=Never \
  --image=curlimages/curl:8.10.1 \
  -- curl -I --max-time 10 http://sample-nginx.default.svc.cluster.local
```

若 kube-proxy 显式配置了 `--nodeport-addresses` 或 `nodePortAddresses` 仅监听回环地址，才使用本地回环地址访问：

```bash
curl "http://127.0.0.1:${NODE_PORT}"
```

验证完成后清理测试资源：

```bash
kubectl delete svc sample-nginx
kubectl delete deployment sample-nginx
```

## 指标验证

## 常用检查命令

节点与 Pod 状态：

```bash
kubectl get nodes -o wide
kubectl get pods -A -o wide
kubectl describe node <node-name>
```

控制平面健康检查：

```bash
kubectl get --raw='/readyz?verbose'
kubectl get --raw='/livez?verbose'
kubectl -n kube-system get pods -o wide
kubectl -n kube-system logs -l component=kube-apiserver
```

containerd 运行状态：

```bash
sudo systemctl status containerd --no-pager
sudo crictl info
sudo crictl ps -a
sudo crictl images
```

kubelet 运行状态：

```bash
sudo systemctl status kubelet --no-pager
sudo journalctl -u kubelet -xe --no-pager
```

网络与 Calico 状态：

```bash
ip addr
ip route
sysctl net.ipv4.ip_forward
lsmod | grep br_netfilter
kubectl get pods -n calico-system -o wide
kubectl get installation default -o yaml
kubectl get ippool -o yaml
```

Metrics Server 状态：

```bash
kubectl -n kube-system get deploy metrics-server
kubectl -n kube-system logs deploy/metrics-server --tail=100
kubectl describe apiservice v1beta1.metrics.k8s.io
kubectl top nodes
```

## 高频问题

### kubeadm init 提示 swap 未关闭

首先确认当前 swap 状态及 fstab 配置：

```bash
swapon --show
grep -i swap /etc/fstab
```

关闭 swap 并注释 fstab 中的相关条目后，重新执行 kubeadm：

```bash
sudo swapoff -a
sudo vim /etc/fstab   # 注释 swap 行
```

### 节点持续处于 NotReady

常见原因包括：CNI 未安装或安装失败、`--pod-network-cidr` 与 Calico IPPool CIDR 不一致、containerd 未正常运行，以及 kubelet 异常退出。按以下顺序逐项排查：

```bash
kubectl get pods -A -o wide
kubectl describe node <node-name>
sudo systemctl status containerd --no-pager
sudo systemctl status kubelet --no-pager
sudo journalctl -u kubelet -xe --no-pager
```

### 镜像拉取失败

确认 containerd 的 registry hosts 配置目录已正确启用：

```bash
grep -n "config_path" -A 3 /etc/containerd/config.toml
sudo systemctl restart containerd
sudo crictl pull registry.k8s.io/pause:3.10.2
sudo crictl pull docker.io/library/nginx:1.27
```

若仍无法拉取，检查 DNS 解析、网络代理配置及防火墙规则，或改用可访问的镜像加速地址。

### kubelet 与容器运行时 cgroup driver 不匹配

Kubernetes 要求 kubelet 与容器运行时使用相同的 cgroup driver，推荐均使用 `systemd`。检查 containerd 配置：

```bash
grep -n "SystemdCgroup" /etc/containerd/config.toml
```

确认值为 `true`，若不符则修改后重启相关服务：

```bash
sudo systemctl restart containerd
sudo systemctl restart kubelet
```

### 测试应用无法访问

按以下步骤逐层排查：

```bash
kubectl get deploy,pod,svc -o wide
kubectl describe pod -l app=nginx
kubectl describe svc nginx
kubectl get nodes -o wide
kubectl get pods -A -o wide
sudo ss -lntup
```

常见原因包括：Pod 镜像拉取失败、节点未进入 Ready 状态、Calico 未正常运行、防火墙未放通 NodePort 端口范围，以及从外部访问时使用了错误的节点 IP。

### kubectl top 无法获取指标

先确认 Metrics Server Pod 和 APIService 状态：

```bash
kubectl -n kube-system get pod -l k8s-app=metrics-server -o wide
kubectl get apiservice v1beta1.metrics.k8s.io
kubectl describe apiservice v1beta1.metrics.k8s.io
```

继续查看日志：

```bash
kubectl -n kube-system logs deploy/metrics-server --tail=100
```

常见原因包括：Metrics Server 尚未就绪、control-plane 无法访问 Metrics Server、Metrics Server 无法访问节点 kubelet 的 `10250` 端口、kubelet serving certificate 校验失败，以及节点地址解析异常。实验环境中的 kubelet 证书问题可按集群初始化章节追加 `--kubelet-insecure-tls` 处理；生产环境应修正 kubelet 证书和节点网络连通性。

## 重置实验集群

若需在实验环境中彻底重建集群，可在相关节点上依次执行以下命令（**生产环境请勿操作**）：

```bash
sudo kubeadm reset
sudo rm -rf /etc/cni/net.d
sudo rm -rf "$HOME/.kube"
sudo systemctl restart containerd
```

该操作将清除当前节点上的 Kubernetes 配置及部分网络状态，但不会自动清理 kube-proxy 写入主机的 iptables、nftables 或 IPVS 规则。若需要彻底复原实验节点，应按当前网络模式额外清理相关规则，执行前请确认操作对象正确。

执行 `kubeadm reset` 前可先记录当前 kube-proxy 使用的镜像和代理模式：

```bash
kubectl -n kube-system get daemonset kube-proxy -o jsonpath='{.spec.template.spec.containers[0].image}{"\n"}'
kubectl -n kube-system get configmap kube-proxy -o jsonpath='{.data.config\.conf}' | grep '^mode:'
```

完成 `kubeadm reset` 且原 kube-proxy 不再运行后，在每个需要清理的 Linux 节点上，使用与集群一致的 kube-proxy 镜像执行 `--cleanup`。以下命令使用 Docker 启动 kube-proxy 容器；没有 Docker 时，应使用等价容器运行时以特权模式和 host network 启动同一镜像执行相同命令：

```bash
KUBE_PROXY_IMAGE="registry.k8s.io/kube-proxy:v1.36.1"
sudo docker run --privileged --network=host \
  -v /lib/modules:/lib/modules:ro \
  --rm "${KUBE_PROXY_IMAGE}" \
  sh -c "kube-proxy --cleanup && echo DONE"
```

命令末尾出现 `DONE` 表示 `kube-proxy --cleanup` 正常结束。不要将 `iptables -F`、`nft flush ruleset` 或 `ipvsadm -C` 作为默认清理方式；这些命令会清除主机上的非 Kubernetes 规则，可能影响宿主机防火墙、NAT 或其他本地网络配置。
