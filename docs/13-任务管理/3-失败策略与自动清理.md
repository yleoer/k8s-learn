# 失败策略与自动清理

Job 的容错由 Pod 重启策略、失败退避、Pod 失败策略、活动截止时间和替换策略共同决定。任务退出码必须准确表达业务结果，否则控制器会把实际失败当成成功，或无意义地重复不可重试错误。

## restartPolicy 与退避

`restartPolicy: Never` 会让失败容器所在 Pod 进入 `Failed`，Job 控制器再创建替代 Pod。`restartPolicy: OnFailure` 会先由 kubelet 在同一 Pod 内重启容器，容器重启次数同样可能计入 Job 失败退避。

`backoffLimit` 默认是 6。失败后的新 Pod 按指数退避创建，延迟从 10 秒开始逐步增加，上限为 6 分钟。达到失败上限后 Job 进入失败终止流程，已经失败的 Job 不会自动重新开始。

> [!TIP]
> 排查任务时通常使用 `restartPolicy: Never`，便于保留每次尝试的 Pod 状态和日志。业务稳定后再根据启动成本和日志采集方式评估 `OnFailure`。

## Pod 失败策略

`podFailurePolicy` 可以按容器退出码或 Pod 条件区分可重试与不可重试错误，并且要求 Pod 模板使用 `restartPolicy: Never`。

下面的任务遇到退出码 `42` 时立即失败，不再消耗全部重试次数：

```yaml [failure-policy-job.yaml]
apiVersion: batch/v1
kind: Job
metadata:
  name: failure-policy-demo
spec:
  backoffLimit: 4
  podFailurePolicy:
    rules:
      - action: FailJob
        onExitCodes:
          containerName: worker
          operator: In
          values: [42]
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: worker
          image: busybox:1.38
          command: ["sh", "-c", "echo 'non-retriable input error' >&2; exit 42"]
```

```bash
kubectl create -f failure-policy-job.yaml
kubectl describe job failure-policy-demo
```

规则按顺序求值，第一个匹配项生效：

| 动作          | 行为                           |
|-------------|------------------------------|
| `FailJob`   | 立即让整个 Job 进入失败终止流程           |
| `Ignore`    | 不计入 `backoffLimit`，并创建替代 Pod |
| `Count`     | 按默认方式计入失败次数                  |
| `FailIndex` | 与逐索引重试配合，让当前索引直接失败           |

没有规则匹配时使用默认计数行为。`Ignore` 适合明确的节点中断等可重试条件，不能用来掩盖持续失败，否则可能形成无界重试。

每条规则通过 `onExitCodes` 或 `onPodConditions` 匹配失败原因。前者可以限定容器名并使用 `In`、`NotIn` 判断非零退出码；后者匹配 Pod 条件的 `type` 和 `status`。Kubernetes 会为抢占、驱逐等中断设置 `DisruptionTarget=True`，可以用 `Ignore` 避免这类基础设施中断消耗重试次数；规则只在 Pod 到达 `Failed` 阶段后求值，应用仍需处理中断期间已经发生的外部副作用。

## Pod 替换策略

自 Kubernetes v1.34 起稳定的 `podReplacementPolicy` 控制何时创建替代 Pod。未设置 `podFailurePolicy` 时，默认值为 `TerminatingOrFailed`，Job 控制器看到 Pod 开始终止就可以创建替代 Pod，短时间内活动和终止中的 Pod 总数可能超过 `parallelism`。

设置 `podFailurePolicy` 后，替换策略固定为 `Failed`：控制器等旧 Pod 到达终态再创建替代 Pod，确保能根据最终状态执行失败策略。未使用失败策略的 Job 也可以显式设置 `podReplacementPolicy: Failed`，减少同一工作并发重叠。

## 活动截止时间

`activeDeadlineSeconds` 限制整个 Job 的活动时长，而不是单个 Pod：

```yaml{2-4}
spec:
  activeDeadlineSeconds: 1800
  backoffLimit: 5
  template:
```

这是 Job 字段片段，不是完整清单。到达截止时间后，控制器终止活动 Pod，并以 `DeadlineExceeded` 标记 Job 失败。活动截止时间优先于 `backoffLimit`；暂停 Job 会停止计时，恢复时重新设置 `status.startTime` 并重新计时。

Job 和 Pod 模板都存在名为 `activeDeadlineSeconds` 的字段，前者限制整个任务，后者限制单个 Pod，配置时要确认字段层级。

## 暂停与恢复

```bash
kubectl patch job parallel-work --type merge -p '{"spec":{"suspend":true}}'
kubectl get job parallel-work
kubectl patch job parallel-work --type merge -p '{"spec":{"suspend":false}}'
```

暂停会终止尚未完成的活动 Pod，已经成功的完成数会保留。恢复后控制器继续协调剩余工作，因此任务仍要能处理被终止后的重放。

## 完成后清理

Job 完成后默认会保留对象和 Pod，便于查看状态和日志。`ttlSecondsAfterFinished` 让 TTL 控制器在 Job 成功或失败后的指定秒数级联删除 Job 及其依赖对象：

```yaml{2,3}
spec:
  ttlSecondsAfterFinished: 3600
  template:
```

这是字段片段，不是完整清单。`0` 表示达到终态后立即具备删除条件，可能来不及收集日志；生产任务应根据日志汇聚、审计和排障窗口设置保留时间。

CronJob 还可以通过成功和失败历史上限清理它创建的 Job。TTL 适用于所有已结束 Job，历史上限只在对应 CronJob 内按数量保留，两者可以同时设置，先满足的清理条件生效。

## 排查顺序

```bash
kubectl describe job <job-name>
kubectl get po -l batch.kubernetes.io/job-name=<job-name> \
  --sort-by=.metadata.creationTimestamp
kubectl describe po <job-pod-name>
kubectl logs <job-pod-name> --all-containers
kubectl get ev --sort-by=.metadata.creationTimestamp
```

重点核对容器实际退出码、OOM 或驱逐条件、`backoffLimit`、截止时间、失败策略命中规则、PVC/Secret 等依赖，以及任务是否因外部系统不可用而反复执行。
