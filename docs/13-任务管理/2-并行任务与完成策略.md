# 并行任务与完成策略

Job 通过 `completions` 表达完成目标，通过 `parallelism` 控制同时运行数量。固定完成数可以使用无索引或带索引模式，外部工作队列则由任务进程自行协调工作项。

## 固定完成数

无索引模式只关心成功 Pod 的总数。下面的 Job 最多同时运行 2 个 Pod，累计 6 个成功后完成：

```yaml [parallel-job.yaml]
apiVersion: batch/v1
kind: Job
metadata:
  name: parallel-work
spec:
  completions: 6
  parallelism: 2
  completionMode: NonIndexed
  backoffLimit: 4
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: worker
          image: busybox:1.38
          command:
            - sh
            - -c
            - echo "processing on ${HOSTNAME}" && sleep 10
```

```bash
kubectl create -f parallel-job.yaml
kubectl get po -l batch.kubernetes.io/job-name=parallel-work -w
```

每次完成在无索引模式下是等价的。即使目标只需要 6 次，节点故障、Pod 删除或状态传播延迟仍可能让相同工作重复执行，任务必须能安全重试。

## Indexed Job

Indexed Job 为每个完成项分配 `0` 到 `completions - 1` 的稳定索引，适合分片计算、参数扫描和静态数据分区：

```yaml [indexed-job.yaml]
apiVersion: batch/v1
kind: Job
metadata:
  name: indexed-work
spec:
  completions: 6
  parallelism: 2
  completionMode: Indexed
  backoffLimit: 6
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: worker
          image: busybox:1.38
          command:
            - sh
            - -c
            - echo "processing shard ${JOB_COMPLETION_INDEX}" && sleep 10
```

```bash
kubectl create -f indexed-job.yaml
kubectl get po -l batch.kubernetes.io/job-name=indexed-work \
  -L batch.kubernetes.io/job-completion-index
```

索引通过 `JOB_COMPLETION_INDEX` 环境变量、`batch.kubernetes.io/job-completion-index` 标签和注解提供，Pod 主机名也会包含索引。控制器可能为同一索引启动多个 Pod，但只有第一个成功结果计入完成状态，因此以索引写外部结果时仍需幂等写入或原子提交。

## 工作队列模式

工作队列模式设置 `parallelism`，省略 `completions`。各 Pod 从消息队列、数据库或对象存储领取任务；任一 Pod 成功后，控制器不会再创建新 Pod，并等待现有 Pod 结束。

这种模式要求进程自行保证：

- 多个 Pod 不会无协调地处理同一个工作项。
- 队列为空后所有工作进程会正常退出。
- 重复领取或进程崩溃后能够重新处理。
- Job 完成前已经确认所有工作项真正结束。

Kubernetes 不提供队列协议，也不能根据业务队列深度自动判断工作完成。

## 逐索引重试

Indexed Job 可以为每个索引分别计算失败次数：

```yaml{5-7}
spec:
  completionMode: Indexed
  completions: 20
  parallelism: 4
  backoffLimitPerIndex: 2
  maxFailedIndexes: 3
  podReplacementPolicy: Failed
```

这是用于说明字段关系的 Job 片段，不是完整清单。某个索引超过 `backoffLimitPerIndex` 后会进入 `status.failedIndexes`，不会立即阻止其他索引继续；失败索引数超过 `maxFailedIndexes` 时，控制器提前终止其余 Pod 并让整个 Job 失败。即使没有超过 `maxFailedIndexes`，所有索引处理结束后只要存在失败索引，整个 Job 仍会标记为 `Failed`；该字段控制的是提前终止阈值，不是允许 Job 带失败索引成功。

逐索引重试自 Kubernetes v1.33 起稳定。设置 `backoffLimitPerIndex` 后，全局 `backoffLimit` 默认变为 `2147483647`。因此应同时设置合理的 `maxFailedIndexes`，或显式设置全局限制，避免大量索引长期重试。

## 成功策略

默认情况下，Indexed Job 要等成功数量达到 `completions`。自 Kubernetes v1.33 起稳定的 `successPolicy` 可以在部分索引或指定数量成功后提前满足成功标准，例如分布式搜索只需一个候选结果、多个区域探测只需达到法定数量。

```yaml{5-8}
spec:
  completionMode: Indexed
  completions: 10
  parallelism: 5
  successPolicy:
    rules:
      - succeededIndexes: 0-4
        succeededCount: 3
```

这只是成功策略字段片段，不是完整 Job。规则按顺序求值，任一规则满足后会终止仍在运行的 Pod；同时配置 `backoffLimit`、`podFailurePolicy` 或截止时间等终止策略时，只要终止标准成立，控制器就会忽略成功策略并让 Job 失败。提前成功会放弃未完成工作，只应用于业务语义明确允许的任务。

## 并行度边界

`parallelism` 是期望上限，不保证同时有这么多 Pod 运行。实际并行度还会受到剩余完成数、失败退避、资源不足、调度约束和控制器限流影响。

过高并行度可能压垮数据库、共享存储或外部 API。并行任务需要同时评估集群资源请求、下游限流、连接数、写入冲突和失败重放成本。
