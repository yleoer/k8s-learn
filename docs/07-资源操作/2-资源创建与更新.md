# 资源创建与更新

Kubernetes 不要求用户手工执行“启动容器、检查进程、重启失败实例”等连续步骤。用户提交资源对象的期望状态，控制器再持续将实际状态向该期望状态协调。资源清单是这份期望状态的可审阅、可版本控制表达；命令式操作适合快速验证或局部处置，但不应取代长期维护的清单。

## 资源对象与清单

一个典型的 Kubernetes 清单由以下部分组成：

| 部分 | 作用 | 由谁维护 |
| --- | --- | --- |
| `apiVersion` | 指定 API 组和版本，决定字段结构 | 用户选择受支持版本 |
| `kind` | 指定对象类型，例如 Deployment | 用户声明 |
| `metadata` | 名称、Namespace、Label、注解等标识信息 | 用户与系统共同维护 |
| `spec` | 描述期望状态 | 用户或配置管理工具维护 |
| `status` | 描述观察到的当前状态 | 控制器、kubelet 等系统组件维护 |

手写清单通常不写 `status`。它不是用户希望系统达成的配置，而是 Kubernetes 对当前运行结果的反馈；把 `kubectl get -o yaml` 的 `status` 复制回期望状态清单会混入运行时信息。

## 创建资源

快速创建简单资源时，可以使用命令式 `create`：

```bash
kubectl create deploy quick-nginx --image=nginx:1.31-alpine
kubectl create ns scratch
```

`create` 向 API Server 请求新建对象；同名对象已经存在时会返回错误。命令式创建适合短期验证，长期资源应落到清单中。`apply` 虽然也能在对象不存在时创建，但本章约定首次提交完整清单时采用 `create -f`，后续只在资源已存在且清单修改后采用 `apply -f`，从而使操作意图清晰可追溯。

## 生成并整理清单

部分资源可以先由 kubectl 生成 YAML，再整理为可维护的清单：

```bash
kubectl create deploy nginx --image=nginx:1.31-alpine --dry-run=client -o yaml
kubectl create ns dev --dry-run=client -o yaml
kubectl create job hello --image=busybox:1.38 --dry-run=client -o yaml -- echo hello
```

`--dry-run=client` 只在客户端组装对象，不请求 API Server，因此不能发现服务端准入、权限或集群策略造成的问题。下面是整理后的完整 Deployment 清单：

```yaml [nginx-deploy.yaml]
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx:1.31-alpine
```

Deployment 不直接运行容器。Deployment Controller 根据 `spec.template` 创建 ReplicaSet，再由 ReplicaSet 维护与 `spec.replicas` 一致数量的 Pod。`spec.selector.matchLabels` 用于让 ReplicaSet 识别它管理的 Pod，因此必须与 `spec.template.metadata.labels` 匹配；不匹配会导致对象无法按预期创建或管理 Pod。Deployment、ReplicaSet 与 Pod 的完整控制器关系在第 09 章展开。

## 声明式更新

首次创建及后续更新使用同一份清单：

```bash
kubectl create -f nginx-deploy.yaml
kubectl diff -f nginx-deploy.yaml
kubectl apply -f nginx-deploy.yaml
```

`kubectl diff` 将本地期望配置与集群中对应对象比较，不持久化变更，适合在更新前审阅差异。默认的 `kubectl apply` 使用客户端声明式合并机制，依据已记录的配置计算需要更新的字段；`--server-side` 则采用 API Server 的字段所有权机制，两者的冲突处理和字段管理方式不同。本章的清单沿用默认 `apply`，不要在同一对象上混用不同更新策略而不理解字段所有权。

需要验证服务端是否接受清单时，可以使用：

```bash
kubectl apply --dry-run=server -f nginx-deploy.yaml
```

`--dry-run=server` 会请求 API Server 进行认证、授权、准入和字段校验，但不会持久化对象变更。它适合验证清单是否符合当前集群策略；实际更新仍应先审阅 `diff`，再执行 `apply`。

## 局部与替换更新

少量临时调整可使用专用命令或补丁：

```bash
kubectl scale deploy nginx --replicas=3
kubectl patch deploy nginx -p '{"spec":{"replicas":2}}'
kubectl edit deploy nginx
```

| 方式 | 实际改变 | 使用边界 |
| --- | --- | --- |
| `scale` | 写入工作负载的 `spec.replicas`，控制器随后增减 Pod | 临时或明确的副本数调整 |
| `patch` | 仅提交指定字段的部分更新 | 脚本化修改少量字段；内置资源默认使用 strategic merge patch，按 Kubernetes 定义的字段和列表规则合并 |
| `edit` | 读取当前对象并在本地编辑器中更新 | 临时处置或排查，不应替代清单维护 |
| `replace` | 用完整对象定义替换服务端对象 | 可能覆盖文件中未保留的配置，不作为默认更新方式 |

`replace` 不会像声明式 `apply` 那样以字段合并为主要目标。需要使用它时，应先从集群读取当前定义、明确哪些字段会被替换，并审阅并发修改风险；不要把最小示例清单直接作为通用替换文件。

## 删除资源

```bash
kubectl delete deploy nginx
kubectl delete -f nginx-deploy.yaml
kubectl delete ns dev
```

删除请求会给对象设置删除状态，相关控制器和 finalizer 完成清理后对象才真正消失。删除 Deployment 通常会级联清理其拥有的 ReplicaSet 与 Pod；删除单个受控制器管理的 Pod 则不会缩容，因为控制器会按期望副本数补建替代 Pod。Namespace 删除会影响其中全部命名空间级对象，具体过程见[第 4 篇](./4-Namespace基本使用.md#创建查询与删除)。

> [!CAUTION]
> 删除前先确认 context、Namespace、资源类型和名称。对由 Deployment、StatefulSet 等控制器创建的 Pod，应修改工作负载清单或使用相应扩缩容入口，而不是以删除 Pod 作为缩容方式。
