# 资源查询与排障

资源状态分散在 API 对象、事件、容器日志和节点执行结果中。`kubectl get`、`describe`、`logs`、`events` 与 `explain` 面向的信息来源不同，不能互相替代。按来源逐层观察，才能判断问题位于对象定义、调度、容器运行还是应用自身。

## 资源列表与详情

```bash
kubectl get po
kubectl get deploy
kubectl get svc
kubectl get ns
kubectl get po,svc -n dev
kubectl get all -n dev
kubectl describe po <pod-name>
```

`get` 从 API Server 读取对象并以摘要列展示，适合回答“对象是否存在、数量和主要状态是什么”。`describe` 也是 kubectl 的客户端视图：它将对象的详细字段、相关事件和部分状态解释汇总为便于排查的文本，并不是一种独立的 Kubernetes 资源。

`kubectl get all` 使用的是 kubectl 定义的资源类别，只会显示一组常见工作负载和 Service 对象，不包含 ConfigMap、Secret、PVC、Ingress、NetworkPolicy 等全部资源。要确认集群支持哪些类型，应使用 `kubectl api-resources`。

## Label、Field 与排序

Label 是附在对象上的键值标记，用于表达应用、环境、组件等可筛选属性。Label selector 按这些键值选择一组对象；Service 用 selector 选择后端 Pod，工作负载控制器也依赖 selector 识别自己管理的 Pod。

```bash
kubectl get po -l app=nginx
kubectl get po -l 'app in (nginx,redis)'
kubectl get po -l 'app'
```

Field selector 不查询任意 YAML 路径，而是只支持各资源 API 明确公开的一组可筛选字段。Pod 的 `status.phase`、Event 的 `type` 是常见示例：

```bash
kubectl get po --field-selector status.phase=Running
kubectl get ev --field-selector type=Warning
kubectl get po --sort-by=.metadata.creationTimestamp
```

`--sort-by` 使用对象中的字段路径对 kubectl 已取得的列表排序；它不等同于 API Server 的 Field selector。字段是否可作为 selector，应以当前集群 API 的支持范围为准。

## 日志、事件与命令执行

```bash
kubectl logs <pod-name>
kubectl logs <pod-name> --previous
kubectl logs <pod-name> -c <container-name> --tail=100
kubectl exec -it <pod-name> -- sh
kubectl get ev --sort-by=.metadata.creationTimestamp
kubectl events --for po/<pod-name>
```

容器日志通常来自容器写入标准输出和标准错误的内容，kubectl 通过 kubelet 提供的日志接口读取。`--previous` 读取同一容器上一次已终止实例的日志，适合分析不断重启的容器；如果容器从未重启或节点上的旧日志已被清理，该选项可能没有可读内容。

Event 是控制器、调度器、kubelet 等组件写入的短期运行通知，例如调度失败、镜像拉取失败或卷挂载失败。Event 的保留时间由集群配置决定，不能替代审计日志或长期监控；它的价值在于将某个对象近期发生的集群侧动作与状态变化关联起来。

`exec` 在正在运行的容器中启动一个额外进程，适合观察环境变量、网络连通性和挂载文件，不会修复应用本身。精简镜像可能没有 shell，此时应转到[临时容器与 Pod 调试](../08-Pod入门/9-临时容器与Pod调试.md)使用 `kubectl debug`。

## API 与字段发现

```bash
kubectl api-resources
kubectl api-versions
kubectl explain po
kubectl explain po.spec.containers
kubectl explain deploy.spec.template
```

`api-resources` 列出资源端点，`api-versions` 列出 API 组和版本。`kubectl explain` 基于当前集群公开的 OpenAPI schema（机器可读的 API 字段定义）查询字段层级、类型和说明，因此能反映已安装 CRD 提供的字段，但不能代替控制器行为和运行时语义的说明。

## 通用观察顺序

发生 Pod 异常时，可沿以下路径缩小范围：

```bash
kubectl get po -o wide
kubectl describe po <pod-name>
kubectl get ev --sort-by=.metadata.creationTimestamp
kubectl logs <pod-name>
kubectl logs <pod-name> --previous
```

先通过 `get` 确认对象是否被调度和处于何种摘要状态，再从 `describe` 与 Event 判断控制面、调度或节点侧是否报告错误，最后通过日志定位容器和应用进程的失败原因。镜像、资源、生命周期和探针在第 08 章展开；Service 后端、EndpointSlice 和访问路径在第 10 章展开。
