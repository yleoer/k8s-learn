# 镜像仓库概述

镜像仓库（Registry）用于保存和分发容器镜像。开发、测试、生产环境中的镜像通过仓库流转，而不是在每台机器上手工构建和拷贝。

本章学习目标：

- 读懂完整镜像地址。
- 理解 Registry、Repository、Tag、Digest 的区别。
- 明白为什么企业环境通常需要私有仓库。
- 知道镜像标签、权限和扫描对生产发布的影响。

## 常见镜像仓库

| 镜像仓库 | 地址 | 说明 |
| --- | --- | --- |
| Harbor | <https://goharbor.io> | 企业私有仓库，支持权限、复制、扫描、清理等能力 |
| Docker Hub | <https://hub.docker.com> | Docker 官方公共仓库，基础镜像的主要来源 |
| GitHub Container Registry | <https://ghcr.io> | 与 GitHub 生态集成，适合开源项目和内部协作 |

## 镜像地址格式

完整镜像地址由仓库地址、命名空间或项目、镜像名和标签组成：

```text
registry.example.com/project/app:v1.0.0
```

| 片段 | 含义 | 示例 |
| --- | --- | --- |
| `registry.example.com` | Registry 地址 | `harbor.example.com` |
| `project` | 项目或命名空间 | `base`、`team-a` |
| `app` | 镜像名称 | `nginx`、`api-server` |
| `v1.0.0` | 标签 | `v1.0.0`、`20260616-abc123` |

常见写法：

```text
harbor.example.com/base/nginx:1.27
ghcr.io/team/app:v1.0.0
nginx:alpine
```

如果没写仓库地址，Docker 默认从 Docker Hub 拉取。没写 tag，默认使用 `latest`。

## Registry、Repository、Tag 与 Digest

这些概念容易混用，可以按层级理解：

| 概念 | 说明 | 示例 |
| --- | --- | --- |
| Registry | 镜像仓库服务地址 | `harbor.example.com` |
| Project / Namespace | 仓库中的项目或命名空间 | `base`、`team-a` |
| Repository | 一个镜像仓库条目 | `base/nginx`、`team-a/api` |
| Tag | 指向某个镜像版本的可读标签 | `1.27-alpine`、`v1.0.0` |
| Digest | 镜像内容哈希，内容不可变 | `sha256:...` |

Tag 便于人阅读和发布管理，但它可以被重新指向；Digest 代表具体内容，更适合精确追溯。生产排障时，除了记录 tag，也建议记录镜像 digest。

查看镜像 digest：

```bash
docker pull nginx:1.27-alpine
docker image inspect nginx:1.27-alpine --format '{{json .RepoDigests}}'
```

## 镜像拉取和推送流程

拉取：

```bash
docker pull nginx:alpine
```

打标签指向目标仓库：

```bash
docker tag nginx:alpine harbor.example.com/base/nginx:alpine
```

登录仓库：

```bash
docker login harbor.example.com
```

推送：

```bash
docker push harbor.example.com/base/nginx:alpine
```

## 多架构镜像在仓库中的存储

不同 CPU 架构（amd64 / arm64）需要对应的镜像。仓库可以用同一个标签管理多架构镜像。

构建并推送多架构镜像：

```bash
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t harbor.example.com/base/app:v1.0.0 \
  --push .
```

拉取时，容器运行时会根据节点架构自动选择匹配的产物，无需手动区分标签。

验证镜像包含的架构：

```bash
docker buildx imagetools inspect harbor.example.com/base/app:v1.0.0
```

输出中应同时包含 `linux/amd64` 和 `linux/arm64`。

## 私有仓库的价值

- 统一保存业务镜像，避免镜像散落在不同节点。
- 支持权限控制，区分开发、测试、生产镜像的访问范围。
- 内网拉取，减少公网带宽依赖。
- 支持镜像扫描、复制、清理和生命周期管理。
- 与 Kubernetes、CI/CD 平台直接集成。

## 生产发布中的仓库规范

| 规范 | 建议 |
| --- | --- |
| 标签策略 | 使用语义化版本、构建号或 Git SHA，不依赖 `latest` 发布生产 |
| 权限控制 | 开发、测试、生产项目分开授权，CI 使用机器人账号 |
| 证书信任 | 使用可信 HTTPS 证书，或把自签 CA 分发到所有 Docker/containerd 节点 |
| 镜像扫描 | 在进入生产项目前进行漏洞扫描，并定义阻断策略 |
| 保留策略 | 保留近期版本和关键稳定版本，定期清理无用 tag |
| 审计追踪 | 记录镜像构建来源、推送人、推送时间和 digest |

Kubernetes 中引用私有镜像时，通常还需要配置 `imagePullSecrets`。它只解决认证问题，不解决证书信任问题；如果节点不信任 Harbor 证书，Pod 仍然会拉取失败。

## 本章回顾

- 镜像仓库是镜像从构建到部署的流转中心。
- 完整镜像地址包含 registry、namespace/project、repository 和 tag。
- tag 方便管理但可变，digest 更适合精确追溯。
- 企业私有仓库的重点不只是“能推拉镜像”，还包括权限、安全、审计、清理和复制。
