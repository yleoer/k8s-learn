# 镜像仓库概述

镜像仓库（Registry）是容器镜像的存储和分发中心。第 03 章通过 `docker build` 制作出的镜像，需要推送到仓库后，才能被其他节点、Kubernetes 集群和 CI/CD 流水线统一拉取使用，避免在每台机器上手工复制镜像文件。

从镜像仓库的角度看，镜像并不是一个单独文件，而是由清单（manifest）、配置（config）和多个层（layer）组成的 OCI 制品。推送和拉取镜像时，客户端会优先复用仓库中已有的层，只上传或下载缺失部分。

## 常见镜像仓库

| 镜像仓库                      | 地址                       | 说明                                |
|---------------------------|--------------------------|-----------------------------------|
| Harbor                    | <https://goharbor.io>    | 企业私有仓库，支持权限控制、复制同步、漏洞扫描、保留策略和垃圾回收 |
| Docker Hub                | <https://hub.docker.com> | Docker 官方公共仓库，是基础镜像和开源镜像的常见来源     |
| GitHub Container Registry | <https://ghcr.io>        | 与 GitHub 生态集成，适合开源项目制品分发和团队协作     |

企业内网通常使用 Harbor 承载业务镜像，并将常用公共镜像同步到内网，降低外网依赖和拉取延迟。自 2020 年 11 月起 Docker Hub 对匿名用户和免费账号实施拉取速率限制，内网缓存 Docker Hub 镜像已从优化项变为企业环境的常见需求。

## 镜像地址格式

一个完整的镜像地址由 Registry 地址、项目或命名空间、镜像名称和标签组成：

```text
registry.example.com/project/app:v1.0.0
```

| 片段                     | 名称          | 说明                        |
|------------------------|-------------|---------------------------|
| `registry.example.com` | Registry 地址 | 仓库服务的主机名或 IP，可包含端口        |
| `project`              | 项目或命名空间     | 镜像的逻辑分组，如 `base`、`team-a` |
| `app`                  | 镜像名称        | 标识具体应用、组件或基础镜像            |
| `v1.0.0`               | Tag（标签）     | 版本标识，省略时默认为 `latest`      |

常见示例：

```text
harbor.example.com/base/nginx:1.27
ghcr.io/team/api-server:v2.0.0
nginx:1.27-alpine
```

省略 Registry 地址时，Docker 默认使用 Docker Hub；省略标签时，默认使用 `latest`。生产环境应显式写出仓库地址和稳定标签，避免镜像来源不清或版本漂移。

## Tag 与 Digest

Tag 是便于阅读和发布管理的版本标识，但它可以被重新指向新的镜像内容。例如，`v1.0.0` 或 `prod` 标签被重新推送后，同一个地址可能对应不同的镜像摘要。

Digest 是镜像内容的不可变哈希，格式如下：

```text
harbor.example.com/business/api-server@sha256:9f3c...
```

发布流程通常使用 Tag 表达业务版本，并在审计、回滚或高安全场景中记录 Digest。Kubernetes 既支持 `image: repo/app:tag`，也支持 `image: repo/app@sha256:...`。

## 镜像推送与拉取

本地镜像进入仓库通常包含打标签、登录和推送三个步骤：

```bash
# 1. 打标签，指向目标仓库
docker tag nginx:1.27-alpine harbor.example.com/base/nginx:1.27-alpine

# 2. 登录仓库
docker login harbor.example.com

# 3. 推送镜像
docker push harbor.example.com/base/nginx:1.27-alpine
```

其他节点可以从同一地址拉取镜像：

```bash
docker pull harbor.example.com/base/nginx:1.27-alpine
```

Kubernetes 部署清单也引用同一个镜像地址：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: harbor-nginx
spec:
  containers:
    - name: nginx
      image: harbor.example.com/base/nginx:1.27-alpine
```

如果仓库为私有项目，拉取方还需要具备对应项目的读取权限；Kubernetes Pod 可通过 `imagePullSecrets` 引用同一命名空间中的 `kubernetes.io/dockerconfigjson` 或 `kubernetes.io/dockercfg` 类型 Secret。

## 多架构镜像与仓库

同一标签下可以同时包含 `linux/amd64` 和 `linux/arm64` 等不同架构的镜像产物。多架构镜像本质上由镜像索引或清单列表指向多个平台的镜像清单，客户端拉取时会根据本机或节点架构自动选择匹配版本。

验证仓库中镜像包含的平台：

```bash
docker buildx imagetools inspect harbor.example.com/base/app:v1.0.0
```

输出中同时出现 `linux/amd64` 和 `linux/arm64`，表示多架构镜像已经正确存储。对于混合架构 Kubernetes 集群，业务镜像应尽量以多架构方式发布，避免调度到不同架构节点后出现拉取成功但运行失败的问题。

## 私有仓库的价值

| 能力    | 说明                               |
|-------|----------------------------------|
| 统一存储  | 避免镜像散落在节点本地，减少重复拉取和版本漂移          |
| 权限控制  | 按团队、项目和环境划分读写权限，区分开发、CI/CD 和运维账号 |
| 内网加速  | 通过内网分发镜像，降低公网带宽、限流和外部服务不可用的影响    |
| 安全治理  | 结合漏洞扫描、镜像签名和准入策略，降低高风险镜像进入生产的概率  |
| 生命周期  | 通过标签保留策略和垃圾回收清理历史版本，控制磁盘增长       |
| 多站点复制 | 在跨机房、跨区域场景下同步镜像，减少跨地域拉取延迟        |
