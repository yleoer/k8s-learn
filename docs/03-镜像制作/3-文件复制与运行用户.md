# 文件复制与运行用户

文件复制决定构建上下文中的内容如何进入镜像，运行用户决定镜像内文件权限和容器进程身份。`COPY`、`ADD`、`WORKDIR` 和 `USER` 经常一起出现，需要同时关注路径、所有权和运行时可写目录。

## COPY 与 ADD

`COPY` 和 `ADD` 都用于把文件从构建上下文放入镜像，但语义不同：**默认用 `COPY`，只在确实需要自动解压本地 tar 包或从远程 URL 获取内容时改用 `ADD`**。

### ADD：自动解压 tar 包与远程获取

`ADD` 的核心特性是：如果源文件是一个本地 tar 包（包括 `.tar`、`.tar.gz`、`.tar.xz`、`.tar.bz2`），它会被自动解压到目标路径；它也支持从远程 URL 或 Git 仓库获取内容。普通文件复制优先使用 `COPY`，避免让 Dockerfile 同时承担下载、解包和复制等多种语义。

先准备一个 tar 包：

```bash
mkdir -p webroot && echo '<h1>hello</h1>' > webroot/index.html
tar czf index.tar.gz -C webroot .
```

```dockerfile [Dockerfile]
FROM nginx:1.31-alpine
ADD ./index.tar.gz /usr/share/nginx/html/
```

构建并查看镜像内文件：

```bash
docker build -t nginx:add-tar .
docker run --rm --entrypoint sh nginx:add-tar -c 'ls -1 /usr/share/nginx/html && cat /usr/share/nginx/html/index.html'
```

`COPY` 示例的文件列表和页面内容与前面的 `ADD` 验证相同，仍为 `index.html` 和 `<h1>hello</h1>`。

构建运行后，`/usr/share/nginx/html/` 下将是解压后的 `index.html`，而非 `index.tar.gz` 文件。

### COPY：普通文件复制

```dockerfile [Dockerfile]
FROM nginx:1.31-alpine
COPY webroot/ /usr/share/nginx/html/
```

构建并查看复制结果：

```bash
docker build -t nginx:copy .
docker run --rm --entrypoint sh nginx:copy -c 'ls -1 /usr/share/nginx/html && cat /usr/share/nginx/html/index.html'
```

::: details 输出类似如下

```text
index.html
<h1>hello</h1>
```

:::

`COPY` 复制目录源时，会复制目录中的内容，而不是把源目录本身作为一层目录放入目标路径。`COPY webroot/ /usr/share/nginx/html/` 把 `webroot` 下的文件放入目标路径，不包含 `webroot` 这一层目录名。

源路径尾部斜杠不会改变 Dockerfile 的目录复制语义：

```dockerfile
COPY webroot /usr/share/nginx/html/
# 结果: /usr/share/nginx/html/...

COPY webroot/ /usr/share/nginx/html/
# 结果: /usr/share/nginx/html/...（webroot 目录名被剥离）
```

实际更容易出错的是目标路径。建议目标路径统一写为绝对路径，并按是否需要保留目录语义决定是否在末尾加 `/`。

### 文件权限控制

在复制的同时设置所有者和权限，避免后续再用 `RUN chown` 或 `RUN chmod` 产生额外的镜像层：

```dockerfile
COPY --chown=nginx:nginx webroot/ /usr/share/nginx/html/
COPY --chmod=644 index.html /usr/share/nginx/html/index.html
COPY --chown=1001:1001 --chmod=755 app /usr/local/bin/app
```

### 选择建议

- 普通文件、目录和编译产物 → `COPY`。
- 本地 tar 包确实需要解压时 → `ADD`。
- 远程 URL 下载 → 优先用 `RUN curl` 或 `RUN wget`，配合校验和验证并在同一层清理。确需使用 `ADD <url>` 时，用 BuildKit 的 `--checksum` 校验内容，例如 `ADD --checksum=sha256:<hash> https://example.com/pkg.tar.gz /tmp/`。

## .dockerignore

构建时，Docker 会把构建命令指定的目录（通常是 `.`）的全部文件作为构建上下文发给 daemon。`node_modules`、`.git`、日志文件等都会被打包传送，影响构建速度，也可能把无关文件带入缓存计算。

```text [.dockerignore]
.git
**/node_modules
**/*.log
dist
.tmp
Dockerfile
.dockerignore
.vscode
.idea
coverage
```

`.dockerignore` 放在构建上下文的根目录。它的匹配规则与 `.gitignore` 类似但并不相同：模式锚定在上下文根目录，`*.log` 只匹配根目录下的日志文件，要匹配任意层级必须写成 `**/*.log`（`node_modules` 同理）；`!` 前缀表示例外规则。

## WORKDIR：工作目录

`WORKDIR` 设置后续指令的执行路径，影响 `RUN`、`CMD`、`ENTRYPOINT`、`COPY` 和 `ADD` 的相对路径解析。

```dockerfile [Dockerfile]
FROM nginx:1.31-alpine
WORKDIR /usr/share/nginx/html
COPY webroot/ .
```

构建并查看运行时默认工作目录：

```bash
docker build -t nginx:workdir .
docker run --rm --entrypoint sh nginx:workdir -c 'pwd && ls -1 && cat index.html'
```

::: details 输出类似如下

```text
/usr/share/nginx/html
index.html
<h1>hello</h1>
```

:::

这里 `COPY webroot/ .` 的 `.` 指当前 `WORKDIR`，即 `/usr/share/nginx/html`。

`WORKDIR` 的行为：

- 如果指定的目录不存在，Docker 会自动创建（相当于隐含了一次 `RUN mkdir -p`）。
- 可以在 Dockerfile 中多次使用，后续设置基于前一次的相对路径。
- 始终建议写绝对路径，避免因上下文切换导致的路径混乱。

## USER：运行用户

`USER` 设定当前构建阶段后续指令（含 `RUN`）以及容器运行时进程使用的身份。基础镜像未声明时默认以 root 运行，生产环境应优先切换到非 root 用户。

```dockerfile [Dockerfile]
FROM alpine:3.23
RUN addgroup -S app && adduser -S app -G app
WORKDIR /app
COPY --chown=app:app app /app/app
USER app
CMD ["/app/app"]
```

要点：

- 先创建用户和组，再 `COPY` 文件并设置 `--chown`，最后通过 `USER` 切换运行身份。
- `USER app` 和 `USER 1001` 在 Dockerfile 中都能声明非 root 运行身份，区别在于镜像元数据里保存的是用户名还是数字 UID。容器运行时可以根据镜像内的 `/etc/passwd` 解析 `app`，因此本地 Docker 运行通常没有问题。
- 面向 Kubernetes 的镜像更建议写数字 UID，例如 `USER 1001:1001`。当 Pod 设置 `securityContext.runAsNonRoot: true` 且没有显式设置 `runAsUser` 时，kubelet 会根据镜像声明的用户判断容器是否会以 UID 0 运行；数字 UID 可以直接判断，用户名 `app` 只是一个字符串，kubelet 无法只凭这个字符串证明它不是 root，因此会拒绝启动容器。
- 如果镜像已经使用 `USER app`，也可以在 Pod 或容器的 `securityContext` 中显式设置非 0 的 `runAsUser`，例如 `runAsUser: 1001`。这样 kubelet 使用 Pod spec 中的数字 UID 进行校验，不再依赖镜像中的用户名字符串。
- 切换用户后，应用只能访问该用户有权限的路径。日志目录、临时文件目录、挂载的数据卷都要提前设置好权限。
- 如果 Dockerfile 先以 root 安装依赖，再通过 `USER` 切换身份，而安装目录仅 root 可写，运行时可能出现权限错误。此时应在安装阶段设置所有权，或将运行时数据写入当前用户有权限的路径。

## 完整示例：四条指令配合

以下 Dockerfile 把 COPY、WORKDIR、USER 组合起来，展示一个典型的可执行文件部署流程：

```dockerfile [Dockerfile]
FROM alpine:3.23

# 创建运行时用户
RUN addgroup -S app && adduser -S app -G app

# 设置工作目录
WORKDIR /app

# 复制二进制并同时设置权限
COPY --chown=app:app --chmod=755 server /app/server

# 切换用户
USER app

EXPOSE 8080
CMD ["/app/server"]
```

构建上下文中的 `server` 二进制（提前通过 Go、Rust 或 C 编译好的可执行文件）被放入 `/app/`，所有者为 `app`，权限为 `755`，进程以 `app` 身份运行。
