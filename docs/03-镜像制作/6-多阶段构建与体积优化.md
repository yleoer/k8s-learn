# 多阶段构建与体积优化

多阶段构建将编译工具、源码和中间产物留在构建阶段，最终只把运行时必需的文件复制到运行阶段。这是减少业务镜像体积、降低攻击面和简化运行环境的常用方法。

镜像分层和普通构建缓存已在 [镜像分层与构建缓存](./5-镜像分层与构建缓存.md) 中记录。本篇聚焦多阶段构建和最终镜像体积控制。

## 多阶段构建

多阶段 Dockerfile 可以包含多个 `FROM`。每个 `FROM` 开始一个新的构建阶段，阶段之间通过 `COPY --from=<stage>` 复制产物。

```dockerfile
FROM golang:1.26-alpine AS builder
WORKDIR /src
COPY . .
RUN go build -o /app/server .

FROM alpine:3.23
WORKDIR /app
COPY --from=builder /app/server /app/server
CMD ["/app/server"]
```

`builder` 阶段包含 Go 工具链和源码，最终阶段只保留 `/app/server`。如果不通过 `COPY --from` 显式复制，构建阶段中的文件不会进入最终镜像。

## Go 应用示例

先准备最小 Go 项目文件：

```bash
cat > go.mod <<'EOF'
module example.com/server

go 1.26
EOF

touch go.sum

cat > main.go <<'EOF'
package main

import (
	"fmt"
	"net/http"
)

func main() {
	http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
		fmt.Fprintln(w, "hello from go")
	})

	http.ListenAndServe(":8080", nil)
}
EOF
```

```dockerfile [Dockerfile]
# === 构建阶段 ===
FROM golang:1.26-alpine AS builder
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -ldflags="-s -w" -o /app/server .

# === 运行阶段 ===
FROM alpine:3.23
RUN apk add --no-cache ca-certificates
WORKDIR /app
COPY --from=builder /app/server /app/server
CMD ["/app/server"]
```

构建镜像，然后直接过滤构建阶段基础镜像和最终镜像，对比两者体积：

```bash
docker build -t server:v1.0.0 .
docker images --filter "reference=golang:*" --filter "reference=server:*"
```

::: details 输出类似如下

```text
IMAGE                   ID             DISK USAGE   CONTENT SIZE   EXTRA
golang:1.26-alpine  <image-id>        <size>             0B
server:v1.0.0           <image-id>        <size>             0B
```

:::

运行验证：

```bash
docker run -d --name go-server -p 8080:8080 server:v1.0.0
curl http://127.0.0.1:8080
docker rm -f go-server
```

::: details 输出类似如下

```text
hello from go
```

:::

输出中的体积会随基础镜像版本、目标架构和构建参数变化。多阶段构建的关键点是 Go 编译器、源码和中间缓存都留在了 `builder` 阶段，最终镜像只保留运行时所需文件。

## 关键优化点

- `CGO_ENABLED=0`：关闭 CGO，生成纯静态二进制，不依赖 glibc。
- `-ldflags="-s -w"`：去除符号表和调试信息，减少二进制体积。
- `--no-cache`：Alpine 用 `apk add --no-cache` 避免留存包索引；Debian 用 `rm -rf /var/lib/apt/lists/*`。
- `ca-certificates`：Alpine 基础镜像自带 `ca-certificates-bundle` 提供的根证书，常规 HTTPS 请求可直接工作；显式安装 `ca-certificates` 可获得 `update-ca-certificates` 工具，便于信任私有 CA。`scratch` 则完全不含根证书，需要显式复制。

## 运行阶段镜像选择

运行阶段不一定要继续使用 `alpine`。完全静态编译的二进制可以选择 `scratch` 或 `gcr.io/distroless/static`；动态链接 glibc 的程序更适合 `debian:trixie-slim` 或 `gcr.io/distroless/base`。基础镜像差异在 [基础镜像与镜像元数据](./2-基础镜像与镜像元数据.md) 中记录。

使用 `scratch` 时，最终阶段通常只复制可执行文件。完整最终阶段复用[基础镜像对比](./2-基础镜像与镜像元数据.md#基础镜像对比)中的 `scratch` 示例，此处不重复 Dockerfile。

如果程序需要 HTTPS 证书校验、时区或本地用户信息，需要从构建阶段或基础镜像中显式复制对应文件；否则运行时会因缺少这些文件而出现证书验证、时间显示或用户解析问题。

## 检查清单

体积优化相关内容可按以下项目检查：

- [ ] 使用了多阶段构建，仅保留运行时产物。
- [ ] 最终镜像不包含编译器、源码、中间缓存和包管理器索引。
- [ ] 基础镜像版本明确（不是 `latest`），体积与兼容性匹配。
- [ ] 静态二进制使用 `-ldflags="-s -w"` 或等同手段去除调试信息。
- [ ] 需要 CA 证书、时区、用户信息时，最终镜像中已包含对应文件。
- [ ] 生产镜像按运行方式配置健康检查，并优先使用非 root 用户。
