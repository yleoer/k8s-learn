# 镜像分层与构建缓存

Docker 镜像由多个只读层叠加而成。理解分层机制，才能判断哪些指令会增加镜像体积，哪些指令只写入镜像配置，以及构建缓存为什么会命中或失效。

本篇只记录普通镜像层缓存和 Dockerfile 指令顺序。BuildKit 的 `RUN --mount=type=cache` 是构建过程中的缓存挂载，单独放在 [BuildKit 构建挂载与 heredoc](./7-BuildKit构建挂载与heredoc.md) 中记录。

### 镜像层特性

- 镜像层是**只读**的。容器启动后，Docker 在镜像层之上追加一个可写层，所有写入操作发生在这里。
- 多个镜像可以**共享**相同的基础层，例如多个基于 `alpine:3.23.5` 的镜像，宿主机只需保存一份 Alpine 层。
- `docker rmi` 删除 tag 时，底层数据不会立即删除；只有当没有任何引用指向该层时，它才会被真正回收。
- `RUN`、`COPY`、`ADD` 等会产生文件系统变更的指令通常会形成镜像层；`CMD`、`EXPOSE`、`LABEL`、`ENV` 等指令主要写入镜像配置元数据。

## RUN：执行构建命令

`RUN` 在构建阶段执行命令，常用于安装依赖、创建用户、下载文件等操作。每条 `RUN` 都会基于上一层创建新的只读层。

```dockerfile [Dockerfile]
FROM alpine:3.23.5
RUN adduser -D app
```

构建并验证：

```bash
docker build -t alpine:user .
docker run --rm alpine:user cat /etc/passwd
```

::: details 输出类似如下

```text
app:x:1000:1000:Linux User,,,:/home/app:/bin/ash
```

:::

## 缓存与指令顺序

Docker 按指令顺序逐层构建，每条指令构建前检查缓存：如果前一层未变且当前指令与上次相同，就复用缓存层；否则从该层开始全部重新构建。对 `COPY` 和 `ADD`，除指令文本外还会比对所复制文件内容的校验和；指令没变但源文件内容变了，缓存同样失效。

指令顺序会直接影响构建效率，变更频繁的内容应尽量放到后面：

```dockerfile
# 先复制依赖描述文件，安装依赖（变化少，缓存可复用）
COPY package*.json ./
RUN npm ci --omit=dev

# 再复制源码（变化频繁，这一层及之后才重建）
COPY . .
```

源码改动时，`COPY package*.json` 层和 `RUN npm ci` 层都被缓存命中，只有 `COPY . .` 及之后需要重建。

## 合并在同一层操作

安装和清理必须在同一个 `RUN` 中完成，这是 Dockerfile 最常见的优化误区。

**错误写法**（apt 缓存残留在中间层，占用镜像空间）：

```dockerfile
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*
```

**正确写法**（安装和清理在同一层，不留下无用文件）：

```dockerfile
RUN apt-get update \
    && apt-get install -y --no-install-recommends ca-certificates curl \
    && rm -rf /var/lib/apt/lists/*
```

原理：每条 `RUN` 生成一个只读层。即使后面的层删除了文件，前面层中的数据仍然占据空间；删除操作只是覆盖文件视图，底层只读层本身不可变。

## 检查清单

每次提交 Dockerfile 前，分层与缓存相关内容可按以下项目检查：

- [ ] 安装和清理在同一个 `RUN` 中完成。
- [ ] 构建缓存、包管理器索引没有残留。
- [ ] 变更频繁的指令（源码 `COPY`）放在相对靠后的位置。
- [ ] 依赖描述文件先复制，依赖安装层可复用缓存。
- [ ] `.dockerignore` 排除了 `.git`、`node_modules`、日志和本地构建产物。
- [ ] 没有在最终镜像中保留调试工具、编译器和无关文档。
