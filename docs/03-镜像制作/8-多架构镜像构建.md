# 多架构镜像构建

如果 Kubernetes 集群中同时有 x86（amd64）和 ARM（arm64）节点，同一个镜像标签需要能根据节点架构自动选择匹配的产物。Docker Buildx 就是解决这个问题的工具。

## 为什么需要多架构镜像

先确认本机架构：

```bash
uname -m
```

| 输出      | 对应架构 |
| --------- | -------- |
| `x86_64`  | `amd64`  |
| `aarch64` | `arm64`  |

在 x86 机器上 `docker build` 出来的镜像默认是 amd64，直接推到 ARM 节点上运行会报 `exec format error`。

两种解决思路：

- **单架构标签**：`app:v1.0.0-amd64` 和 `app:v1.0.0-arm64` 分开构建，部署时按节点架构手动选标签——简单但繁琐。
- **多架构镜像（推荐）**：同一个 `app:v1.0.0` 标签下同时包含 amd64 和 arm64 产物，运行时自动匹配——对部署方透明。

## Buildx 环境准备

检查当前构建器：

```bash
docker buildx ls
```

使用 `docker-container` driver 创建独立构建器，便于启用 BuildKit 的多平台构建能力：

```bash
docker buildx create \
  --name mybuilder \
  --use \
  --platform linux/amd64,linux/arm64 \
  --driver docker-container
```

初始化并验证：

```bash
docker buildx inspect --bootstrap
```

输出中的 `Platforms` 应包含 `linux/amd64` 和 `linux/arm64`。

## 安装跨架构模拟支持

在 x86 机器上构建 arm64 镜像时，需要对 arm64 指令进行模拟执行。推荐使用 `tonistiigi/binfmt` 镜像注册 QEMU 模拟器：

```bash
docker run --privileged --rm tonistiigi/binfmt --install all
```

`binfmt_misc` 是 Linux 内核机制，能够识别非本机架构的 ELF 二进制文件头，并自动交给 QEMU 解释执行。执行 `--install all` 后，宿主机即可处理 arm64 等非本机指令，`RUN` 才能在跨架构构建过程中正常执行。

验证安装：

```bash
ls /proc/sys/fs/binfmt_misc/
```

::: details binfmt_misc 输出类似如下

```text
$ ls /proc/sys/fs/binfmt_misc/
python3.12  qemu-aarch64  qemu-arm  qemu-loongarch64  qemu-mips64  qemu-mips64el  qemu-ppc64le  qemu-riscv64  qemu-s390x  register  status
```

:::

## 构建并推送多架构镜像

```bash
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t harbor.example.com/team/app:v1.0.0 \
  --push .
```

`--push` 直接将多架构产物推送到仓库，本地 `docker images` 不会显示该镜像，因为它由 Buildx 管理，不会导入本地 Docker 镜像存储。

如果只想构建单一架构并加载到本地：

```bash
docker buildx build \
  --platform linux/arm64 \
  -t app:arm64-local \
  --load .
```

在传统 graphdriver 镜像存储下，`--load` 只适用于单架构构建；启用 containerd image store 的引擎已支持多架构 `--load`，Docker Engine 29 起全新安装默认使用该存储，从旧版本升级的引擎仍沿用原存储。多架构产物通常使用 `--push` 推送到仓库，或使用 OCI / Docker tar 等输出方式保存为归档。

## 验证多架构镜像

```bash
docker buildx imagetools inspect harbor.example.com/team/app:v1.0.0
```

::: details imagetools inspect 输出类似如下

```text

Manifests:
  Name:        harbor.example.com/team/app:v1.0.0@sha256:abc123...
  MediaType:   application/vnd.oci.image.manifest.v1+json
  Platform:    linux/amd64

  Name:        harbor.example.com/team/app:v1.0.0@sha256:def456...
  MediaType:   application/vnd.oci.image.manifest.v1+json
  Platform:    linux/arm64
```

:::

输出中的 `Manifests` 列表是核心——每个条目代表一个架构的独立镜像，容器运行时会根据节点架构自动选择匹配的 manifest。Buildx 推送时默认还会附加 provenance 构建证明，清单中可能因此多出 `unknown/unknown` 平台的附加条目，属于正常现象；个别不兼容该格式的仓库可在构建时加 `--provenance=false` 关闭。

## 常见问题

**基础镜像不支持目标架构**

不是所有镜像都提供多架构版本。构建前先确认基础镜像的可用架构：

```bash
docker buildx imagetools inspect nginx:1.31-alpine
```

输出中应能看到目标的 `Platform`。

**二进制文件架构不匹配**

多阶段构建中，如果向镜像 COPY 了预编译的二进制文件，必须确保它与目标架构一致。最简单的做法是在多阶段构建中统一使用 `--platform`，让编译和运行在同一架构下完成：

```dockerfile [Dockerfile]
FROM --platform=$BUILDPLATFORM golang:1.26-alpine AS builder
ARG TARGETOS
ARG TARGETARCH
WORKDIR /src
COPY . .
RUN CGO_ENABLED=0 GOOS=$TARGETOS GOARCH=$TARGETARCH go build -o /app/server .

FROM alpine:3.23
COPY --from=builder /app/server /app/server
CMD ["/app/server"]
```

Buildx 会自动注入 `BUILDPLATFORM`（构建机平台，如 `linux/amd64`）、`TARGETOS` / `TARGETARCH`（目标平台的操作系统与架构）等全局 ARG。这些 ARG 在 `FROM` 之后默认不可见，需要像示例那样在阶段内用 `ARG` 声明后才能使用。

**`--load` 报错**

传统 graphdriver 镜像存储下 `--load` 只支持单架构，同时传入多个 `--platform` 会直接报错；启用 containerd image store 后多架构 `--load` 可用（Docker Engine 29 起全新安装已默认启用）。多架构构建通常使用 `--push` 推到仓库，再由节点按架构拉取；离线场景可改用归档输出并确认导入工具支持多平台内容。
