# 多架构镜像构建

在 x86 机器上构建 ARM 镜像，或同时构建 amd64、arm64 多架构镜像，通常使用 Docker Buildx。

本节目标：

- 理解为什么同一个镜像标签可以同时支持 amd64 和 arm64。
- 掌握 `docker buildx build --platform` 的基本用法。
- 学会验证镜像仓库中的 manifest 是否包含目标架构。

前置条件：

```bash
docker version
docker buildx version
```

如果 `docker buildx version` 不存在，说明当前 Docker 环境缺少 Buildx 插件，需要先升级 Docker Engine 或安装 Buildx。

## 查看构建器

```bash
docker buildx ls
```

## 创建多架构构建器

```bash
docker buildx create \
  --name mybuilder \
  --use \
  --platform linux/amd64,linux/arm64 \
  --driver docker-container
```

初始化构建器：

```bash
docker buildx inspect --bootstrap
```

## 安装跨架构模拟支持

```bash
docker run --privileged --rm tonistiigi/binfmt --install all
```

`binfmt` 让宿主机能够模拟运行非本机架构的二进制文件，是跨架构构建的基础。

## 构建并推送多架构镜像

```bash
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t registry.example.com/demo/nginx:v2 \
  --push \
  -f ./Dockerfile .
```

`--push` 会将多架构 manifest 和所有架构的镜像一起推送到仓库。

验证仓库中的多架构信息：

```bash
docker buildx imagetools inspect registry.example.com/demo/nginx:v2
```

预期能看到类似架构列表：

```text
Platform: linux/amd64
Platform: linux/arm64
```

## 只构建并加载到本地

```bash
docker buildx build \
  --platform linux/arm64/v8 \
  -t nginx:armv2 \
  -f ./Dockerfile . \
  --load
```

`--load` 通常只适合单架构镜像，多架构镜像直接加载到本地 Docker 有一定限制。

## 注意事项

- 基础镜像也必须支持对应架构（如 `nginx:1.27-alpine` 同时提供 amd64 和 arm64）。
- 如果向镜像中拷贝二进制文件，二进制文件也必须匹配目标架构。
- `--load` 通常只适合单架构镜像。
- 多架构镜像通常需要 `--push` 到镜像仓库，再由 K8s 节点按自身架构拉取。

## Kubernetes 中的使用方式

Deployment 中通常仍然使用同一个镜像标签：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: demo-api
spec:
  replicas: 2
  selector:
    matchLabels:
      app: demo-api
  template:
    metadata:
      labels:
        app: demo-api
    spec:
      containers:
        - name: demo-api
          image: registry.example.com/demo/api:v1.0.0
```

如果 `v1.0.0` 是多架构镜像，amd64 节点会拉取 amd64 产物，arm64 节点会拉取 arm64 产物。前提是镜像仓库中确实存在对应架构的 manifest。

如果某个应用只能运行在特定架构节点上，应使用 `nodeSelector`、节点亲和性或 taint/toleration 控制调度，而不是依赖拉取失败来暴露问题。

## 常见错误

| 现象 | 可能原因 | 排查方式 |
| --- | --- | --- |
| arm64 节点拉取后无法启动 | 镜像标签只包含 amd64 | `docker buildx imagetools inspect <image>` |
| 构建过程很慢 | 使用 QEMU 模拟跨架构执行 | 尽量使用原生架构构建节点或 CI 构建矩阵 |
| `exec format error` | 容器内二进制架构不匹配 | `file <binary>` 或检查构建参数 |
| `--load` 后只看到一个架构 | Docker 本地镜像存储不适合加载多架构 manifest | 使用 `--push` 推送到仓库验证 |

## 生产建议

- CI 中为每次发布生成不可变标签，例如 `v1.0.0`、Git commit SHA 或构建号。
- 多架构镜像发布后，必须验证 manifest 中包含所有目标平台。
- Go、Rust 等编译型语言要明确设置 `GOOS`、`GOARCH` 或对应构建参数。
- 不要让同一个 tag 在不同时间指向不同架构集合，否则会导致部分节点表现不一致。

## 本章回顾

- 多架构镜像的关键是 manifest list，而不是多个随意命名的 tag。
- `buildx` 可以构建并推送多架构镜像，生产场景更推荐 `--push`。
- Kubernetes 会按节点架构自动选择镜像产物，但前提是仓库中已经发布对应架构。
