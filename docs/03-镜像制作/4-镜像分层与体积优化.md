# 镜像分层与体积优化

镜像是一层一层叠加出来的。理解镜像分层，有助于优化构建速度和镜像大小。镜像越小，拉取越快，漏洞面越小，分发成本越低。

## 查看构建历史

```bash
docker history nginx:1.27-alpine
```

## 镜像层特点

- 镜像层是只读的。
- 多个镜像可以共享相同基础层。
- 容器启动后会在镜像层上添加一个可写层。
- 删除 tag 不一定删除层，只有没有引用时才会真正删除。

## 缓存与指令顺序

Docker 按指令顺序判断缓存。一旦某一层变化，后续层通常都要重新构建。

```dockerfile
COPY package*.json ./
RUN npm ci --omit=dev
COPY . .
```

先复制依赖描述文件并安装依赖，再复制源码。这样源码变化时依赖安装层仍然可以复用缓存。

## 合并在同一层操作

安装和清理要在同一个 `RUN` 中完成：

```dockerfile
RUN apt-get update \
    && apt-get install -y --no-install-recommends ca-certificates curl \
    && rm -rf /var/lib/apt/lists/*
```

如果把清理写成单独的 `RUN`，前面层的缓存文件仍然占用空间。

## 多阶段构建

编译工具和源码只留在构建阶段，最终镜像只保留运行所需文件。

Go 应用示例：

```dockerfile
FROM golang:1.23-alpine AS builder
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -ldflags="-s -w" -o /app/server .

FROM alpine:3.20
WORKDIR /app
COPY --from=builder /app/server /app/server
CMD ["/app/server"]
```

关键优化点：

- `CGO_ENABLED=0` 静态编译，不依赖 glibc。
- `-ldflags="-s -w"` 去除调试信息，减小二进制体积。
- `--from=builder` 从构建阶段复制产物。

## 选择合适的基础镜像

| 镜像 | 特点 | 适用场景 |
| --- | --- | --- |
| `alpine` | 约 7 MB，musl libc | 静态编译程序、简单依赖的应用 |
| `debian:slim` | 约 80 MB，glibc | 兼容性要求较高的应用 |
| distroless | 无包管理器、无 shell | 安全要求高的生产镜像 |

Alpine 使用 musl libc，部分依赖 glibc 的程序可能需要额外处理。

## .dockerignore

使用 `.dockerignore` 排除无关文件，减小构建上下文：

```text
.git
node_modules
*.log
dist
.tmp
Dockerfile
.dockerignore
```

## 优化检查清单

- 使用明确且较小的基础镜像。
- 多阶段构建，只保留运行时产物。
- 合并安装和清理步骤到同一个 RUN。
- 不把构建缓存、包管理器缓存留进镜像。
- 使用 `.dockerignore` 排除无关文件。
- 删除不必要的调试工具和临时文件。
- 变更频繁的指令放后面，变更少的放前面。
