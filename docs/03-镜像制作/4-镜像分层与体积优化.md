# 镜像分层与体积优化

Docker 镜像由多个只读层叠加而成。理解分层机制，才能真正控制镜像的体积和构建速度。镜像越小，拉取越快、攻击面越窄、分发成本越低。

输出中可以看到：

### 镜像层特性

- 镜像层是**只读**的。容器启动后，Docker 在镜像层之上追加一个可写层，所有写入操作发生在这里。
- 多个镜像可以**共享**相同的基础层——例如 10 个基于 `alpine:3.20` 的镜像，宿主机只需保存一份 Alpine 层。
- `docker rmi` 删除 tag 时，底层数据不会立即删除——只有当没有任何引用指向该层时，它才会被真正回收。
- `RUN`、`COPY`、`ADD` 等会产生文件系统变更的指令通常会形成镜像层；`CMD`、`EXPOSE`、`LABEL`、`ENV` 等指令主要写入镜像配置元数据。

## 缓存与指令顺序

Docker 按指令顺序逐层构建，每条指令构建前检查缓存：如果前一层未变且当前指令与上次相同，就复用缓存层；否则从该层开始全部重新构建。

指令顺序会直接影响构建效率，变更频繁的内容应尽量放到后面：

```dockerfile
# 先复制依赖描述文件，安装依赖（变化少 → 缓存可复用）
COPY package*.json ./
RUN npm ci --omit=dev

# 再复制源码（变化频繁 → 这一层及之后才重建）
COPY . .
```

源码改动时，`COPY package*.json` 层和 `RUN npm ci` 层都被缓存命中，只有 `COPY . .` 及之后需要重建。

## 合并在同一层操作

安装和清理必须在同一个 `RUN` 中完成，这是 Dockerfile 最常见的优化误区。

**错误写法**（apt 缓存残留在中间层，占用镜像空间）：

```dockerfile
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*
```

**正确写法**（安装和清理在同一层，不留下无用文件）：

```dockerfile
RUN apt-get update \
    && apt-get install -y --no-install-recommends ca-certificates curl \
    && rm -rf /var/lib/apt/lists/*
```

原理：每条 `RUN` 生成一个只读层。即使后面的层删除了文件，前面层中的数据仍然占据空间；删除操作只是覆盖文件视图，底层只读层本身不可变。

## 多阶段构建

多阶段构建将编译工具和中间产物留在构建阶段，最终只把运行时必需的文件复制到最终镜像。这是**减少镜像体积最有效的手段**。

Go 应用示例：

先准备最小 Go 项目文件：

```bash
cat > go.mod <<'EOF'
module example.com/server

go 1.23
EOF

touch go.sum

cat > main.go <<'EOF'
package main

import (
	"fmt"
	"net/http"
)

func main() {
	http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
		fmt.Fprintln(w, "hello from go")
	})

	http.ListenAndServe(":8080", nil)
}
EOF
```

```dockerfile
# === 构建阶段 ===
FROM golang:1.23-alpine AS builder
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -ldflags="-s -w" -o /app/server .

# === 运行阶段 ===
FROM alpine:3.20
RUN apk add --no-cache ca-certificates
WORKDIR /app
COPY --from=builder /app/server /app/server
CMD ["/app/server"]
```

构建完成后，可对比阶段镜像和最终镜像：

```text
docker images | grep -E 'golang|server'
golang:1.23-alpine   72fd06a5dd05        246MB             0B
server:v1.0.0        ee6a564e5e83       13.4MB             0B
```

输出中的体积会随基础镜像版本、目标架构和构建参数变化。多阶段构建的关键点是 Go 编译器、源码和中间缓存都留在了 `builder` 阶段，最终镜像只保留运行时所需文件。

关键优化点：

- `CGO_ENABLED=0`：关闭 CGO，生成纯静态二进制，不依赖 glibc。
- `-ldflags="-s -w"`：去除符号表和调试信息，减少二进制体积。
- `--no-cache`：Alpine 用 `apk add --no-cache` 避免留存包索引；Debian 用 `rm -rf /var/lib/apt/lists/*`。
- `ca-certificates`：Alpine 最小镜像不含 CA 根证书，静态二进制的 HTTPS 请求会失败，需要显式安装。

## 选择合适的基础镜像

| 镜像          | 体积   | C 库      | 适用场景                               |
| ------------- | ------ | --------- | -------------------------------------- |
| `alpine`      | ~7 MB  | musl libc | 静态编译程序、简单依赖应用             |
| `debian:slim` | ~80 MB | glibc     | 兼容性要求高、依赖动态链接库           |
| `distroless`  | ~3 MB  | glibc     | 高安全要求、不需要 shell 的生产镜像，Google 维护 |
| `scratch`     | 0 B    | 无        | 完全静态编译的二进制，不需要任何系统库或文件 |

选择原则是在体积和兼容性之间取得平衡。若应用强依赖 glibc 或存在复杂原生依赖，Debian Slim 往往比 Alpine 更稳妥。

## .dockerignore

构建时，Docker 会把 Dockerfile 所在目录的全部文件作为构建上下文发给 daemon。`node_modules`、`.git`、日志文件等全会被打包传送，严重影响构建速度。

```text
.git
node_modules
*.log
dist
.tmp
Dockerfile
.dockerignore
.vscode
.idea
coverage
```

`.dockerignore` 的语法和 `.gitignore` 一致，创建后放在 Dockerfile 同级目录即可。

## 优化检查清单

每次提交 Dockerfile 前，逐项对照：

- [ ] 基础镜像版本明确（不是 `latest`），体积合理。
- [ ] 使用了多阶段构建，仅保留运行时产物。
- [ ] 安装和清理在同一个 `RUN` 中完成。
- [ ] 构建缓存、包管理器索引没有残留。
- [ ] `.dockerignore` 排除了 `.git`、`node_modules`、日志和缓存。
- [ ] 变更频繁的指令（源码 COPY）放在最后。
- [ ] 没有在镜像中安装调试工具、编译器和文档。
- [ ] 二进制使用 `-ldflags="-s -w"` 或等同手段去除调试信息。
- [ ] 生产镜像按运行方式配置健康检查，并优先使用非 root 用户。
