# 基础镜像与镜像元数据

基础镜像决定容器内的文件系统、包管理器、默认 shell、C 库和运行时工具。镜像元数据则记录端口、标签、默认配置等信息，便于本地命令、镜像仓库、扫描工具和部署平台识别镜像。

本篇只记录基础镜像选型和不直接改变文件内容的常见元数据指令。文件复制、运行用户和启动命令分别在后续文章中展开。

## FROM：选择基础镜像

`FROM` 决定当前构建阶段从哪个镜像开始。除全局 `ARG` 外，`FROM` 通常是 Dockerfile 的第一条指令。

```dockerfile [Dockerfile]
FROM alpine:3.23.5
```

基础镜像选型原则：

- 使用明确 tag，避免依赖 `latest` 作为生产版本。
- 优先选择官方镜像或维护活跃、来源可信的镜像。
- 在体积、兼容性和调试便利性之间权衡，不只按镜像大小选择。
- 关键发布记录保存镜像 digest，避免同名 tag 后续变化影响复现。

`ARG` 可用于控制基础镜像版本，但必须声明在 `FROM` 前：

```dockerfile [Dockerfile]
ARG ALPINE_VERSION=3.23.5
FROM alpine:${ALPINE_VERSION}
```

```bash
docker build --build-arg ALPINE_VERSION=3.23.5 -t alpine:base .
```

## 基础镜像对比

| 镜像                         | 体积     | C 库       | 适用场景                       |
|----------------------------|--------|-----------|----------------------------|
| `alpine`                   | ~8 MB  | musl libc | 静态编译程序、简单依赖应用              |
| `debian:bookworm-slim`     | ~80 MB | glibc     | 兼容性要求高、依赖动态链接库             |
| `gcr.io/distroless/static` | ~2 MB  | 无         | 完全静态编译、高安全要求、不需要 shell     |
| `gcr.io/distroless/base`   | ~20 MB | glibc     | 动态链接 glibc、高安全要求、不需要 shell |
| `scratch`                  | 0 B    | 无         | 完全静态编译的二进制，不需要任何系统库或文件     |

选择原则是在体积和兼容性之间取得平衡。若应用强依赖 glibc 或存在复杂原生依赖，Debian Slim 往往比 Alpine 更稳妥。distroless 系列由 Google 维护，镜像内没有 shell 和包管理器：`static` 变体只含 CA 证书、时区等最基础的运行时文件，适合纯静态二进制；`base` 变体在此基础上提供 glibc。

`scratch` 是 Docker 预留的空基础镜像，不是一个可进入的 Linux 发行版镜像。`FROM scratch` 表示最终镜像从空文件系统开始，Dockerfile 后续复制进去什么，镜像里才有什么；它没有 shell、包管理器、libc、CA 证书、时区数据和 `/etc/passwd`。因此 `scratch` 通常只适合完全静态编译的单文件程序，尤其是 Go、Rust 等可以产出静态二进制的服务。如果程序需要 HTTPS 证书校验、时区、本地用户信息或其他运行时文件，需要在多阶段构建中显式复制这些文件；如果程序依赖动态链接库，使用 `scratch` 会因缺少动态链接器或共享库而启动失败。

静态二进制切换到 `scratch` 的最终阶段通常只保留可执行文件：

```dockerfile [Dockerfile]
FROM scratch
WORKDIR /app
COPY --from=builder /app/server /app/server
CMD ["/app/server"]
```

## EXPOSE：声明监听端口

`EXPOSE` 表示镜像作者认为容器运行时会监听哪些端口。它会写入镜像元数据，方便使用者、工具或 `docker inspect` 查看，但不会把端口发布到宿主机。

```dockerfile [Dockerfile]
FROM nginx:1.31.2-alpine
COPY ./html /usr/share/nginx/html
EXPOSE 80
```

端口发布发生在 `docker run` 阶段：

| 写法        | 作用                                      |
|-----------|-----------------------------------------|
| `EXPOSE` | Dockerfile 中的文档约定，只声明容器计划监听的端口，不发布端口 |
| `-p`      | 把指定容器端口发布到宿主机指定端口，例如 `-p 8080:80`    |
| `-P`      | 把镜像中所有 `EXPOSE` 声明的端口发布到宿主机随机高位端口    |

以下命令沿用 [Dockerfile 与构建上下文](./1-Dockerfile与构建上下文.md) 中构建的 `nginx:demo` 镜像。只声明 `EXPOSE 80` 时，镜像元数据中能看到端口声明：

```bash
docker inspect nginx:demo --format '{{json .Config.ExposedPorts}}'
```

::: details 输出类似如下

```text
{"80/tcp":{}}
```

:::

但仅运行容器不会产生宿主机端口映射：

```bash
docker run -d --name expose-only nginx:demo
docker inspect expose-only --format '{{json .NetworkSettings.Ports}}'
```

::: details 输出类似如下

```text
{"80/tcp":null}
```

:::

使用 `-p` 可以明确指定宿主机端口：

```bash
docker run -d --name publish-one -p 8081:80 nginx:demo
docker port publish-one 80
curl http://127.0.0.1:8081
```

::: details 端口查看输出类似如下

```text
0.0.0.0:8081
[::]:8081
```

:::

使用 `-P` 时，Docker 会把所有已声明的 `EXPOSE` 端口发布到宿主机随机高位端口：

```bash
docker run -d --name publish-all -P nginx:demo
docker port publish-all 80
```

::: details 端口查看输出类似如下

```text
0.0.0.0:32768
[::]:32768
```

:::

清理示例容器：

```bash
docker rm -f expose-only publish-one publish-all
```

## LABEL：镜像元数据

`LABEL` 用来给镜像添加结构化元数据，例如维护者、版本、源码地址、构建时间等，便于仓库、扫描工具和平台识别。

```dockerfile [Dockerfile]
FROM alpine:3.23.5
LABEL maintainer="yleoer" version="demo"
LABEL multiple="true"
```

```bash
docker build -t alpine:label .
docker inspect alpine:label --format '{{json .Config.Labels}}'
```

推荐使用 OCI 标准标签，与 Harbor、GitHub Container Registry 等平台兼容：

```dockerfile
LABEL maintainer="platform@example.com"
LABEL org.opencontainers.image.title="demo-app"
LABEL org.opencontainers.image.version="1.0.0"
LABEL org.opencontainers.image.source="https://example.com/repo/demo-app"
LABEL org.opencontainers.image.description="Demo application image"
```
