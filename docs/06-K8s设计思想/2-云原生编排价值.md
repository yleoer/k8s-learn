# 云原生编排价值

容器使应用交付物更加标准化，但容器本身并不等同于生产级平台。多节点部署、版本发布、故障恢复、流量接入、配置隔离和资源治理等需求，是 Kubernetes 需要解决的核心问题。

## 单机管理局限

Docker 和 containerd 适合单机容器管理，但在服务数量和团队规模持续增长后，会逐步暴露以下短板：

| 痛点 | 具体表现 |
| --- | --- |
| 生命周期分散 | 容器异常、节点故障、副本补偿需要额外脚本处理 |
| 服务发现脆弱 | 容器 IP 随重建变化，调用方难以稳定连接 |
| 扩缩与发布困难 | 多实例部署、滚动更新、回滚需要人工编排 |
| 配置侵入镜像 | 多环境配置写进镜像，交付物无法复用 |
| 端口与入口混乱 | 多服务多节点端口冲突，外部访问入口难以统一 |
| 流量治理缺失 | 灰度、限流、熔断需要额外组件 |
| 资源无边界 | 多团队共用机器时，资源占用缺少统一约束 |

在少量服务场景中，可以通过脚本和人工约定暂时处理这些问题。随着服务数量、团队数量和环境数量增加，维护成本会快速上升。

## 集群编排能力

Kubernetes 将多台机器抽象成统一资源池。用户不再关注服务运行在哪台具体机器上，而是通过资源对象描述期望状态，由集群完成分配、调度和保障。

Kubernetes 的核心能力包括：

- 开源开放，生态成熟，可运行在自建机房、公有云和混合云环境。
- 声明式资源模型，配置可版本化、可审计、可复用。
- 弹性伸缩，按副本数或指标自动调整实例数量。
- 服务发现和负载均衡，Pod 动态变化时调用方无需感知。
- 健康检查与自愈，及时替换异常实例。
- 滚动更新和版本回滚，降低发布风险。
- 多环境、多租户隔离，划分项目、团队和资源边界。

## 对开发人员的变化

开发人员的交付方式从交付软件包和机器配置，转变为交付镜像和 Kubernetes 资源清单：

- 交付物从压缩包和机器配置转向镜像与资源清单。
- 运行入口从固定机器转向 Deployment、Service 和 Ingress。
- 配置从代码和镜像中拆分出来，交由 ConfigMap、Secret 或外部配置中心管理。
- 副本数、健康检查和发布策略通过 YAML 声明，并由 CI/CD 流水线执行。

开发人员需要掌握容器化、健康检查、资源请求、环境变量、日志输出和优雅退出等实践。否则，即使应用能够部署到 Kubernetes 中，也可能在发布或扩缩容过程中出现问题。

## 对运维人员的变化

运维人员的关注重点从单台机器上的进程状态，转向集群资源池健康、调度结果和容量规划：

- 机器从按 IP 分配应用转向统一纳管为资源池。
- 应用副本自动分散到不同节点，节点故障后自动迁移。
- 资源请求、限制和配额约束团队使用边界。
- 发布、回滚、扩容、下线流程由平台能力承接。
- 监控、日志、告警围绕 K8s 资源对象建立统一视图。

运维也需要应对新的复杂度：集群升级、网络插件、存储插件、证书管理、权限体系、控制面高可用和资源碎片等。

## 平台复杂度

Kubernetes 提供了强大的抽象能力，但并不会消除复杂度。它会将分散在脚本、人工流程和机器配置中的复杂度集中到平台层：

| 挑战 | 说明 |
| --- | --- |
| 概念数量多 | Pod、Deployment、Service、Ingress、ConfigMap 等需要逐一建立关联 |
| 排障链路长 | 问题可能出在镜像、调度、网络、存储、配置或应用自身 |
| 权限边界复杂 | 多用户、多命名空间和集群级资源需要细致授权 |
| 平台依赖增强 | 应用发布和恢复依赖控制面与节点组件稳定运行 |
| 资源治理要求高 | requests、limits、Quota 配置不当直接影响利用率和稳定性 |

因此，学习 Kubernetes 不能只记忆命令，还需要理解其设计思想。只有理解“声明期望状态、控制器协调、节点执行”这条主线，后续排障和架构设计才不会碎片化。

## 小结

容器解决了应用如何打包和运行的问题；Kubernetes 进一步解决了大量容器如何在集群中长期、稳定、安全运行的问题。
