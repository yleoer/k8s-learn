# 资源默认值验证

默认值应以创建后的对象规范为准，而不是仅查看 LimitRange。下面的 Pod 故意不声明 resources，用于验证命名空间默认注入；只应在已创建 LimitRange 的测试命名空间执行。

```yaml [defaulted-pod.yaml]
apiVersion: v1
kind: Pod
metadata:
  name: defaulted-pod
  namespace: team-a
spec:
  containers:
    - name: pause
      image: alpine:3.23
      command: ["sh", "-c", "sleep 3600"]
```

```bash
kubectl create -f defaulted-pod.yaml
kubectl get po defaulted-pod -n team-a -o jsonpath='{.spec.containers[0].resources}'
kubectl delete po defaulted-pod -n team-a
```

返回的 JSON 中应包含来自 `defaultRequest` 与 `default` 的请求和限制。若创建被 ResourceQuota 拒绝，先确认 LimitRange 是否存在于同一命名空间，以及其约束是否允许默认后的值。

## 设计边界

默认 resources 能避免无声明 Pod 绕开计算配额，但不能替代对服务负载的测量。平台默认值过大可能压缩集群可调度容量，过小则会让关键服务在压力下竞争不足的 CPU 或内存。
