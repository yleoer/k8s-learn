# 认证主体与授权边界

Kubernetes 先认证请求者，再由一个或多个授权器判断请求。RBAC 中主体可以是 `User`、`Group` 或 `ServiceAccount`；普通用户不是 Kubernetes API 中的持久对象，通常由外部身份提供方、客户端证书或受控 kubeconfig 表示。

Role 和 RoleBinding 的作用域是命名空间；ClusterRole 可以包含集群范围资源规则或可在多个命名空间复用的规则。RoleBinding 可以引用 ClusterRole，但它只在自身命名空间授予其中的权限；ClusterRoleBinding 才会在全局范围生效。

## 最小权限原则

授权规则应只包含实际需要的 `apiGroups`、`resources`、`verbs` 与可选 `resourceNames`。`*`、`secrets` 的读取、`pods/exec`、workload `create`/`patch` 和 RBAC 资源的写权限都可能扩大影响面，应分别审查。

> [!IMPORTANT]
> 能创建 Pod 或控制工作负载模板的主体，往往可以间接获得节点、Secret 或 ServiceAccount 的更多能力。评估权限时必须考虑可创建资源的完整 Pod spec，而不是只检查单个 API verb。

## 最小授权验证

复用[Role 与 RoleBinding](./3-Role与RoleBinding.md)中的完整 `team-a-developer-rbac.yaml` 创建受限 ServiceAccount 与 RoleBinding。随后以该主体检查普通资源和子资源，确认规则没有因资源名称相近而被意外放宽：

```bash
kubectl auth can-i get po -n team-a \
  --as=system:serviceaccount:team-a:developer
kubectl auth can-i get po/log -n team-a \
  --as=system:serviceaccount:team-a:developer
kubectl auth can-i create po/exec -n team-a \
  --as=system:serviceaccount:team-a:developer
```

预期前两项为 `yes`，最后一项为 `no`。这只能验证 RBAC 授权结果；准入、网络策略和应用自身认证仍需要在实际访问路径中分别验证。
