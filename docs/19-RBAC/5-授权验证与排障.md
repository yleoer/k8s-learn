# 授权验证与排障

`kubectl auth can-i` 用于验证某个主体对一个资源动作是否被授权。它是配置检查工具，不能替代端到端业务验证，也不会展示所有可能的准入、网络或应用层拒绝原因。

## 验证规则

```bash
kubectl auth can-i get po -n team-a \
  --as=system:serviceaccount:team-a:developer
kubectl auth can-i get po/log -n team-a \
  --as=system:serviceaccount:team-a:developer
kubectl auth can-i create po/exec -n team-a \
  --as=system:serviceaccount:team-a:developer
kubectl auth can-i --list -n team-a \
  --as=system:serviceaccount:team-a:developer
```

`pods/log` 与 `pods/exec` 是子资源，需要在 RBAC 中使用完整资源名。查看 `pods` 并不自动包含读取日志或执行命令的权限。

## 排查顺序

先确认认证身份，再确认请求的 API group、资源、子资源、verb 与命名空间，最后检查对应 Binding 是否存在并引用预期 Role。对集群外用户，还要核对 kubeconfig 的当前 context、证书或外部身份提供方声明。

> [!CAUTION]
> 不要为了排障临时授予 `cluster-admin`。使用一个专门的、短期且可审计的管理员流程，并在验证完成后回收权限。
