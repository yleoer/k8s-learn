# 扩容、快照与排障

持久卷投入使用后，常见维护动作包括扩容、创建快照和处理 PVC 或挂载异常。这些能力需要 Kubernetes API、CSI 驱动与存储后端同时支持，不能只根据清单字段判断实际能力。

## PVC 扩容

扩容需要满足以下条件：

- PVC 对应的 StorageClass 设置 `allowVolumeExpansion: true`。
- CSI 驱动和存储后端支持扩容。
- 新请求容量大于当前容量；Kubernetes 不支持通过 PVC 缩容。

修改已有 PVC 的请求容量：

```bash
kubectl patch pvc nfs-csi-data \
  --type merge \
  -p '{"spec":{"resources":{"requests":{"storage":"2Gi"}}}}'
```

控制面会扩展现有 PV 对应的卷，不会创建新 PV。文件系统扩容可能在 Pod 启动时完成，也可能由支持在线扩容的驱动在 Pod 运行时完成。

> [!WARNING]
> 不要直接修改 PV 的 `spec.capacity` 试图触发后端扩容。这样可能只改变 API 中的容量记录，使控制面误以为扩容已经完成。

对于上一页的 NFS CSI 子目录，PVC 容量可以更新，但普通 NFS 目录不会自动获得对应配额。容量隔离仍需在 NFS 服务端使用文件系统配额或支持配额的存储产品实现。

观察扩容状态：

```bash
kubectl get pvc nfs-csi-data -w
kubectl describe pvc nfs-csi-data
kubectl get ev --sort-by=.metadata.creationTimestamp
```

请求超过后端能力时，控制器会持续重试。应根据 PVC 事件、`.status.allocatedResourceStatuses` 和 CSI 组件日志处理，不要在未确认数据保护的情况下删除 PVC。

Kubernetes v1.34 起，扩容失败恢复已稳定支持把 PVC 请求改为一个小于上次请求、但仍大于 `.status.capacity` 的值，再次尝试扩容。这不属于缩容：新请求不能低于卷当前已经达到的容量。只有旧集群或无法使用该机制时，才需要由管理员把 PV 临时改为 `Retain`、重建 PVC 并预绑定原 PV；该流程涉及删除对象，执行前必须单独保护数据。

## VolumeSnapshot 边界

`VolumeSnapshot`、`VolumeSnapshotContent` 和 `VolumeSnapshotClass` 是 CRD，不属于 Kubernetes 核心 API。集群还需要安装快照 CRD、快照控制器和 CSI snapshotter，具体 CSI 驱动也必须实现快照能力。

三类对象的关系与 PV/PVC 类似：

| 对象                      | 作用域   | 作用                                     |
|-------------------------|-------|----------------------------------------|
| `VolumeSnapshot`        | 命名空间级 | 用户对某个 PVC 快照的请求                        |
| `VolumeSnapshotContent` | 集群级   | 表示后端中的实际快照，与请求一对一绑定                    |
| `VolumeSnapshotClass`   | 集群级   | 指定 CSI 驱动、参数和 `Delete` 或 `Retain` 删除策略 |

下面只展示动态快照所需的字段关系，不是可直接提交的通用清单；`<snapshot-class>` 必须替换为集群中已安装且与源 PVC 驱动一致的 VolumeSnapshotClass：

```yaml
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata:
  name: app-data-snapshot
spec:
  volumeSnapshotClassName: <snapshot-class>
  source:
    persistentVolumeClaimName: app-data
```

恢复时创建新 PVC，并通过 `dataSource` 指向已经 `readyToUse` 的快照。下面同样只是字段关系片段，不能脱离实际 StorageClass、容量和访问模式直接提交：

```yaml
spec:
  storageClassName: <storage-class>
  dataSource:
    name: app-data-snapshot
    kind: VolumeSnapshot
    apiGroup: snapshot.storage.k8s.io
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
```

快照通常是存储级时间点副本，不自动保证应用一致性。数据库等应用应在快照前执行冻结、刷盘或应用原生备份，并验证恢复流程。`VolumeSnapshotClass.deletionPolicy: Delete` 会随请求删除后端快照；`Retain` 会保留内容，需要管理员后续清理。

本章使用的 NFS CSI Driver v4.13.4 通过 `tar` 复制文件来实现快照，不是存储阵列或写时复制文件系统提供的瞬时快照。创建耗时与数据量相关，副本仍位于同一 NFS 故障域；它可以验证 VolumeSnapshot API 流程，但不能替代独立备份。除本页列出的 CRD 和快照控制器外，控制器 Pod 中还必须运行 CSI snapshotter Sidecar。

## PVC Pending

PVC 一直 `Pending` 时，先查看事件：

```bash
kubectl describe pvc <pvc-name>
kubectl get sc
kubectl get pv
kubectl get ev --sort-by=.metadata.creationTimestamp
```

常见原因包括：

- `storageClassName` 拼写错误、StorageClass 不存在或供给器未安装。
- 静态 PV 容量、访问模式、StorageClass、`volumeMode` 或标签选择器不匹配。
- 动态供给器无权限、后端不可达、参数错误或容量不足。
- `WaitForFirstConsumer` 正在等待引用 PVC 的 Pod，这是预期状态。
- 使用 `WaitForFirstConsumer` 的 Pod 设置了 `nodeName`，绕过调度器。
- PVC 指定非空 `selector`，因此不能动态供给。

## Pod Pending 或挂载失败

PVC 已 `Bound` 但 Pod 无法运行时：

```bash
kubectl describe po <pod-name>
kubectl get po <pod-name> -o wide
kubectl get pvc <pvc-name> -o wide
kubectl get pv <pv-name> -o yaml
```

重点检查 Pod 与 PVC 是否在同一命名空间、节点是否满足 PV `nodeAffinity`、访问模式是否冲突，以及事件中的 `FailedAttachVolume`、`FailedMount`、超时和权限错误。

NFS 场景还要检查：

```bash
kubectl -n kube-system get po -l app=csi-nfs-node -o wide
kubectl -n kube-system logs -l app=csi-nfs-controller -c nfs --tail=100
showmount -e <nfs-server-ip>
```

确认节点安装了 `nfs-common`，NFS 导出允许节点网段访问，服务端目录存在，`nfsvers` 与服务器兼容，防火墙允许所需端口，Pod 的 UID/GID 与 NFS 目录权限匹配。`root_squash` 导致容器内 root 映射为匿名用户是正常安全行为，应调整应用安全上下文和目录权限，不应直接关闭保护。

## 删除卡住

PVC 正被 Pod 使用，或 PV 仍与 PVC 绑定时，保护 finalizer 会让资源保持 `Terminating`。先确认引用关系和存储控制器状态：

```bash
kubectl get po -A -o yaml
kubectl describe pvc <pvc-name>
kubectl describe pv <pv-name>
```

不要直接移除 finalizer。先停止使用该卷的 Pod，确认 CSI 控制器和后端恢复正常，再让控制器完成卸载、解绑和回收；强行删除 finalizer 可能遗留挂载、泄漏后端卷或造成数据丢失。
