# 命名空间治理

环境、团队和租户的隔离单位通常是命名空间。是否为每个环境分别使用集群或命名空间，取决于安全边界、升级节奏、网络隔离、存储隔离和运维责任，不能仅由 ResourceQuota 决定。

## 组成

一个可操作的租户命名空间至少需要：ResourceQuota 控制总量、LimitRange 限制单对象、RBAC 限定主体、NetworkPolicy 限定通信，以及监控和审计提供可见性。每项对象有不同边界，缺失任一项都会留下治理空洞。

## 创建顺序

先创建命名空间、配额和 LimitRange，再向租户授予创建工作负载的 RBAC 权限。这样首个 Pod 从创建开始就受到默认资源和累计配额约束，避免先部署后补治理时出现不可预测的拒绝。

```bash
kubectl create ns team-a
kubectl create -f team-a-quota.yaml
kubectl create -f team-a-limits.yaml
kubectl create -f team-a-developer-rbac.yaml
```

后三个清单分别在[ResourceQuota 配置](./1-ResourceQuota配置.md#计算和对象配额)中的 `team-a-quota.yaml`、[限制范围](../17-限制范围/1-LimitRange资源.md#容器默认值与范围)中的 `team-a-limits.yaml` 和 [RBAC](../19-RBAC/3-Role与RoleBinding.md)中的 `team-a-developer-rbac.yaml` 中定义。这里复用文件名，不重复粘贴清单。

> [!CAUTION]
> 把开发、测试和生产放在同一命名空间会让配额、权限和生命周期难以拆分。需要共享资源时优先通过受控服务接口、镜像仓库和 CI/CD 流程协作，而不是放宽命名空间边界。
