# 配额观察与排障

配额拒绝会在 API 请求阶段发生，常见错误包含 `exceeded quota` 与具体资源键。排查时应同时查看定义的 `hard`、实时的 `used` 和待创建 Pod 的 resources。

## 检查命令

```bash
kubectl get quota -n team-a
kubectl describe quota team-a-quota -n team-a
kubectl get po -n team-a
kubectl get pvc -n team-a
kubectl get ev -n team-a --sort-by=.metadata.creationTimestamp
```

如果资源被删除后 `used` 没有立即下降，应先确认对象的删除是否完成、Pod 是否终止，以及控制器状态是否正常。不要编辑 `status.used` 或用扩大配额掩盖资源泄漏。

## 设计检查

| 问题 | 检查方向 |
| --- | --- |
| 新 Pod 被拒绝 | 是否缺少 requests 或 limits；LimitRange 是否先于配额创建 |
| PVC 被拒绝 | `requests.storage`、PVC 数量与特定 StorageClass 配额 |
| 配额没有保护关键服务 | 是否仅限制总量而未为关键命名空间规划容量和优先级 |
| 租户能删除配额 | RBAC 是否允许 `resourcequotas` 的 `delete` 或 `update` |
