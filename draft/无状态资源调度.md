# 无状态资源调度

## Replication Controller 和 ReplicaSet

//了解即可

Replication Controller（复制控制器，RC）和 ReplicaSet（复制集，RS）三两种简单部署Pod的方式。

Replication Controller 可确保 Pod 副本数达到期望值，也就是RC定义的数量。换句话说，可确保一个Pod或一组同类Pod总是可用。

ReplicaSet 是支持基于集合的标签选择器的下一代Replication Controller，它主要用作 Deployment 协调创建/删除和更新Pod，和Replication Controller唯一的区别是，ReplicaSet 支持标签选择器。

在实际应用中，虽然ReplicaSet和Replication Controller可以单独使用，但是一般建议使用Deployment来自动管理ReplicaSet，除非自定义的Pod不需要更新或有其他编排等。

```yaml
apiVersion: v1
kind: ReplicationController
metadata:
  name: nginx
spec:
  replicas: 3
  selector:
    app: nginx
  template:
    metadata:
      name: nginx
      labels:
        app: nginx
    spec:
      contaienrs:
      - name: nginx
        image: nginx
```

```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: frontend
  labels:
    app: guestbook
    tier: frontend
spec:
  replicas: 3
  matchLabels:
    tier: frontend
  matchExpressions:
    - {key: tier, operator: In, values: [frontend]}
  template:
    metadata:
      labels:
        app: guestbook
        tier: frontend
    spec:
      containers:
      - name: php-redis
        image: gcr.io/google_sampes/gb-frontend:v3
```

## Deployment

Deployment 是 Kubernetes 的无状态资源调度器，可以使用声明式的方式描述应用的期望状态，之后由Deployment控制自动将当前状态改变为期望状态。

Deployment 通过 RS 管理 Pod，同时提供了如下的能力：

- 声明式创建和更新
- 自动化部署和回滚
- 一键式扩缩容
- 自动灾难修复
- 更高级的更新方式

Deployment 一般用于部署公司的无状态服务，这个也是最常用的控制器，因为企业内部现在都是以微服务为主，而微服务实现无状态化也是最佳实践，可以利用Deployment的高级功能做到无缝迁移/自动扩缩容/自动灾难恢复/一键回滚等功能。

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deploment
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      tier: frontend
  template:
    metadata:
      labels:
        app: guestbook
        tier: frontend
    spec:
      containers:
      - name: php-redis
        image: gcr.io/google_samples/gb-frontend:v3
```

### 管理 Pod 的流程

deployment 具有 namespace 隔离性

- 创建一个 Deployment，副本数为3
- 在 namespace 创建 deployement nginx
- 创建 RS nginx-xxx
- 在 node 创建 Pod ningx-xxx-a

### 创建

生成一个 Deployment 的Yaml

```bash
kubectl create -f nginx-deploy.yaml
kubectl get deploy
kubectl get rs
kubectl get po
kubectl rollout status deploy nginx
```

### 更新和回滚

当且仅当 deployment 的 pod 模板更改时，才可以触发更新。

```bash
kubectl set image deploy nginx nginx=nginx:1.15.12 --record
kubectl edit deploy nginx
```

会创建新的 rs

```bash
kubectl get rs
```

可以通过 `kubectl describe `查看过程

回滚：

```bash
# 查看历史
kubectl rollout history deploy nginx
# 回滚到上一个版本
kubectl rollout undo deploy nginx
# 回滚到指定版本
kubectl rollout undo deploy nginx --to-revision=2
```

## 暂停和恢复

更改某一处的配置，更改后立即触发更新，大多数情况下可能需要针对一个资源文件更改多处地方，而并不需要多次触发更新，此时可以使用 deployment 暂停功能，临时禁用更新操作，对 deployment 进行多次修改后进行更新。

```bash
# 暂停更新
kubectl rollout pause deploy nginx
# 恢复更新
kubectl rollout resume deploy nginx
```

### 扩容和缩容

```bash
# 扩容到 5 个
kubectl scale deploy nginx --replicas=5
```

## 更新策略

- Recreate：重建，先删掉所有旧的Pod再创建新的Pod
- RollingUpdate：滚动更新，先创建新的 Pod，再删除旧的 Pod，可以指定 maxUnavailable 和 maxSurge 定制化更新策略
  - maxUnavailable：指定在滚动更新时最大不可用的 Pod 数量，默认是 25%，可以设置为数字或百分比
  - maxSurge：可以超过期望值的最大Pod数，默认为25%
