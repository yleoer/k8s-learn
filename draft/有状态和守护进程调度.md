# 有状态和守护进程调度

## StatefulSet 

### 介绍

StatefulSet 是 Kubernetes 的有状态资源调度器，可以使用声明式的方式描述应用的期望状态，之后由StatefulSet控制器自动且有序的将当前状态改变为期望状态，常用于部署有状态的且需要有序启动的应用程序，比如 MySQL，Zookeeper集群等。

StatefulSet 基于 Headless Service 给每个实例创建一个唯一且固定的标识符，之后各实例之间可以通过该标识符进行通信。

StatefulSet 常用于有如下需求的场景：

- 需要稳定的独一无二的网络标识符
- 需要稳定的持久化数据
- 需要有序的/优雅的部署和扩展
- 需要有序的自动滚动更新

### Headless Service 介绍

和 Deployment 类似，一个 StatefulSet 也同样管理着基于相同容器规范的 Pod，不同的是，StatefulSet 为每个Pod维护了一个粘性标识。

而StatefulSet创建的Pod一般使用Headless Service（无头服务）进行Pod之前的通信，和普通的Service的区别在于Headless Service没有ClusterIP，它使用的是Endpoint进行互相通信，Headless一般的格式为：

```
statefulSetName-{0..N-1}.serviceName.namespace.svc.cluster.local
```

### StatefulSet 资源定义

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: web
spec:
  serviceName: "nginx"
  replicas: 2
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:stable
---
apiVersion: v1
kind: Service
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  ports:
  - port: 80
    name: web
  clusterIP: None
  selector:
    app: nginx
```

### 扩容和缩容

通过 kubectl scale/ kubectl edit 和 kubectl patch 来扩容/缩容一个 StatefulSet

### 更新策略

On Delete 策略：1.7以前的默认策略，当修改 statefulSet.spec.template 字段时，StatefulSet 控制器不会自动更新 Pod，必须手动删除 Pod 才能使控制器创建新的 Pod

RollingUpdate 策略：1.7后的默认策略，会自动更新所有的 Pod，采用与序号索引相反的顺序进行滚动更新

查看更新策略：

```bash
kubectl get sts web -oyaml | grep -A 1 "updateStrategy"
```

分段更新：partition: 3, 比如0-2不更新，3-4更新

### 回滚

和 deployment 基本一致

### 删除

使用级联删除时，StatefulSet 和它的 Pod 都会被删除。

### 并发 Pod 管理

通过 spec.podManagementPolicy 字段配置 Pod 的管理策略：

- OrderdReady：有序管理，默认方式，Pod 创建和删除序号的正序和倒序进行操作
- Parallel：并发管理，Pod 创建和删除同时并行启动和删除

### 内部通信

StatefulSet 创建的 Pod 一般使用 Headless Service 进行 Pod 之间的通信，和普通的 Service 的区别在于 Headless Service 没有 ClusterIP，它使用的是 Endpoint 进行互相通信。

使用任意一个 Pod 即可通过唯一的网络标识符访问 StatefulSet 的节点

```bash
kubectl exec -it web-2 -- curl web-1.nginx
```

不同 Namespace 的

```bash
ping web-1.nginx.default
```

## DaemonSet

守护进程集，与 Deployment 和 StatefulSet 区别是，前者会根据算法选择最优的节点去运行 Pod，而 DaemonSet 会在符合匹配条件的节点上都部署一个 Pod，如果没有其他限制条件，会在所有的节点上都运行一个 Pod。

同时当有新节点加入集群时，也会新增一个Pod，当节点从集群中移除时，这些Pod会被回收，删除DaemonSet将会删除它创建的所有Pod。

通常会用于以下需求：

- 运行集群存储的进程，例如在每个节点上运行 Glusterd/Ceph等
- 在每个节点运行日志收集进程，例如Fluentd/Filebeat
- 在每个节点运行监控进程，比如Prometheus Node Exporter
- 其他任何需要在每个节点运行一个服务的需求
