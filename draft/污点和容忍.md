# 污点和容忍

污点和容忍是kubernetes提供的一种强大的调度控制机制，可以实现资源的精细化管理和调度优化。

污点用于标记节点，容忍用于控制Pod的调度行为。常用于节点维护/资源隔离/故障恢复/故障隔离等场景。

## 什么是污点

污点作用于节点，主要用来标记节点的属性或状态，实现在默认情况下可以让Pod无法调度到这些标记了污点的节点上。

## 污点的影响效果（Effect）

- NoSchedule: 禁止调度Pod到该节点上，但是已经运行在该节点的服务不受影响。适用于资源隔离和节点维护的场景。
- NoExecute：禁止调度Pod到该节点上，同时已经运行在该节点上的Pod也会被驱逐（终止并重新调度）。适用于节点故障/紧急维护和故障快速恢复的场景。
- PreferNoSchedule：类似于NoSchedule，但不是一个硬性限制，调度器会尽量避免将新的Pod调度到这个节点上，但如果其他节点都不满足条件，Pod仍然会被调度到这个节点上。适用于软性资源隔离的场景。

## 污点配置

```bash
kubectl taint node NODE_NAME TAINT_KEY=TAINT_VALUE:EFFECT

kubectl taint node gpu-node01 gpu=true:NoSchedule
```

### 常用命令

```bash
# 查看节点的污点
kubectl get node work01 -o go-template --template {{.spec.taints}}
kubectl ldescribe node work01 | grep Taints -A 10

# 删除污点
# 基于 key
kubectl taint node work01 ssd-
# 基于 key+effect
kubectl taint node work01 ssd:PreferNoSchedule-
# 基于完整格式
kubectl taint node work01 ssd=true:PreferNoSchedule-

# 修改污点
kubectl taint node work01 ssd=true:PreferNoSchedule --overwrite
```

### 常见内置污点

- node.kubernetes.io/not-ready: 节点为准备好，相当于节点状态Ready的值为False
- node.kubernetes.io/unreachable: Node Controller 访问不到节点，相当于节点状态Ready的值为Unknown
- node.kubernetes.io/out-of-disk: 节点磁盘耗尽
- node.kubernetes.io/memory-pressure: 节点存在内存压力
- node.kubernetes.io/disk-pressure: 节点存在磁盘压力
- node.kubernetes.io/pid-pressuren: 节点存在PID压力
- node.kubernetes.io/network-unavailable: 节点网络不可达
- node.kubernetes.io/unschedulable: 节点不可调度

## 什么是容忍

容忍作用于Pod，主要用来让Pod可以接受某个污点的限制，也就是让某些Pod容忍节点上配置的污点，可以让一些需要特殊配置的Pod能够调用到具有污点和特殊资源的节点上。

污点和容忍相互配合，可以用来避免Pod被分配到不合适的节点上。

## 容忍配置

完全匹配

```yaml
tolerations:
- key: "taintKey"
  operator: "Equal"
  value: "taintValue"
  effect: "NoSchedule"
```

不完全匹配

```yaml
tolerations:
- key: "taintKey"
  operator: "Exists"
  effect: "NoSchedule"
```

Key匹配
```yaml
tolerations:
- key: "taintKey"
  operator: "Exists"
```

容忍所有污点（不推荐）

```yaml
tolerations:
- operator: "Exists"
```

容忍一段时间

```yaml
tolerations:
- key: "taintKey"
  operator: "Equal"
  value: "taintValue"
  effect: "NoExecute"
  tolerationSeconds: 3600
```

## 具体使用场景

### 主节点禁止调度

在生存环境中，kubernetes的主节点除了部署系统组件外，不推荐再部署任何服务。

```bash
kubectl taint node master node-role.kubernetes.io/control-plane:NoSchedule
```

也可以添加 NoExecute 类型的污点

```bash
kubectl taint node master node-role.kubernetes.io/control-plane:NoExecute
```

### 新节点禁止调度

当Kubernetes集群添加新节点时，通常情况下不会立即调度Pod到该节点，需要经过完整的可用性测试之后才可以调度pod，此时也可以使用污点先临时禁止该节点的调度

```bash
kubectl taint node work01 new-node=true:NoSchedule
```

待新节点测试完毕后，再允许该节点可以进行调度

```bash
kubect taint node work01 new-node-
```

### 节点维护流程

当kubernetes的节点需要进行下线维护时，需要先把该节点的服务进行驱逐和重新调度。

需要根据实际情况判断是直接驱逐还是选择重新调度，比如某个pod只有一个副本，或者某个服务比较重要，就不能直接进行驱逐，而是需要先把节点关闭调度，然后再进行服务的重新部署。

关闭维护节点的调度：

```bash
kubectl taint node work01 maintain:NoSchedule
```

重新触发某个pod的部署

```bash
kubectl rollout restart deploy nginx
```

对维护节点进行驱逐

```bash
kubectl taint node work01 maintain:NoExecute
```

节点维护完成后恢复调度

```bash
kubectl taint node work01 maintain-
```

除了自定义污点，还可以使用快捷命令将节点设置为维护状态

```bash
kubectl cordon work01
```

此时节点会被标记一个 SchedulingDisabled 状态，但是已经允许在该节点的 Pod 不受影响

```bash
kubectl get node work01
```

驱逐节点上面的服务

```bash
kubectl drain work01 --ignore-daemonsets --delete-emptydir-data
```

### 节点特殊资源保留

特殊节点应该尽量保持不要特殊资源的pod不要调度到这些节点上，此时可以通过污点进行控制。

比如包含GPU 的节点不能被任意调度

```bash
kubectl taint node work02 gpt=true:NoSchedule
```

具有其他特殊资源，尽量不要调度

```bash
kubectl taint node work02 ssd=true:PreferNoSchedule
```

### 使用容忍调度到具有污点的节点

```yaml
nodeSelector:
  gpu: "true"
tolerations:
- key: "gpu"
  operator: "Exists"
  effect: "NoSchedule"
```

### 专用节点隔离

专用节点，比如 ingress, gateway, storage，这些节点不建议和其他服务交叉使用，所以需要利用污点和容忍将这些节点隔离起来

```bash
kubectl label node work02 ingress=true
kubeectl taint node work02 ingress=true:NoSchedule
```

### 节点宕机快速恢复服务

当节点故障时，kubernetes会自动恢复故障节点上的服务，但是默认情况下，节点故障五分钟才会重新调度服务，此时可以利用污点的 tolerationSeconds 快速恢复服务

```yaml
tolerations:
- key: "node.kubernetes.io/unreachable"
  operator: "Exists"
  tolerationSeconds: "10"
```
