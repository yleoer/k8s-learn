# 任务管理

## Job

主要用来执行单次任务，比如执行一次备份任务/微调一次模型等。对于每次任务，Job控制器会创建一个或多个Pod执行相关指令，并且确保成功的个数达到预期的值才会把Job标记为成功，否则将会标记为失败。

具有命名空间限制

### 特点与优势

- 环境隔离：Job可以使用不同的镜像执行任务，无需考虑版本冲突
- 单次执行：Job任务通常是短暂的，运行一次后就结束
- 状态追踪：Job控制器会追踪所有Pod的完成情况，一旦达到指定的成功次数，Job才会被标记为完成
- 失败重试：任务执行失败，可以按需重新执行
- 并行执行：同一个任务可以拆分不同的Pod并行执行，提高执行速度

### 资源定义

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: pi
spec:
  backoffLimit: 4
  template: # 以下是 Pod 定义
    spec:
      containers:
        - name: pi
          image: perl:5.34.0
          command: ["perl", "-Mbignum=bpi", "-wle", "print bpi(2000)"]
      restartPolicy: Never
```

- backoffLimit: 如果任务执行失败，失败多少次后不再执行
- completions: 有多少个Pod执行成功，认为任务是成功的
  - 为空默认和parallelism一样
- parallelism: 并行执行任务的数量
  - 如果大于未完成任务数，只会创建未完成的数量
- ttlSecondsAfterFinished: Job 在执行任务结束之后自动清理，设置为0表示执行结束立即删除，不设置则不会清除

### 创建 Job

// 还需要yaml形式，job 资源不能修改

```bash
kubectl create job hello --image=busybox -- echo "Hello, Job"
```

查看 job

```bash
kubectl get job
```

## CronJob

是 Job 资源的延伸，主要用于周期性执行任务，比如定期备份数据库/定期对某个服务健康检查等。和Linux的计划任务功能类似，主要用于按计划执行某个操作。

### 特点与优势

- 周期执行：使用cron表达式定期创建job执行任务
- 并行执行：具备多种并发策略，调用job更加灵活
- 可靠性高：可以调度到k8s的任意节点运行，防止单点故障
- 环境隔离：可以使用不同的镜像执行任务，无需考虑版本冲突
- 历史记录：可以保留多个任务和失败的任务，便于问题追踪和审计

### 资源定义

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: hello
spec:
  schedule: "* * * * *"
  jobTemplate: # Job 相关配置
    spec:
    
```

- schedule: 调度周期，和 Linux 一致
- concurrencyPolicy: 并发调度策略
  - Allow: 允许同时运行多个任务
  - Forbid: 不允许并发任务
  - Replace: 如果之前的任务尚未完成，新的任务会替换之前的任务
- suspend: 如果设置为true，则暂停后续的任务，默认为false
- successfulJobsHistoryLimit: 保留多少已完成的任务
- failedJobsHistoryLimit: 保留多少失败的任务

### 创建

// 还需要yaml形式

```bash
kubectl crete cj hello --image=busybox --schedule='*/1 * * * *' -- echo "hello, cron job"
```

查看 cron job，时区默认是 UTC

```bash
kubectl get cj
```

### 调度时区

如果 CronJob 未标注调度时区，Kubernetes 会以 kube-controller-manager 组件的时区进行调度，如果该组件的时区和本地的不一样，则会出现无法按照规定时间进行调度。

在 kubernetes v1.25 版本，cronjob 增加了 .spec.timeZone 字段用于配置 cronjob 的调度时区
