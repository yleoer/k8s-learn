# 配置管理

## ConfigMap

ConfigMap 也是 K8s 的一种资源，主要用于存储配置数据，如程序的环境变量/配置文件等。ConfigMap 可以实现把应用程序的配置信息从容器镜像或者代码中分离出来，从而可以更容易的管理和更新配置，而不必重新构建应用程序。

主要用途：

- 存储配置信息：剥离应用程序配置，实现配置分离
- 动态配置：动态更改程序的配置，无需重新部署整个应用
- 共享配置：数据共享，多个Pod可以共享ConfigMap数据

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: example-config
data:
  key1: value1
  key2: value2
```

### 创建 ConfigMap

通过 yaml 文件创建 ConfigMap

```bash
kubectl create -f basic-cm.yaml

kubectl get cm

# 查看数据
kubectl describe cm example-config
```

基于多行yaml创建

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: multiple-config
data:
  multiple.conf: |-
    port: 6379
    bind: 0.0.0.0
    pass: PASSWORD
```

基于单个文件创建

```bash
# 默认key为文件名
kubectl craete cm nginx-config --from-file=nginx.conf
# 自定义key为nginxconf
kubectl create cm nginx-config2 --from-file=nginxconf=nginx.conf
```

基于多个文件创建

```bash
kubectl create cm multile-file --from-file=redis.conf --from-file=my.cnf
```

基于文件夹创建

```bash
kubectl create cm multile-dir --from-file=conf/
```

基于环境变量创建

创建 ConfigMap 时，可以通过 --from-env-file 创建环境变量类型的 ConfigMap。创建环境变量类型的 ConfigMap，只需要在文件内写入 Key=Value 格式的内容即可创建

```bash
cat env-file
DB_HOST=localhost
DB_PORT=3306
DB_USER=root

kubectl create cm env-cm --from-env-file=env-file

kubectl get cm env-cm -oyaml
apiVersion: v1
data:
  DB_HOST=localhost
  DB_PORT=3306
  DB_USER=root
```

基于 literal 创建

literal 一般用于创建环境变量形式的 ConfigMap

```bash
kubectl create configmap example-config --from-literal=key1=config1 --from-literal=key2=config2
```

### 更新 ConfigMap

基于 yaml 修改，需要注意文件是否是最新的

```bash
kubectl replace -f basic-cm.yaml
```

直接通过 edit 修改

```bash
kubectl edit cm basic-cm
```

通过修改本地文件

```bash
kubectl create cm nginx-conf --from-file=nginx.conf --dry-run=client -oyaml | kubectl replace -f -
```

### 挂载 ConfigMap

以文件形式挂载，在 Volume 中添加 ConfigMap，即可在 Pod 中的多个容器中挂载

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-test-cm
  labels:
    app: nginx-test-cm
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx-test-cm
  template:
    metadata:
      labels:
        app: nginx-test-cm
    spec:
      volumes:
      - name: conf
        configMap:
          name: multile-file
      containers:
      - name: nginx-test-cm
        image: nginx:1.15.23
        volumeMounts:
        - name: conf
          mountPath: /mnt
        - name: conf
          mountPath: /data # 可以把同一个挂载不同目录
```

查看挂载详情

```bash
kubectl exec -it nginx-test-cm -- ls /mnt
```

指定挂载的文件名：

```yaml
volumes:
- name: conf
  configMap:
    name: multile-file
    items:
    - key: my.cnf
      path: mysql.conf
```

此时挂载的文件名会被更改为 mysql.conf，同时只会挂载 items 匹配的 key

默认情况下，如果指定的key名不存在，pod会一直处于ContainerCreating

可以添加 optional 字段，把 key 必须存在改为非必须：

```yaml
volumes:
- name: conf
  configMap:
    name: multile-file
    items:
    - key: my.cnf
      path: mysql.conf
    optional: true
```

自定义挂载权限

进制转化：https://tool.oschina.net/hexconvert

ConfigMap 和 Sercret 挂载时，默认的挂载权限是 644（没有执行权限），可以在挂载时，指定挂载权限：

```yaml
volumes:
- name: conf
  configMap:
    name: multile-file
    items:
    - key: my.cnf
      path: mysql.conf
      mode: 0777 # 局部权限，对单个 item 配置生效，八进制
    defaultMode: 420 # 默认权限，对所有的 item 生效，十进制
```

文件挂载覆盖的问题

ConfigMap和Secret 挂载时，会直接覆盖原目录的内容，所以在挂载时需要注意。比如需要挂载 nginx.conf 时，如果直接挂载至 /etc/nginx 目录，会导致 nginx 无法启动。

解决挂载覆盖：

```yaml
volumeMounts:
- mountPath: /etc/nginx/nginx.conf # 指定挂载具体的文件路径
  name: nginxconf
  subPath: nginx.conf # 指定挂载具体的文件
```

以环境变量形式挂载

```yaml
# 指定 key 去挂载
env:
- name: DB_HOST # 变量名
  valueFrom:
    configMapKeyRef:
      name: env-cm # ConfigMap 名称
      key: DB_HOST # key 名
# 批量生成
envFrom:
- configMapRef:
  name: env-cm
prefix: CM_ # 添加变量前缀
```

key 不存在同样会导致 Pod 一直处于 CreateContainerConfigError 状态

// env 和 envFrom 同时存在的时候，谁的优先级更高？

// 当 configMap 修改后，需要重启 pod 才能获取新的 env

## Secret

和 ConfigMap 类似，Secret 也是 K8s 的一种资源，同样可以存储配置数据。和 ConfigMap 不同的是 Secret 提供了一种相对安全的方式来管理这些数据。同时相对于 ConfigMap，Secret 也有不同的类型，不同的类型具有不同的使用场景。比如 Docker Registry Secret 可以用来管理镜像仓库的用户名密码。

主要用途：

- 存储敏感信息：可以存储密码/Key等信息
- 数据保护：Secret的数据可以被加密存储，防止未经授权的访问
- 动态配置：和ConfigMap类似
- 共享配置：和ConfigMap类似

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: example-secret
type: Opaque
data:
  passwordk: cGFzc2dvcmQ= # base64 编码后的 "password"
  token: dG0rZW4=
```

### Secret 常用类型

- Opaque: 通用型 Secret，默认类型
- kubernetes.io/dockerconfigjson: 下载私有仓库镜像使用的 Secret，和宿主机的 /root/.docker/config.json 一致，宿主机登陆后即可产生该文件
- kubernetes.io/tls: 用于存储 HTTPS 域名证书文件的 Secret，可以被 Ingress 使用
- bootstrap.kubernetes.io/token: 一种简单的 bearer token，用于创建新集群或将新节点添加到现有集群，在集群安装时可用于自动颁发集群的证书
- kubernetes.io/basic-auth: 用于使用基本认证（账号密码）的Secret，可以使用 Opaque 取代
- kubernetes.io/ssh-auth: 用于存储 ssh 密钥的 Secret
- kubernetes.io/service-account-token: 作用于 ServiceAccount, 包含一个令牌，用于标识API服务账户

### 创建 Secret

使用 literal 创建，创建后会自动加密

```bash
kubectl create secret generic basic-auth-secret \
--from-literal=username=admin \
--from-literal=password=securepassword
```

查看创建的 Secret 资源

```bash
kubectl get secret
```

通过yaml创建的时候，写入 data 的需要先对数据进行加密，放在 stringData 下，无需先进行加密。

基于文件和目录创建，和 ConfigMap 类似：

```bash
kubectl create secret generic nginxconf --from-file=conf/nginx.conf
```

使用 Secret 管理镜像仓库密钥

如果需要拉取私有仓库的镜像，需要给 deploy 等资源配置镜像仓库的密钥，此时可以先创建镜像仓库的密钥

```bash
kubectl create secret docker-registry myregistrykey \
--docker-server=DOCKER_REGISTRY_SERVER \
--docker-username=DOCKER_USER \
--docker-password=DOCKER_PASSWORD \
--docker-email=DOCKER_EMAIL
```

之后在 Pod 字段下添加该 Secret 即可

```yaml
spec:
  imagePullSecrets:
  - name: myregistrykey
```

使用secret管理域名证书，然后用于 Ingress

```bash
# 生成测试证书
openssl req -x509 -nodes -days 365 \
-newkey rsa:2048 -keyout tls.key -out tls.crt -subj "/CN=test.com"

# 创建 Secret
kubectl create secret tls nginx-test-tls --key=tls.key --cert=tls.cert
```

然后在 Ingress 中使用

```yaml
tls:
- secretName: nginx-test-tls
```

### 挂载 Secret

挂载 Secret 和 ConfigMap 类似，也可以以文件和配置变量挂载

## 只读配置

使用 immutable 字段把资源设置为只读模式，这样就不能再更新 data 下的任何数据

```yaml
apiVersion: v1
kind: ConfigMap
immutable: true
```

## 备份还原

目前不支持备份和还原，可以使用 git 仓库管理 ConfigMap 和 Secret 的版本。

## 注意事项

- 需要提前创建 ConfigMap 和 Secret
- ConfigMap 和 Secret 必须和 Pod 或者是引用它资源在同一命名空间
- 如果引用了某个key，需要确保引用的 key 必须存在
- envFrom/valueFrom 无法热更新环境变量，subPath 也是无法热更新的
- envFrom 配置环境变量，如果key无效，会忽略
- ConfigMap 和 Secret 最好不要太大
- ConfigMap 和 Secret 支持热更新，但程序未必支持
